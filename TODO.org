* TODO workflow

This file is the canonical work queue for humans and coding agents.

** Rules

1. Keep unfinished work above =Completed archive= and completed work below it.
2. Use exactly one lifecycle keyword on actionable headings:
   =TODO=, =NEXT=, =DOING=, =BLOCKED=, or =DONE=.
3. At most one top-level item may be =DOING= at a time. An agent must mark the
   item it is implementing before changing production code.
4. Every actionable item must state an observable completion contract. Prefer
   checkboxes naming syntax, semantics, diagnostics, and regression coverage.
5. Implement test-first: add a focused failing regression, confirm the expected
   failure, implement the smallest coherent change, then run broader suites.
6. A task is =DONE= only when its completion contract passes and relevant CI is
   green. Record the tests and commit under the item.
7. When a task becomes =DONE=, move the whole entry to the bottom of
   =Completed archive=. Do not leave completed projects mixed into active work.
8. =BLOCKED= entries must name the blocking item or missing decision. Move them
   back to =TODO= or =NEXT= as soon as the blocker is removed.
9. Do not silently broaden an item. Add newly discovered work as a linked TODO
   and keep the current task scoped to its stated contract.
10. Preserve design rationale. Cleanup may reorganize or merge duplicates, but
    must not discard decisions, test evidence, or unresolved questions.

** Priority

- =DOING=: currently being implemented.
- =NEXT=: agreed next work; ordered from top to bottom.
- =TODO=: accepted backlog, not yet scheduled.
- =BLOCKED=: accepted work that cannot proceed yet.
- Research-only ideas must say =[RESEARCH]= in the heading.

** Definition of done

- The user-facing behavior works from a clean checkout.
- Focused regression coverage exists and passes.
- The affected core/compiler suites pass on Linux and Windows where relevant.
- Documentation and this TODO record describe the behavior that actually ships.
- The implementation is committed and pushed; CI is green.

** TODO Use musttail for closure tail calls [1/2]
  - [X] Annotate straight-line closure-ABI tail dispatch with LLVM =musttail=
        by jumping directly to the closure function pointer.
  - [ ] Teach branch/if lowering to emit direct returns in tail branches so
        branch-local closure tail calls can also satisfy LLVM's =musttail=
        verifier rule.

  Your closures capture environments and dispatch through function pointers.
  Monadic bind chains like =maybe-bind (f x) g= create N-deep tail call
  chains through =RuntimeClosure= dispatch. Without =musttail=, LLVM may
  or may not optimize these as tail calls depending on ABI and the
  presence of variadic args.

  =musttail= is an LLVM IR directive (=musttail call ...) that forces the
  backend to emit a jump instead of a call + ret. It's the difference
  between =$ do x <- f; g x= O(n) and O(1) stack depth.

  Concretely: in =codegen.c=, when emitting a call in tail position of
  a function body (specifically the last expression in a =lambda= or
  =define=), use =LLVMSetTailCall= and add the =musttail= attribute.
  This is mechanical: walk =codegen_expr= for the tail-call positions
  and emit =musttail call= instead of =call=.

** Tiered eval: bytecode VM + unoptimized/optimized LLVM [0/6]

 eval is not a REPL nicety in a Lisp — it's a language primitive, and right
 now repl_eval_line always pays the full LLVM path no matter how trivial the
 form: fresh_module (new LLVMModuleRef, redeclare_env_symbols walks the
 *entire* env hash table re-declaring every global/func as extern so the
 verifier doesn't reject cross-module refs), codegen_expr, LLVMVerifyFunction
 + LLVMVerifyModule, LLVMAddModule, map_runtime_in_module +
 map_imported_in_module + map_ffi_in_module, LLVMGetFunctionAddress. (+ 1 2)
 pays the same module-creation tax as a real function definition. That's not
 a REPL latency annoyance, it's an asymptotic cost baked into the language's
 own evaluation model — every macro-expanded form, every incremental define,
 every future (eval form) call goes through this.

 bytecode.c (register-based bytecode VM, structured control flow only,
 single-pass verifier, visual tracer, sectioned binary format) already
 exists and is unit-tested, but is not wired into repl.c at all yet.

 The shape: three tiers, cost paid only when it buys something —

   1. Bytecode interpreter (bc_vm_run)  — zero entry cost, default for eval
      and any toplevel form. No module creation, no env-wide redeclare walk,
      no LLVM verification, no dlsym address fetch.
   2. Unoptimized LLVM (-O0, no pass pipeline) — reuses the EXISTING
      fresh_module/close_and_run machinery verbatim, just fed IR lowered
      from BcProgram instead of from codegen_expr's AST walk. Promoted to
      once something is called enough to amortize module overhead.
   3. Optimized LLVM (real pass pipeline) — genuinely hot code only.

 Rejected: a third hand-rolled x64 JIT tier between 1 and 2. The existing
 baseline JIT in bytecode.c (x64_emit_probed_stack_alloc, x64_mov_*,
 x64_binop_slot) only lowers straight-line I64 arithmetic — no if, no loop,
 no calls — and extending it to real control flow means hand-rolling
 register allocation, branch lowering, and calling-convention handling that
 LLVM already does correctly. Concretely this also means the JIT path
 inherits LLVM's correct sdiv/srem lowering on x86 for the INT64_MIN / -1
 case for free, instead of hand-rolling that guard in raw machine code.
 Tier 2 = tier 3's existing pipeline, just -O0.

 The dispatch boundary is a property checkable BEFORE codegen, not a fuzzy
 heuristic: does this form touch anything boxed (RuntimeValue*, lists, maps,
 closures, define) or not? Pure I64/F64/Bool arithmetic and control flow ->
 eligible for the bytecode path. Anything boxed -> existing LLVM path,
 completely untouched.

** TODO Fix Soundness Holes [0/6]
*** TODO Move all implicit coercions out of dep_conv

    =dep_conv_vals= must be strict definitional equality only.
    Currently allows ~Int ≡ Float~, ~Int ≡ Char~, ~Nat ≡ Int~ inside conversion
    which is mathematically unsound — ~Vec Nat~ and ~Vec Int~ become definitionally
    equal, opening a path to proving =False=.
    - Move all coercions to the elaboration layer only
    - =dep_conv= should never return =True= for structurally different ground types

*** TODO Fix meta_solve occurs check

    Currently evaluates the solution in the empty environment:

#+begin_src c

#+end_src

    De Bruijn indices in =solution= can refer to variables that don't exist in
    the empty env, so =dep_eval= produces neutrals and =meta_occurs= misses
    occurrences inside those neutrals' spines.
    - Occurs check must run in the same environment the solution was elaborated in

*** TODO Implement proper Miller pattern unification

    The current stub is labeled TODO and is unsound:

#+begin_src c
/* TODO: Proper pattern unification requires checking that v1->spine
 * contains only distinct bound variables */

#+end_src

    Without checking the Miller pattern condition, metas can be solved with
    terms containing variables outside the meta's scope — a real scoping
    soundness hole, not a missing optimization.
    - Check that spine of =v1= contains only distinct bound variables
    - Prune variables not in scope before solving

* TODO QTT (Quantitative Type Theory)

QTT adds usage quantities to each binder. Quantities are typically 0, 1, and ω.

0 means the variable is never used. The bound term can be erased completely
before code generation. This provides guaranteed dead code elimination.

1 means the variable is used exactly once. No memoization is beneficial
because the result is consumed immediately. No reference counting is needed
because there is only one reference. Values bound with quantity 1 can be
moved rather than copied.

ω means the variable is used many times (zero or more). Memoization may be
beneficial depending on the cost of recomputation versus cache access.
Reference counting is required for heap-allocated mutable data with ω usage.

The quantity is attached to the binder, not the type. A function parameter
can be annotated with a quantity, and the language guarantees that the
function obeys that usage.

Related concepts: linear types (exactly once), affine types (at most once),
relevant types (at least once), unrestricted types (any number), resource
semantics, erasure analysis, uniqueness typing.

* Inference rules

Inference rules come in this form:

                   ⌜P₁⌝ is the first  Premise.
     P₁ P₂         ⌜P₂⌝ is the second Premise.
    ───────        ⌜──⌝ is the inference line.
       C           ⌜C⌝  is the Conclusion.

The idea is to make them first class in the language.

 Example:

 #+begin_src monad

     x : Int   3 : Int
if  ───────────────────
        x + 3 : Int

then printLn "It type checked!"

 #+end_src

 #+begin_src monad

            x : Int
         ─────────────
          x + 3 : Int

    (λx. x + 3) : Int -> Int

 #+end_src

  - Per-function optimization without module overhead.

  - Access to the new =LLVMCreatePassBuilder= API which takes the

  an explicit pipeline. Set =opt_level=0= for the unoptimized tier
  and =opt_level=2= for the hot tier. No more relying on whatever
  MCJIT's legacy pipeline happened to do.
