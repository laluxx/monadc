[DOC id:doc.traceability.purpose src:context/traceability.org conf:high]
  The traceability spine is the path from implementation to living memory to
  executable regression. It prevents three common failures: tests with no design
  meaning, documentation with no source anchor, and fixes that do not promote
  failing witnesses into regression coverage.

#+CAPTION: Preferred traceability chain
| Step | Artifact | Example question |
|------+----------+------------------|
| 1 | Source observation | What does the code actually do? |
| 2 | Context contract | What should this phase guarantee? |
| 3 | Test fixture | Which executable witness protects the guarantee? |
| 4 | Test tier | Is it green regression, known-fail, or future? |
| 5 | Failure artifact | What changed on the last run? |
| 6 | Promotion record | When fixed, what moved from known-fail to regression? |

[DEC id:dec.traceability.no-orphan-tests date:2026-06-16 by:llm]
  New tests should have TEST-CONTEXT whenever they assert a language or compiler
  contract. Context-free smoke tests are allowed, but they should be marked as
  smoke, not as semantic coverage.

[DEC id:dec.traceability.no-orphan-context date:2026-06-16 by:llm]
  New durable context claims about compiler behavior should either cite a source
  observation or name the tests that protect the claim.

#+CAPTION: Promotion steps
| State | Action |
|-------+--------|
| future | Spec exists but compiler does not claim support. |
| known-fail | Bug witness captures intended current gap. |
| fixed locally | Run the specific witness and inspect artifacts. |
| regression | Remove known-failure marker and keep metadata. |
| documented | Add FIX/DEC record linking the test to the category. |
