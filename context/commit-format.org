plain text in =git log=, and to serve as a machine-readable context
source for LLM agents working on the codebase.

[DEC id:dec.commit.purpose date:2026-06-11 by:human]
  A commit message is a durable record of what changed, why it changed,
  and what future readers need to know to work safely near this code. It
  is not a summary of the diff — the diff speaks for itself. The message
  explains the decision, the constraint, the invariant, and the contract.

[DEC id:dec.commit.three-questions date:2026-06-11 by:codex]
  Every commit message must answer three questions:

1. What is the shape of this change? /the first line/
2. What are the major areas of work and how do they fit together? /the section headers/
3. What does a future reader need to know that the diff does not show? /the body prose/

  Every commit message follows this shape. All three parts are mandatory.

#+BEGIN_EXAMPLE
<type>(<scope>): <summary>

<body>

There is no optional footer section. Notes that would go in a footer
(breaking changes, issue references, co-authors) go in the body prose
of the relevant section, inline with context.

[DEC id:dec.commit.types date:2026-06-11 by:codex]
  The type prefix must be one of the values in the table below. No other
  values are permitted. When a commit genuinely spans multiple types, use
  the type of the primary intent — use =feat= if a new capability is the
  reason the commit exists.

#+CAPTION: Allowed Commit Types
| Type     | When to use                                                       |
|----------+-------------------------------------------------------------------|
| =feat=     | A new language feature, compiler pass, or user-visible capability |
| =fix=      | A bug fix — something was wrong and is now correct                |
| =refactor= | Code restructuring with no behaviour change                       |
| =test=     | New or updated tests, corpus rows, or test infrastructure         |
| =context=  | Changes only to =context/= Org files or the visualizer              |
| =build=    | Makefile, CI, install, or toolchain changes                       |
| =docs=     | Documentation outside the context corpus (READMEs, man pages)     |
| =perf=     | A measurable performance improvement                              |
| =chore=    | Maintenance with no semantic effect (renaming, moves, cleanup)    |

[DEC id:dec.commit.context-type date:2026-06-11 by:human]
  =context= is a first-class type alongside =feat= and =fix=. Updating the
  Org corpus is real work that deserves its own commit and its own history.
  It must not be hidden inside =chore=.

[DEC id:dec.commit.prose-not-bullets date:2026-06-11 by:human]
  Write prose, not bullet lists. Each paragraph covers one decision, one
  constraint, or one contract. Bullet lists obscure the relationship
  between ideas and make the message harder to parse as a context record.

#+CAPTION: What Belongs in the Body
| Include                                            | Exclude                              |
|----------------------------------------------------+--------------------------------------|
| New invariants introduced by the change            | A prose retelling of the diff        |
| Constraints the diff does not make obvious         | Vague statements of intent           |
| Reasoning behind non-obvious design choices        | Praise or self-congratulation        |
| Subsystem boundary descriptions                    | Bullet lists                         |
| Known limitations or named follow-up work          | Anything already visible in the diff |
| =TEST-ID= values for new tests                       |                                      |
| Context node =:ID:= values for new or updated nodes  |                                      |

[DEC id:dec.commit.tense date:2026-06-11 by:codex]
  Use imperative mood throughout. The commit /adds/, /fixes/, /removes/,
  /introduces/, /extends/ — never /added/, /fixed/, /I added/, /we
  changed/. Use active voice. Use present tense for invariants: "All
  =AST_PATH= nodes store their text in the =string= union field" — this is
  true now and must remain true.

[DEC id:dec.commit.naming date:2026-06-11 by:codex]
  Name every symbol exactly as it appears in source: =AST_PATH=, not "the
  path AST node" or "ASTPath". Symbol names, file names, function names,
  flag names, and type names are written in =monospace= inline. Line-wrap
  prose at 72 characters. Code examples or file paths that cannot be
  wrapped may exceed 72 characters on a single line.

For a small, single-area change with no section headers needed:

#+BEGIN_EXAMPLE
fix(wisp): preserve ~[ as one balanced token before ]

The Wisp tokeniser was splitting `define darr ~[1 2 3]` into
`(define darr ~[1)` followed by orphaned `2` and `3]` tokens.
The fix detects `~[` at the start of a token and scans forward
to the matching `]`, respecting nested brackets and string literals.
The expression parser and infix collector are updated to treat tokens
beginning with `~[` as grouped atomic expressions so they are never
placed in operator position.

This is a prerequisite for the reader correctly receiving `~[...]` as
adjacent `~` and `[...]` and setting `AST_ARRAY.array.is_heap = true`.
#+END_EXAMPLE

For a large, multi-area change spanning several compiler layers:

#+BEGIN_EXAMPLE
feat: path literals, heap arrays, test suite, and context corpus

────────────────────────────────────────────────────────────────────────
READER: Path literals (AST_PATH / TOK_PATH)
────────────────────────────────────────────────────────────────────────

Filesystem path literals are now first-class expressions. The lexer
recognises four surface forms before the symbol branch so the full
token is consumed greedily: `/abs/path`, `~/home/path`, `./rel/path`,
and `../parent/path`. Bare `/` is excluded so the division operator is
unaffected.

`AST_PATH` stores its text in the `string` union field — the same field
used by `AST_STRING`. This is intentional: the runtime representation is
identical (a bare `i8*`), but the distinct AST and type tags allow
future passes to apply path-specific static checks without changing the
codegen ABI.

All compiler phases have been updated: `ast_clone`, `ast_free`,
`ast_print`, `ast_to_json_sb`, `infer_expr`, `dep_term_of_ast_internal`,
and all codegen paths that previously switched on `AST_STRING` now also
handle `AST_PATH`. `TYPE_PATH` maps to a pointer at the LLVM level and
is exposed to users as the type name `Path`.

────────────────────────────────────────────────────────────────────────
WISP / READER: Heap array literals (~[...])
────────────────────────────────────────────────────────────────────────

Stack array literals `[1 2 3]` now have a heap-backed counterpart
`~[1 2 3]`. Wisp's responsibility is to preserve `~[...]` as a single
balanced token; the reader's responsibility is to set `is_heap = true`
on the resulting `AST_ARRAY` node. These two responsibilities must not
be conflated: a future change to either layer should not need to touch
the other.

`type_arr_heap(T)` constructs a `TYPE_ARR` with `arr_size = -1`,
`arr_is_fat = false`, and `arr_is_heap = true`. `type_to_string`
renders this as `"Heap :: T"`. `type_from_name("Heap")` resolves to
this type. `ast_clone` preserves `is_heap` so desugaring passes cannot
silently drop the flag.

────────────────────────────────────────────────────────────────────────
TESTS: Structured suite with table-driven language corpus
────────────────────────────────────────────────────────────────────────

`tests/run.py` is a new test runner. Tests are discovered in two modes:
fixture tests (`tests/*.mon` files with `TEST-ID` comment metadata) and
corpus rows (`tests/language_corpus.tsv`, one language atom per row).
`TEST-EXPECT` drives assertions: `inventory`, `parse-json`, `compile`,
`run`. A `.stdout` sidecar triggers a golden-suffix check.

New fixture `tests.reader.path-heap-literals` covers the full pipeline
from Wisp through parse, JSON emission, codegen, and runtime execution.
The `make test` target now builds all and runs the full suite.

────────────────────────────────────────────────────────────────────────
CONTEXT: Repository memory corpus and live visualizer
────────────────────────────────────────────────────────────────────────

`context/` is a new directory of Org files recording why the compiler
is the way it is, not only what it does. Nodes carry stable `:ID:`
properties prefixed `monadc.context.`; inline `[OBS]`, `[INF]`, and
`[DEC]` records separate direct observations from inferences and
decisions. IDs are immutable once published; superseding records must
be appended rather than rewriting history.

`context-visualizer.py` serves the corpus as a live typed graph at
`/api/context` and exposes `POST /api/todo` for writing TODO state
back to the Org files. The dashboard (`context/visualizer/`) renders
the graph as an animated canvas with a fuzzy finder, TODO work queue,
and a Hoogle-style method browser. New context nodes introduced:
`monadc.context.reader.path-literals`,
`monadc.context.reader.heap-arrays`,
`monadc.context.tests.metadata-format`.
#+END_EXAMPLE

[THINK id:think.commit-format.section-order-pipeline-mirror conf:high]
  The section order (wisp → reader → types → infer → codegen → tests → context)
  mirrors the compilation pipeline's phase ordering. This is an elegant
  design choice: reading a commit message in section order is like reading
  the changes in pipeline order, from source parsing to code generation.
  A developer reviewing a commit can mentally trace the changes through
  the pipeline, which makes the commit message both a changelog and a
  pipeline trace. This is particularly valuable for a compiler where a
  single feature change often touches multiple phases — the section order
  tells you what changed at each stage without having to understand the
  full commit diff.

[THINK id:think.commit-format.72-char-enforcement conf:medium]
  The 72-character line limit for both the summary line and body text
  follows conventional Git best practices (the summary limit is 50-72,
  the body limit is 72). This is useful for command-line Git tools that
  display summaries in narrow terminals and for tools like =git log
  --oneline= that truncate at 72. The enforcement cost is that developers
  must manually wrap lines, which is tedious. In practice, most editors
  have auto-wrap at 72, and the =git commit= message editor typically
  shows a column ruler. The format spec explicitly states no automated
  enforcement, which is pragmatic: the rules are guidelines for writing,
  not hooks for CI.

[THINK id:think.commit-format.imperative-voice-rule conf:medium]
  The imperative voice requirement (commit-format.org §Subject) is the
  standard Git convention ("Fix bug" not "Fixed bug" or "Fixes bug").
  It keeps the subject line consistent with Git's own merge messages
  and makes the commit message read as an instruction: "if applied, this
  commit will [subject]." The rule is well-known in the Git community
  and easy for new contributors to follow. The format spec's additional
  rule about active present tense for invariants is more subtle: it
  distinguishes between "what this commit changes" and "what the code
  guarantees." A commit message that says "lexer: handle \\n escape in
  char literals" describes the change; one that says "wisp: wisp
  expands \\\\n to real newline" describes the invariant that the commit
  establishes or documents.

[THINK id:think.commit-format.scope-naming cons:medium]
  The commit scope naming (wisp, reader, types, infer, codegen, tests,
  context, build, docs, perf, chore, fix, feat, refactor) maps directly
  to the context file set (wisp.org, reader.org, types.org, etc.). This
  means the commit scope is also a reference to the context documentation
  for that subsystem. A commit with scope =reader= tells the reviewer
  "to understand these changes, read reader.org." This bidirectional
  mapping between commit scopes and context files is a subtle but
  powerful feature of the project's documentation integration: every
  commit message implicitly links to the context file that explains the
  subsystem.

[THINK id:think.commit-format.rule-overhead conf:medium]
  The commit format rules are detailed (16 sections covering everything
  from summary line format to section ordering to bullet prohibition).
  This level of detail creates a barrier to entry for casual contributors
  who are not steeped in the project's workflow conventions. The rules
  are optimized for the primary developers (you + LLM agents) who are
  intimately familiar with the pipeline structure and context system.
  For external contributors, the rules file itself is the onboarding
  document: if you can follow these rules, you understand the project
  well enough to contribute meaningfully. The detail is a feature, not
  a bug — it encodes the project's mental model in the contribution
  workflow.
