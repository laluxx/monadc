[DEC id:dec.guidelines.hard-vs-soft date:2026-06-12 by:human]
  =Hard Constraints= are mandatory rules that must always be followed ("must do").
  =Soft Preferences= are recommended practices that should be followed when
  practical ("should do"). Violating a soft preference is acceptable when there
  is a good reason, but the reason should be documented.

[DEC id:dec.guidelines.always-improve-context date:2026-06-11 by:human]
  Always improve the repository context as work progresses. Add important
  discoveries, decisions, verification results, and follow-up tasks to the Org
  context corpus when they would help future work.

#+CAPTION: Context Upkeep Triggers
| Trigger                            | Required Context Action                          |
|------------------------------------+--------------------------------------------------|
| New compiler behavior is fixed     | Record the observed bug and implemented contract |
| A standing user preference appears | Add or update a rule in guidelines.org           |
| A reusable workflow is discovered  | Add it to the relevant topic file                |
| A follow-up task is identified     | Add it to context/todo.org or a topic TODO       |

[DEC id:dec.guidelines.context-during-conversation date:2026-06-11 by:human]
  When the user states a standing preference or describes a durable workflow
  during conversation, add it to the context corpus during the same work
  session instead of leaving it only in chat history.

[DEC id:dec.guidelines.visualizer-quality-bar date:2026-06-11 by:human]
  Project visualizations should meet a high usability and design bar: clear
  dashboard structure, standard interaction patterns, smooth motion, and
  direct editing where the visualization represents mutable project state.

[DEC id:dec.guidelines.extend-not-replace date:2026-06-12 by:codex]
  When improving the context corpus, always extend (add new headings, records,
  files) rather than silently replacing existing content. Use superseding
  records when correcting factual errors. This preserves the audit trail.

[DEC id:dec.guidelines.stale-cleanup date:2026-06-12 by:human]
  When the LLM encounters a context fact that is stale, outdated, or superseded
  by newer code, it must delete or supersede it. Do not let dead facts accumulate.
  Pruning is as important as adding. Use =SUPERSEDED= status on headings or
  remove records that no longer reflect reality. The context should be a
  faithful model of the current codebase, not an archive of everything ever
  believed.

[DEC id:dec.guidelines.always-extend-context date:2026-06-12 by:human]
  Never forget to write back and extend the context. Everything the LLM does
  should leave a trace in the context corpus. Every discovery, decision,
  observation, TODO, and interaction that adds durable value must be recorded
  before the session ends. This is the core mechanism that turns ephemeral
  conversations into permanent collaborative memory.

[DEC id:dec.guidelines.context-first date:2026-06-12 by:human]
  Before starting any work, read the relevant context files. After finishing,
  extend the context with what was done, discovered, decided, and left undone.
  Never leave the context behind — always extend it.

[DEC id:dec.guidelines.opinions-are-todos date:2026-06-12 by:human]
  Every opinion recorded in =context/opinions.org= must be tracked as a TODO
  item in =context/todo.org=. Opinions are not decorative commentary — they
  are identified improvement opportunities that carry an implicit action
  item. When an opinion is written, a corresponding TODO must be created.
  When an opinion is addressed, its TODO is marked done. This ensures the
  opinions file drives real change instead of becoming noise.

[DEC id:dec.guidelines.living-coverage date:2026-06-11 by:human]
  Explore MonadC as a living language system: keep adding small behavioral
  tests as language behavior is discovered, prefer codegen-backed compile/run
  fixtures when possible, and update context records continuously so the test
  suite and repository memory evolve together.

[DEC id:dec.guidelines.coverage-over-green date:2026-06-11 by:human]
  Do not stop coverage expansion merely because a codegen-backed test exposes a
  deep compiler/runtime failure. Fix only trivial fixture syntax mistakes during
  coverage work; leave meaningful deep failures visible as executable pressure
  on the implementation.

[DEC id:dec.guidelines.coverage-then-triage date:2026-06-11 by:human]
  After a coverage subset is large enough to reveal useful behavior, triage the
  failing tests instead of treating every failure the same. Classify each failure
  as an invalid fixture, a missing runner expectation mode, or a compiler/runtime
  defect; fix syntax/contract mistakes immediately and use the linked context
  record when choosing the next compiler fix.

[DEC id:dec.guidelines.skip-uncertain-failures date:2026-06-11 by:human]
  During broad coverage expansion, do not spend long debugging an uncertain
  failure. If a fixture emits JSON and the failure is not clearly a syntax or
  test-contract mistake, keep the coverage and move on so reader, codegen,
  closure, HM, and dependent-type surfaces continue expanding.

[DEC id:dec.guidelines.context-recurring-todos date:2026-06-12 by:codex]
  Use =context/todo.org= for durable, cross-session TODOs discovered during
  work. Small ephemeral TODOs ("fix this one fixture") should go in the
  conversation summary, not in permanent context records.
