[DOC id:doc.category.purpose src:context/category-context.org conf:high]
  Category context is the rule that every context topic is not just a document:
  it is an object in a typed graph. Objects have contracts. Morphisms have
  meanings. Composition must preserve meaning. A test can verify a language
  contract; a source observation can support an inference; a decision can
  supersede an opinion. These edges should be explicit enough for humans and
  LLMs to traverse safely.

[THINK id:think.category.context-as-garden-map src:user-request conf:high]
  The user called this garden. The category-context layer is the trellis: it
  decides what counts as a plant, what counts as a path, what counts as a weed,
  and how future gardeners can extend without turning the corpus into mulch.

#+CAPTION: Context category objects
| Object | Concrete form | Identity | Must answer |
|--------+---------------+----------+-------------|
| File | =context/*.org= | path + title | What domain does this file own? |
| Category | durable heading | =:ID:= | What contract or concept does this node own? |
| Record | bracket block | =id:= | What atomic claim/task/decision is recorded? |
| Source region | =src:path:line= | path + line/range | What evidence anchors the claim? |
| Test fixture | =TEST-ID= | test ID | What behavior or failure does this protect? |
| Run artifact | result JSON/log | timestamp + run config | What happened in one execution? |
| External reference | paper/link/manual | citation key/link | What outside theory informs the project? |

[DEC id:dec.category.objects-stable date:2026-06-16 by:llm]
  File, category, record, source region, test fixture, run artifact, and external
  reference are the canonical object types. New object types must justify what
  identity they add that these do not cover.

#+CAPTION: Context morphisms
| Morphism | From | To | Meaning |
|----------+------+----+---------|
| =describes= | heading/record | source region | Text explains implementation facts. |
| =supports= | OBS/DOC | INF/THINK/DEC | Evidence backs a derived claim. |
| =decides= | DEC | contract/category | Decision binds future work. |
| =supersedes= | DEC/FIX | record/heading | New record replaces older guidance. |
| =verifies= | test fixture | context heading | Test protects the named contract. |
| =exercises= | test fixture | source phase | Test drives code through phase behavior. |
| =depends-on= | category | category | Category cannot be reasoned about alone. |
| =routes-to= | navigation entry | file | Prefix or need maps to file. |
| =owns= | category | failures/tests | Category is responsible for triage. |
| =blocks= | TODO/bug | contract | Task prevents contract from being true. |

[DEC id:dec.category.edge-vocabulary date:2026-06-16 by:llm]
  Prefer these edge names in prose, tables, and future generated graph exports.
  Do not invent vague edge names like =related= when a stronger relation fits.

#+CAPTION: Useful compositions
| Composition | Valid reading |
|-------------+---------------|
| test =verifies= heading, heading =describes= source | test indirectly guards source behavior |
| OBS =supports= INF, INF =supports= DEC | decision is only as strong as the evidence chain |
| TODO =blocks= contract, test =verifies= contract | failing test is expected until TODO closes |
| DEC =supersedes= THINK | opinion becomes binding project direction |
| navigation =routes-to= file, file contains heading | ID prefix lookup reaches durable node |

[DEC id:dec.category.no-untyped-composition date:2026-06-16 by:llm]
  A chain of links should not be treated as evidence unless each edge has a
  meaningful morphism. =file link + similar words= is not enough to infer a
  verified relationship.

[INF id:inf.category.failure-triage from:dec.category.edge-vocabulary,dec.category.no-untyped-composition conf:high]
  The fastest triage path is: failing TEST-ID → TEST-CONTEXT heading → category
  contract → owned compiler phase → source observations and known TODOs.

#+CAPTION: Anti-patterns
| Anti-pattern | Why it hurts | Fix |
|--------------+--------------+-----|
| Decorative category labels | Looks structured but cannot guide work | Add owned phase and failure symptoms. |
| Dangling TEST-CONTEXT | Test claims to verify a nonexistent contract | Create heading or correct reference. |
| Vague links | =related to= cannot compose safely | Use typed morphism. |
| Opinion as OBS | Confuses judgment with evidence | Use THINK or IDEA. |
| Stale source lines | False confidence after refactors | Revalidate =src:= anchors. |
| Giant catch-all file | Breaks routing and ownership | Split or add subcontracts. |

[DEC id:dec.category.graph-export-json date:2026-06-16 by:llm]
  A generated graph export should include nodes for files, headings, and records,
  and edges for containment, file links, ID links, and record-level =from:= /
  =supersedes:= references. The export is diagnostic data, not authoritative
  hand-written context.

Generated graph outputs should not be committed unless intentionally used as a
snapshot. Prefer regenerating them for analysis.
