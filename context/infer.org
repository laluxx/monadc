[OBS id:obs.context.infer.file src:infer.h,infer.c conf:high]
  This file records the architecture of the Hindley-Milner type inference engine
  in MonadC (=infer.h= / =infer.c=). It covers: substitution (union-find for type
  variables), constraint generation, unification (Robinson's algorithm with
  occurs check), expression inference, generalisation/instantiation (let-polymorphism),
  zonking, builtins registration, and the top-level pipeline. The HM engine is
  the fast path of the dual type system (see =monadc.context.language.type-system-surface=).

[OBS id:obs.infer.scheme-struct src:infer.h:42-46 conf:high]
  `TypeScheme` carries `quantified[]` (array of universally-quantified type
  variable IDs), `quantified_count`, and `type` (the body `Type*` containing
  `TYPE_VAR` nodes whose IDs appear in `quantified[]`). Example: the identity
  function's scheme is `∀0. Arrow(Var(0), Var(0))`.

[OBS id:obs.infer.type-subst-struct src:infer.h:52-55 conf:high]
  `TypeSubst` maps quantified scheme var IDs to concrete types at a call site.
  Used by codegen to drive monomorphization: caller records which concrete types
  each quantified var resolved to, then generates a specialised copy.

[OBS id:obs.infer.scheme-mono src:infer.c:896-901 conf:high]
  `scheme_mono(Type *t)` creates a trivial TypeScheme with no quantifiers —
  the monomorphic case. Used internally for lambda parameters and temporary
  pre-binds.

[OBS id:obs.infer.scheme-clone src:infer.c:904-911 conf:high]
  `scheme_clone(TypeScheme *s)` creates a shallow copy of a scheme. The
  quantified list is deep-copied (malloc + memcpy) but the body Type is shared
  (not deep-copied). Schemes live as long as the InferCtx that created them.

[OBS id:obs.infer.scheme-free src:infer.c:913-917 conf:high]
  `scheme_free(TypeScheme *s)` frees the quantified array and the scheme struct
  itself, but does NOT free the body Type — types are owned by the substitution.

[OBS id:obs.infer.instantiate-with-subst src:infer.c:848-891 conf:high]
  `infer_instantiate_with_subst(InferCtx *ctx, TypeScheme *scheme, TypeSubst *ts)`
  works like `infer_instantiate` but also returns the mapping from quantified
  var IDs to fresh var IDs via the `TypeSubst` out-parameter. This mapping is
  consumed by codegen monomorphization after unification resolves the fresh vars
  to concrete types. The caller must free `ts->from` and `ts->to` (but not the
  `Type*` pointers).

[OBS id:obs.infer.env-struct src:infer.h:70-80 conf:high]
  `InferEnv` is a singly-linked scope chain implemented as a fixed-size hash
  table (64 buckets, djb2 hash). Each entry is an `InferEnvEntry` with `name`,
  `scheme`, and `next` pointer. The `parent` field forms the lexical scope
  chain: a child env created by `infer_env_create_child` inherits all bindings
  from its parent.

[OBS id:obs.infer.env-create src:infer.c:24-30 conf:high]
  `infer_env_create(void)` allocates a fresh environment with 64 empty hash
  buckets and no parent.

[OBS id:obs.infer.env-create-child src:infer.c:32-35 conf:high]
  `infer_env_create_child(InferEnv *parent)` creates a new scope with the given
  parent. Used for let-bindings, lambda parameters, and define scopes.

[OBS id:obs.infer.env-free src:infer.c:38-52 conf:high]
  `infer_env_free(InferEnv *env)` walks all buckets, frees entry names and
  entries. Does NOT free schemes — schemes are owned by the InferCtx.

[OBS id:obs.infer.env-insert src:infer.c:54-68 conf:high]
  `infer_env_insert(InferEnv *env, const char *name, TypeScheme *scheme)` inserts
  or overwrites a binding. Overwrites the scheme in-place for existing names
  (old scheme freed by caller or ctx). Otherwise prepends a new entry to the
  hash bucket chain.

[OBS id:obs.infer.env-lookup src:infer.c:71-89 conf:high]
  `infer_env_lookup(InferCtx *ctx, const char *name)` searches the local HM
  scope chain (walking parent links). Falls back to the TT Global Environment
  (dctx) — the "Idris Bridge" — but this path is DISABLED due to use-after-free
  segfaults from stale dctx pointers passed by codegen. The dependent elaborator
  now pre-seeds `ast->inferred_type` on all nodes, making the bridge unnecessary.

[OBS id:obs.infer.env-remove src:infer.c:980-996 conf:high]
  `infer_env_remove(InferEnv *env, const char *name)` removes a binding from
  the environment. Used before generalisation in define handling to prevent old
  type vars from corrupting `infer_free_vars_env`.

[OBS id:obs.infer.subst-struct src:infer.h:99-104 conf:high]
  `Substitution` contains `union_find[]` (parent pointers — each var is its own
  root initially), `bound[]` (concrete Type* at root nodes — NULL means free),
  `capacity` (INFER_MAX_VARS = 4096), and `next_id` (next fresh var ID).
  The invariant: `bound[id]` is non-NULL only when `union_find[id] == id`
  (i.e., at root nodes). Non-root nodes have NULL bound.

[OBS id:obs.infer.subst-create src:infer.c:94-103 conf:high]
  `subst_create(void)` allocates union_find and bound arrays for INFER_MAX_VARS,
  initialises each var as its own root, sets next_id to 0.

[OBS id:obs.infer.subst-free src:infer.c:105-110 conf:high]
  `subst_free(Substitution *s)` frees union_find, bound, and the struct itself.
  Does NOT free bound types — types are owned by the type arena.

[OBS id:obs.infer.subst-fresh src:infer.c:112-121 conf:high]
  `subst_fresh(Substitution *s)` allocates a new type variable ID, initialises
  it as its own root with NULL bound. Panics with exit(1) if the capacity
  (INFER_MAX_VARS = 4096) is exceeded.

[OBS id:obs.infer.subst-find src:infer.c:123-130 conf:high]
  `subst_find(Substitution *s, int id)` performs path-compressed find with
  path halving: `union_find[id] = union_find[union_find[id]]` on each iteration.
  Returns the root ID. This is the core operation of union-find — all lookups
  go through find to resolve aliases.

[OBS id:obs.infer.subst-bind src:infer.c:132-139 conf:high]
  `subst_bind(Substitution *s, int id, Type *t)` binds a variable root to a
  concrete Type. Has a critical guard: never overwrites an existing concrete
  binding — doing so would create substitution cycles where subst_apply_depth
  chases var → type → var → type forever.

[OBS id:obs.infer.subst-union src:infer.c:141-156 conf:high]
  `subst_union(Substitution *s, int a, int b)` merges two variable roots. If
  both roots have concrete bindings, returns true without merging — the caller
  (infer_unify_one_internal) handles structural unification instead. Otherwise
  merges rb into ra and carries rb's binding to ra if ra has none.

[OBS id:obs.infer.subst-apply-shallow src:infer.c:158-166 conf:high]
  `subst_apply_shallow(Substitution *s, Type *t)` performs one-level
  dereference: if t is a TYPE_VAR whose root has a concrete binding, returns
  the bound type. Also performs path compression directly on the AST node
  (mutating `t->var_id` to the root) for asymptotic performance improvement.

[OBS id:obs.infer.subst-apply-depth src:infer.c:168-226 conf:high]
  `subst_apply_depth(Substitution *s, Type *t, int depth)` recursively walks
  a type, replacing all resolved TYPE_VAR nodes with their concrete bindings.
  Has a cycle guard at depth 64. Handles TYPE_LIST, TYPE_OPTIONAL, TYPE_PTR,
  TYPE_COLL, TYPE_ARR, and TYPE_ARROW by recursing into sub-types. Returns a
  new Type if any sub-type changed, or the original if unchanged. This is the
  core substitution application — it resolves the entire type graph.

[OBS id:obs.infer.subst-apply src:infer.c:228-230 conf:high]
  `subst_apply(Substitution *s, Type *t)` is a thin wrapper calling
  `subst_apply_depth(s, t, 0)`.

[INF id:inf.infer.subst-connect from:obs.infer.subst-apply,obs.infer.subst-bind,obs.infer.subst-union conf:high]
  connects-to -> `monadc.context.infer.unification` (unification mutates the

[OBS id:obs.infer.constraint-struct src:infer.h:122-127 conf:high]
  `TypeConstraint` stores `lhs` (Type*), `rhs` (Type*), `line`, and `col` —
  the two types to be unified and the source location for error messages.
  Constraints are stored as a flat dynamic array in InferCtx.

[OBS id:obs.infer.constrain-append src:infer.c:301-308 conf:high]
  `infer_constrain(InferCtx *ctx, Type *a, Type *b, int line, int col)` appends
  a new constraint to the ctx constraint buffer. Doubles capacity when full
  (starts at 256). This is the sole mutation point of the constraint list —
  all inference code calls this to defer unification.

[THINK id:think.infer.constraint-defer-why conf:high]
  Why defer unification instead of solving eagerly? Deferred constraints enable
  W-by-W (walk-by-walk) inference for let-generalisation: constraint generation
  runs over the full definition body, then unification solves all constraints
  before generalisation. This is the correct architecture for constraint-based
  HM (see Heeren et al, 2002). Eager unification would force a left-to-right
  bias that can produce unnecessary monomorphic bindings.

  connects-to -> `monadc.context.infer.expression-inference` (infer_expr
  generates constraints for every application) ·
  `monadc.context.infer.unification` (infer_unify_all consumes the constraint
  list) ·
  `monadc.context.infer.occurs-check` (occurs check runs during unification,
  triggered by constraints)

  substitution with fresh type variable IDs. The env persists across definitions

  solved and discarded within one inference run. This prevents ID collision
  between independent definitions and keeps the union-find structure small.

[OBS id:obs.infer.constraint-gen-pattern src:infer.c:301-308 conf:high]
  Constraint generation follows a uniform pattern: for each sub-expression,
  call `infer_expr` recursively (which returns the inferred Type), then call
  `infer_constrain(ctx, inferred, expected, line, col)` to record the
  constraint. The expected type may be a fresh variable (if the type is not
  yet known) or a concrete type (from annotations).

[OBS id:obs.infer.constraint-gen-define src:infer.c:1208-1273 conf:high]
  For `define` forms: generates a fresh self-type for recursive references,
  infers the body, constrains self ~ body, then solves all pending constraints
  BEFORE generalising. This W-by-W pattern is critical for recursive definitions:
  generalising before solving would quantify type vars that should resolve to
  concrete types, producing unsound polymorphic schemes.

[OBS id:obs.infer.constraint-gen-application src:infer.c:1492-1537 conf:high]
  For function application (fn arg1 arg2 ...): infers the function type `fn_t`,
  creates a fresh return type `ret_t`, builds an expected arrow type from the
  argument types and return, and constrains `fn_t ~ expected`. This is the
  standard HM application rule: if `f : A → B` and argument types match, the
  return type is resolved through unification of the function's arrow with
  the expected arrow constructed from argument/return types.

[OBS id:obs.infer.constraint-gen-if-nil-optional src:infer.c:1331-1344 conf:high]
  Special case: `if` branches with one Optional/Nil branch and one concrete
  branch get smart Optional promotion — the concrete branch is wrapped in
  `type_optional()` before constraining. This makes `(if cond nil x)` infer
  as `Optional(T)` rather than failing to unify Nil with T.

[OBS id:obs.infer.constraint-gen-collection-as-function src:infer.c:416-470,1470-1488 conf:high]
  Collection types (TYPE_SET, TYPE_MAP, TYPE_COLL, TYPE_ARR, TYPE_LIST) used as
  functions skip arrow unification — the codegen handles collection indexing
  directly. In unification, `Coll ~ (Int → elem)` is decomposed into
  `Int ~ Int` + `elem ~ elem`. In expression inference, if the function head
  resolves to a collection type, arguments are inferred for side effects and
  a fresh return type is generated.

[INF id:inf.infer.constraint-gen-connect from:obs.infer.constraint-gen-pattern,obs.infer.constraint-gen-define,obs.infer.constraint-gen-application conf:high]
  connects-to -> `monadc.context.infer.expression-inference` (constraint

  `infer_unify_one_internal`. If occurs check fails, the error message is
  `"type error: infinite type (occurs check failed)"` with source location.

  would chase var → type → var → type forever.

[THINK id:think.infer.occurs-why-necessary conf:high]
  The occurs check is necessary because without it, binding `'a ~ 'a → Int`
  creates a type `'a = 'a → Int = ('a → Int) → Int = ...` — an infinite type
  that the substitution walker cannot terminate. Robinson's unification
  algorithm requires the occurs check as the fourth case (after reflexivity,
  variable binding, and structural decomposition).

[OBS id:obs.infer.unify-one-internal src:infer.c:338-703 conf:high]
  `infer_unify_one_internal(InferCtx *ctx, Type *a, Type *b, int line, int col)`
  is the core unification algorithm. It:
  1. Applies shallow substitution to both sides
  2. Handles NULL and TYPE_UNKNOWN (always succeeds)
  3. Var-Var: merges roots via subst_union; if both are bound, unifies bindings
  4. Var-Type: binds the var root to the type (with occurs check and nil→Optional guard)
  5. Type-Var: symmetric to var-type
  6. Ground-Ground: kind matching with a coercion lattice:
     - TYPE_FN compatible with TYPE_ARROW
     - TYPE_FN ~ TYPE_FN always ok
     - Collection-as-function: `Coll ~ (Int → elem)` coercion
     - Int/IntArbitrary/Ratio w idening
     - Int→Float, Int→Char implicit coercions
     - TYPE_COLL compatible with List, Arr, String, Set
     - TYPE_APP ~ TYPE_LAYOUT compatibility
     - Single-element TYPE_LIST as parenthesized scalar
  7. Structural decomposition for TYPE_ARROW, TYPE_LIST, TYPE_ARR

[OBS id:obs.infer.unify-all src:infer.c:696-703 conf:high]
  `infer_unify_all(InferCtx *ctx)` iterates all constraints in the buffer and
  calls `infer_unify_one` on each. Returns false and sets `ctx->had_error` +
  `ctx->error_msg` on the first type error. The constraint buffer is NOT cleared
  (the caller resets `constraint_count` to 0 as needed — see define handling).

[OBS id:obs.infer.unify-one src:infer.c:315-335 conf:high]
  `infer_unify_one(InferCtx *ctx, Type *a, Type *b, int line, int col)` is a
  tracing wrapper around `infer_unify_one_internal`. When `g_trace_enabled`,
  prints the unification attempt, indents for nested calls, and prints OK/FAIL.
  This is the public API; all external callers go through this wrapper.

[OBS id:obs.infer.unify-coercion-lattice src:infer.c:414-549 conf:high]
  The implicit coercion lattice in unification:
  - Int ↔ Float (widening)
  - Int ↔ Char (chars are small integers)
  - IntArbitrary ↔ Int (widen to i64)
  - F80 ↔ Float (extended precision)
  - Ratio ↔ Int, Ratio ↔ Float (rational is numeric)
  - TYPE_FN ↔ TYPE_ARROW (unannotated Fn compatible with any arrow)
  - TYPE_COLL ↔ List, Arr, String, Set (collection hierarchy)
  - TYPE_APP ↔ TYPE_LAYOUT (type constructor applications)
  - Nil binding → Optional(fresh) via the Nil guard

[OBS id:obs.infer.unify-nil-guard src:infer.c:376-386,404-411 conf:high]
  Critical design: never bind a free variable directly to TYPE_NIL — Nil is not
  a concrete element type. Binding `'x ~ Nil` would poison any later constraint
  `'x ~ element_type` because Nil wins the race and element_type never gets
  resolved. Instead, Nil binding produces `Optional(fresh)` so subsequent
  constraints resolve the inner element type correctly.

[THINK id:think.infer.unify-collection-as-function conf:high]
  The collection-as-function coercion (infer.c:426-470) is the type system
  acknowledging the set-theoretic model: if types are sets, then a collection
  IS a function from its index type to its element type. When unifying
  `TYPE_LIST ~ TYPE_ARROW`, the algorithm decomposes into
  `Int ~ arrow_param` and `elem ~ arrow_ret`. Safety guards prevent this
  coercion when the arrow param is already a complex type (arrow, collection,
  string, bool, float) that can never be an index type — this prevents
  cascading unification explosions from polymorphic operators like `=` and `>=`.

[OBS id:obs.infer.generalise src:infer.c:787-813 conf:high]
  `infer_generalise(InferCtx *ctx, Type *t, InferEnv *outer_env)`:
  1. Applies substitution fully to t (resolves all known bindings)
  2. Collects free type variables in t via `infer_free_vars_type`
  3. Collects free type variables in the outer environment via `infer_free_vars_env`
  4. Quantifies all vars that are free in t but NOT free in the environment
  This is the standard HM generalisation rule: a let-bound expression's type
  is polymorphic over all type variables that were not introduced by lambda
  parameters in the enclosing scope.

[OBS id:obs.infer.instantiate src:infer.c:815-846 conf:high]
  `infer_instantiate(InferCtx *ctx, TypeScheme *scheme)`:
  1. If quantified_count == 0, applies substitution to the scheme type (monomorphic)
  2. Otherwise, allocates fresh type variable IDs for each quantified var
  3. Temporarily binds quantified vars to the fresh vars in the substitution
  4. Applies substitution to get the instantiated type
  5. Clears the temporary bindings
  This is the standard HM instantiation: each use of a polymorphic name gets
  fresh type variables that may unify independently at different call sites.

[OBS id:obs.infer.free-vars-type src:infer.c:714-747 conf:high]
  `infer_free_vars_type(Substitution *s, Type *t, int *out, int *count, int cap)`:
  collects free (unresolved) TYPE_VAR IDs in a type. Applies shallow substitution
  first so bound vars are not reported. Handles TYPE_VAR (add if not already in
  array), and recurses into TYPE_LIST, TYPE_OPTIONAL, TYPE_PTR, TYPE_COLL,
  TYPE_ARR, TYPE_ARROW.

[OBS id:obs.infer.free-vars-env src:infer.c:749-783 conf:high]
  `infer_free_vars_env(InferCtx *ctx, InferEnv *env, int *out, int *count, int cap)`:
  collects free type vars in the outer environment's non-generalised schemes.
  Only walks the single env level passed (not the parent chain) — parent envs
  contain schemes from previous InferCtx runs whose type var IDs belong to
  foreign substitutions and cannot be safely dereferenced with the current
  ctx->subst. Skips fully generalised schemes (quantified_count > 0).

[THINK id:think.infer.gen-inst-why-define-pattern conf:high]
  The define handling follows the W-by-W (walk-by-walk) pattern: generate
  constraints → solve → generalise. This is essential for recursive definitions:
  if generalisation ran before solving, free type vars that should resolve to
  concrete types (like the return type of a recursive call) would get universally
  quantified, producing unsound polymorphic schemes and infinite-type occurs-check
  failures at call sites. See Heeren, "Generalizing Hindley-Milner Type Inference
  Algorithms", 2002.

  Zonking as a post-pass is the standard approach: generate constraints → solve
  → apply substitution to the output types. The AST's inferred_type is the
  output channel from inference to codegen.

:CONTEXT_DESCRIPTION: The Nil→Optional guard prevents binding free type variables to TYPE_NIL directly. Instead, Nil produces Optional(fresh) so subsequent constraints can still resolve the inner element type. This is a pragmatic solution to the Nil-is-not-a-type problem.
:CONTEXT_VERSION: 1

[OBS id:obs.infer.nil-guard-in-unify src:infer.c:376-386,404-411 conf:high]
  In `infer_unify_one_internal`, binding a free var `'x ~ Nil` is intercepted.
  Instead of binding `'x` to `TYPE_NIL`, the algorithm creates a fresh inner
  type variable and binds `'x` to `Optional(inner)`. This is triggered for both

[THINK id:think.infer.nil-guard-why conf:high]

  it is a singleton type. But `nil` appears in two roles: as the empty list
  terminator (where its type should be `List('a)` not `Nil`) and as the absent
  value (where its type should be `Optional('a)` not `Nil`). Directly binding
  `'x ~ Nil` makes `'x` forever `Nil`, which means any later constraint
  `'x ~ String` would fail. The guard ensures Nil usage creates an Optional
  wrapper, preserving the ability to resolve the inner type from context.

[OBS id:obs.infer.coll-as-fn-in-unify src:infer.c:416-470 conf:high]
  When unifying a collection type against TYPE_ARROW, the algorithm decomposes
  the arrow's parameter to Int (the index type) and the arrow's return to the
  element type. Safety guard: only applies when the arrow param resolves to a
  simple index-like type (Int, Char, or unbound fresh var). Complex types
  (arrow, collection, string, bool, float) that can never be index types skip
  the coercion — this prevents `=` from binding `String ~ (Int → ?)` which
  would cause cascading unification failures.

[OBS id:obs.infer.coll-as-fn-in-expr src:infer.c:1470-1488 conf:high]
  In `infer_expr`, when the function head symbol resolves to a collection type
  in the env, the application skips arrow unification entirely. Arguments are
  inferred for side-effects only. String indexing returns Char; other collection
  indexing returns a fresh type. Codegen handles the actual indexing via
  `rt_coll_index` or the collection-specific path.

[THINK id:think.infer.coll-as-fn-semantics conf:high]
  Collection-as-function is the type system expressing the set-theoretic model.
  In the semantics, a set IS its characteristic function, an array IS a function
  from Int to element, a map IS a function from key to value. The HM type system
  cannot express dependent function types (Π), so it approximates collection
  application as "the collection acts like an Int-indexed function on the left
  arrow." The safety guards exist because this approximation interacts badly
  with parametric polymorphism — `=` has type `∀a. a → a → Bool`, and without
  guards, `(= my-set 1)` would try to unify `Set` with `(Int → Bool)`, failing
  or producing strange cascade errors.
