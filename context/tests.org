#+BEGIN_QUOTE
LLMs should treat this repository as a living memory system. The =context/=
Org files explain why behavior exists, while =tests/*.mon= fixtures prove the
smallest observable atoms of that behavior. A test header reconnects executable
evidence to context IDs: when a test fails, read its =TEST-CONTEXT= nodes before
changing code; when code changes, add or update the smallest test atom and its
context record.
#+END_QUOTE

[DEC id:dec.tests.smallest-atoms date:2026-06-11 by:human]
  The test suite should be separated into the smallest possible behavioral atoms.
  Each atom should prove one compiler promise, point at the context that explains
  that promise, and be cheap enough to run continuously.

[DEC id:dec.tests.living-system date:2026-06-11 by:human]
  Documentation, philosophy, context, and tests are all part of the same living
  system. The system records not only what the compiler does, but why that
  behavior matters and how it is verified.

#+CAPTION: Living Verification Loop
| Layer      | Artifact          | Role                                     | Update Trigger                 |
|------------+-------------------+------------------------------------------+--------------------------------|
| Philosophy | =context/tests.org= | why tests are atomic and connected       | standing preference changes    |

| Fixture    | =tests/*.mon=       | smallest source example                  | bug report or feature contract |
| Runner     | =tests/run.py=      | executable verification                  | new assertion style is needed  |

Every =tests/*.mon= fixture starts with comment metadata. The runner validates
the required keys before running behavioral checks.

#+BEGIN_EXAMPLE
;; TEST-ID: tests.<domain>.<atom>
;; TEST-CONTEXT: monadc.context.<topic>.<node>[, monadc.context.<topic>.<node>...]
;; TEST-PURPOSE: one sentence describing the exact behavior under test
;; TEST-ATOM: one smallest observable promise
;; TEST-EXPECT: parse-json|compile|run|fail:<phase>|json:<shape>[, compile, run]
#+END_EXAMPLE

#+CAPTION: Test Metadata Keys
| Key          | Required | Meaning                                |
|--------------+----------+----------------------------------------|
| =TEST-ID=      | yes      | stable test identity                   |
| =TEST-CONTEXT= | yes      | context node IDs that explain the test |
| =TEST-PURPOSE= | yes      | human-readable reason                  |
| =TEST-ATOM=    | yes      | one behavior, not a scenario bundle    |
| =TEST-EXPECT=  | yes      | observable outcome class               |

[DEC id:dec.tests.comment-metadata date:2026-06-11 by:codex]
  Test metadata uses =;; TEST-*= comments so fixtures remain ordinary Monad
  programs and the metadata is visible to both the compiler and plain-text tools.

[DEC id:dec.tests.sidecar-metadata date:2026-06-11 by:human]
  Each durable regression may also have a sidecar Org file named after the test
  fixture, for example =tests/path_heap_literals.org= for
  =tests/path_heap_literals.mon=. The sidecar records richer metadata, context
  links, regression contracts, and historical notes that would make the executable
  fixture noisy.

#+CAPTION: Test Artifact Pairing
| Artifact       | Naming              | Purpose                                      |
|----------------+---------------------+----------------------------------------------|
| Fixture        | =tests/<name>.mon=    | executable behavior atom                     |
| Stdout golden  | =tests/<name>.stdout= | expected runtime stdout or diagnostic text   |
| AST golden     | =tests/<name>.json=   | expected =--emit-json= AST after loc normalization |
| Desugar golden | =tests/<name>.desugar= | expected printed desugared AST block         |
| Sidecar        | =tests/<name>.org=    | metadata, context links, regression contract |
| Runner         | =tests/run.py=        | formatted execution and assertions           |

[DEC id:dec.tests.runner-colors-cleanup date:2026-06-11 by:human]
  The test runner is named =tests/run.py=. It must delete test-generated
  artifacts immediately after each test, render the arrow in yellow, dots in gray,
  success checkmark in green, =PASS= in bold green, =FAIL= in bold red, and time
  units such as =ms= in cyan.

[DEC id:dec.tests.emacs-progress-lines date:2026-06-12 by:human supersedes:dec.tests.runner-colors-cleanup]
  Per-test progress rows must begin at column zero with an Emacs
  =compilation-mode=-parseable location prefix:
  =/absolute/path/to/test.mon:line:col: note:= for passing tests and
  =/absolute/path/to/test.mon:line:col: error:= for failing tests. The old
  leading =-▶= marker is removed. The progress counter remains in the metadata
  bracket and the =0001/1063= part is rendered in bright white.

[DEC id:dec.tests.progress-counter-only date:2026-06-12 by:human supersedes:dec.tests.emacs-progress-lines]
  Per-test rows should not expose internal runner metadata such as fixture-vs-TSV
  source kind, expectation mode, or context-link count. The visible bracket should
  contain only the bright-white ordinal counter, for example =[0001/1063]=. The
  runner computes the longest visible line prefix before execution and pads rows
  so =PASS= and =FAIL= begin in the same column across the whole suite. This
  keeps the output both Emacs-compatible and scan-friendly.

[DEC id:dec.tests.ast-json-goldens date:2026-06-12 by:human]
  A fixture with a sibling =.json= file has a first-class expected AST golden.
  The runner always executes =--emit-json= before compile/run checks and compares
  the emitted AST against the golden after stripping volatile source-location keys
  =line= and =col=. This makes parser/desugar AST shape observable, not merely
  runtime stdout. Expected-failure fixtures may also carry =.json= goldens so a
  syntactically valid but ill-typed program can assert both its parsed AST and
  its compile-time rejection.

[DEC id:dec.tests.desugar-goldens date:2026-06-12 by:human]
  A fixture with a sibling =.desugar= file must match the printed
  ==== desugared AST ===/=== end desugared AST ==== block exactly after trimming.
  Use this for reader sugar such as Wisp =for= expansion where stdout alone would
  not prove that the expected intermediate form was produced.

[DEC id:dec.tests.expected-failures date:2026-06-12 by:human]
  Supposed-to-fail tests are first-class regression fixtures. Use
  =TEST-EXPECT: fail:compile= for a fixture that should be rejected by the
  compiler, and pair it with a sibling =.stdout= file containing the expected
  diagnostic text. The runner normalizes the repository root to =<repo>= for
  these diagnostic golden checks.

[DEC id:dec.tests.last-first-failure-record date:2026-06-12 by:human]
  =tests/run.py= writes =tests/.last-first-failure.org= after every suite run.
  The file records the first failing test from that run, its failure reason,
  fixture or materialized source, =TEST-*= metadata, linked context IDs, and
  normalized diagnostic output. This gives the next debugging pass a stable
  starting point even when the full suite output is long or the earliest failure
  changes between runs.

[OBS id:obs.tests.fuzzing-codegen-2026-06-13 src:tests/fuzzing/fuzz_codegen.py,Makefile conf:high]
   =tests/fuzzing/fuzz_codegen.py= is the first deterministic type-directed
   fuzzing harness. It discovers real property assets from
   =tests/fuzzing/properties/*.fuzz=, generates typed Int/Bool programs with
   typed helper definitions, compiles them with =monad=, runs the executable,
   and checks stdout against generated algebraic laws. The current
   QuickCheck-style law set covers addition/multiplication commutativity,
   addition associativity, distributivity, arithmetic identities and inverses,

   in =?= because the compiler enforces that naming rule. Richer direct Boolean
   literal generation found a current =False= codegen gap, so it should be
   re-enabled only with a focused regression fix. The runner intentionally
   prints output like =tests/run.py=: runner-width dividers, aligned
   =path:line:column:= prefixes, seed compile/run rows, one row per generated
   law, microsecond generation timing, expected/actual values, source property
   locations, and failure replay files under =tests/fuzzing/last=. Full
   generated expressions are available with =--show-expr=. On failure the
   runner performs a first shrink pass by trying a one-law counterexample and
   then the smallest failing prefix. Current integration is type-directed and
   compiler-pipeline-backed: generated programs carry explicit type signatures
   and are checked by =dep.c=/=infer.c= before codegen. A future deeper layer
   should introspect exported type information from the compiler itself instead
   of maintaining the generator type universe only in Python and =*.fuzz=

[DOC id:doc.tests.fuzzing-system-2026-06-13 src:tests/fuzzing/fuzz_codegen.py,tests/fuzzing/properties,Makefile conf:high]
   Purpose: =tests/fuzzing= is the property-based test layer for compiler
   behavior that is too broad to cover with hand-written runtime fixtures
   alone. It is meant to find interactions between the reader, dependent
   checker (=dep.c=), HM inference (=infer.c=), codegen, and runtime printing by
   generating many small typed programs and checking algebraic laws.

   Entry point: run it through Make, not by memorizing script paths:
   =make fuzzing= or =make test-fuzzing=. The Makefile target maps to
   =python3 tests/fuzzing/fuzz_codegen.py= and exists because repository helper
   scripts should have Makefile front doors.

   Property files: each law lives in =tests/fuzzing/properties/*.fuzz= with
   fields =name=, =section=, =args=, =type=, =expect=, =description=, and =law=.
   The =args= field declares the generator types, for example =a:Int b:Int= or
   =cond:Bool=. The =law= field is a template where ={name}= placeholders are
   replaced by generated expressions of the declared type. Positive laws use
   =expect: True=. Negative properties are first-class and use =expect: False=,
   for example strict self-comparison =(fuzz-lt? a a)=.

   Type-system integration: the fuzzer is type-directed at generation time and
   type-checked by the compiler at execution time. It emits typed helper
   definitions such as =fuzz-add :: Int -> Int -> Int=, predicate helpers such
   as =fuzz-lt? :: Int -> Int -> Bool=, and generated =show= forms. The compiler
   must parse, desugar, run =dep.c=/=infer.c=, lower through codegen, link, run,
   and print the expected value for every generated law. This is stronger than
   parser fuzzing but not yet full compiler-introspective fuzzing; the next
   level is to export compiler type/environment metadata and let the fuzzer
   derive generators from that.

   Output contract: the fuzz runner follows =tests/run.py= presentation rules.
   Rows start with =path:line:column:= for Emacs compilation-mode, use aligned
   prefixes and the shared divider width, show PASS/FAIL plus duration, and
   keep generated expressions hidden by default. Use =--show-expr= for full law
   bodies, =--list-properties= for property documentation, and =--keep= to keep
   generated programs even when they pass.

   Failure handling: failing generated programs are copied to
   =tests/fuzzing/last=. The runner also attempts a small source-level shrink:
   isolate a one-law counterexample when possible, otherwise report the
   smallest failing prefix of the generated law list.

[THINK id:think.tests.tsv-format-strength conf:high]
  The TSV test format (input\t\expected\tcontext-id) is deceptively
  powerful. It reduces each test to its minimum viable expression: a
  concrete input, an expected output, and a link back to the design
  intent. This format is easier to write than full test functions, easier
  to read than assertion-heavy test suites, and the context-id column
  creates the verifies-edge in the visualizer graph. The format shines
  for compiler tests where the test is "does this expression produce the
  right output?" — which is the vast majority of compiler correctness
  testing. It doesn't work well for tests that require setup/teardown,
  mocking, or multiple assertions, but those are rare in a compiler
  context.

[THINK id:think.tests.corpus-growth-trajectory conf:high]
  At 25 feature groups with 923 files, the test corpus is already large
  and growing. The TSV format scales well for read/parse/print tests
  (add one row per test) but less well for compile/run tests (each
  requires a .mon source file, an expected output file, and a TSV entry).
  The current growth trajectory (roughly doubling every 6 months of
  active development) means the test suite will reach 2000+ files within
  a year. At that scale, the TSV runner's current approach (shell-based,
  grep for test IDs, sequential execution) will slow down noticeably.
  A parallel test runner (pytest-based, using the TSV parser functions
  in run.py) would be a natural evolution.

[THINK id:think.tests.continuous-mode-value conf:high]
  The continuous test mode (make test-continuous, polling every 500ms)
  is an underappreciated feature. It transforms the development workflow
  from "write code, run tests, wait, fix" to "write code, see results
  appear". For a compiler where a single change can break hundreds of
  tests, immediate feedback is invaluable. The polling approach (rather
  than inotify/fsevents) is the right call for portability — it works
  on every filesystem and every platform without dependencies — though
  it means CPU is wasted on repeated test runs. The 500ms polling
  interval is a good default: fast enough to feel responsive, slow enough
  to batch rapid saves.

[THINK id:think.tests.verification-gap conf:medium]
  The test-visualizer bridge (TEST-ID → context node → verifies edge)
  is brilliant for traceability, but it only works in one direction:
  tests link to context, but context doesn't link back to tests. A
  context node cannot easily answer "which tests verify me?" without
  grepping all TSV files for its ID. Adding a reverse-index (a
  context→tests mapping generated by the visualizer or a helper script)
  would close the loop and make the test-coverage graph bidirectional.
  This is low-effort (grep all TSV files for context IDs, emit a map)
  and high-value (any context node author can immediately see test
  coverage).

[THINK id:think.tests.false-positive-risk conf:high]
  The TSV format has a structural false-positive risk: if a test's
  expected output is incorrect (hand-typed wrong, not updated after a
  behavior change), the test passes with a wrong expected value. There
  is no mechanism to distinguish "test verifies correct behavior" from
  "test encodes wrong expectation". The context-ID link mitigates this
  (each expected output references a design intent in a context node),
  but nothing enforces that the expected output matches the context
  intent. A schema validation pass that checks expected-output format
  and range would catch some of these, but the fundamental risk remains:
  tests encode expectations, expectations can be wrong.
