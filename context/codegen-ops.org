[OBS id:obs.codegen-ops.arch.layer1 src:codegen.c:3382-3988 conf:high]
  *Layer 1 — AST type switch* (=codegen_expr=). The top-level switch on
  =ast->type= handles leaf AST nodes directly:
  - =AST_SYMBOL= (line 3382): nil literal, True/False booleans,
    =__adt_tag_CtorName=, dot-chain field lookup, variable resolution,
    ADT constructor values, function value wrapping.
  - =AST_NUMBER= (line 3380): numeric literal → boxed =rt_make_int= or
    =rt_make_float=.
  - =AST_ARRAY= (line 3380): array literal with optional fat-pointer
    runtime sizing; the =lo .. hi= range also dispatches here.
  - =AST_LIST= (line 3990): empty list error, inline string indexing
    (=("str" i)=), inline array indexing (=[1 2 3] i=), quoted list
    indexing (='(a b c) i=), ratio-as-function (=(1/4 x)=), then
    falls through to strcmp dispatch on the head symbol.
  - =AST_LAYOUT= (line 13027), =AST_DATA= (line 13034),
    =AST_CLASS= (line 13041), =AST_INSTANCE= (line 13067),
    =AST_LAMBDA= (line 13074), =AST_TESTS= (line 12996).

[OBS id:obs.codegen-ops.arch.layer2 src:codegen.c:4101-12677 conf:high]
  *Layer 2 — strcmp dispatch* (inside =ast->type == AST_LIST= when
  =head->type == AST_SYMBOL=). Each known operation symbol is compared
  via =strcmp= in a long chain of =if= statements spanning
  codegen.c:4101-12677. These are callable from paren-mode. If no
  strcmp matches, falls through to layer 3.

[OBS id:obs.codegen-ops.arch.layer3 src:codegen.c:9790-11976,12677 conf:high]
  *Layer 3 — env lookup and generic call* (inside =AST_LIST= when no
  strcmp matched). The head symbol is resolved via
  =resolve_symbol_with_modules= (line 3429). If found:
  - =ENV_VAR= with =TYPE_SET= → set-as-function dispatch (line 9880).
  - =ENV_VAR= with =TYPE_LIST= and 2 args → list-as-function index
    dispatch (line 9909).
  - =ENV_VAR= with =TYPE_MAP= → map-as-function dispatch (line 9931).
  - =ENV_VAR= with =TYPE_COLL/=TYPE_LIST= → collection indexing via
    dynamic tag dispatch (line 9970).
  - =ENV_VAR= with =TYPE_ARR= and 2 args → array indexing (line 9700).
  - =ENV_FUNC= → typed or closure-ABI call dispatch (line 11800+).
  - Type cast when name matches known type (line 11978-11980):
    Int, Float, Char, String, Hex, Oct, Bin, F32, I8, U8, I16,
    U16, I32, U32, I64, U64, I128, U128.
  - Dot-chain head resolution (line 12533-12619).
  - =(dot base field)= layout field access (line 12623).
  - If nothing matches: =CODEGEN_ERROR "unknown function: …"=
    (line 12675).

[OBS id:obs.codegen-ops.arch.builtins src:codegen.c:13143-13302 conf:high]
  *=register_builtins* (codegen.c:13143-13302) inserts 94 entries into
  the environment via =env_insert_builtin=. All strcmp-dispatch
  operations are also registered here. However, some entries are
  *register_builtins-only* — they have no strcmp handler and are

  Type predicates dispatch via a static table loop (line 5559-5560):
  | Predicate | Tag          |     Tag2 |
  |-----------+--------------+----------|
  | Int?      | RT_INT       |       -1 |
  | Float?    | RT_FLOAT     |       -1 |
  | Number?   | RT_INT       | RT_FLOAT |
  | Char?     | RT_CHAR      |       -1 |
  | String?   | RT_STRING    |       -1 |
  | Bool?     | -3 (special) |       -1 |
  | Symbol?   | RT_SYMBOL    |       -1 |
  | Keyword?  | RT_KEYWORD   |       -1 |
  | List?     | RT_LIST      |       -1 |
  | Ratio?    | RT_RATIO     |       -1 |
  | Set?      | RT_SET       |       -1 |
  | Map?      | RT_MAP       |       -1 |
  | Fn?       | RT_CLOSURE   |       -1 |
  | Hex?      | RT_INT       |       -1 |
  | Bin?      | RT_INT       |       -1 |
  | Oct?      | RT_INT       |       -1 |
  | Arr?      | -2 (special) |       -1 |

[OBS id:obs.codegen-ops.conv.cast-loop src:codegen.c:11979-11981 conf:high]
  Type conversion uses a static table loop (line 11980):
  Int, Float, Char, String, Hex, Bin, Oct, F32, I8, U8, I16, U16,
  I32, U32, I64, U64, I128, U128. Each takes 1 required argument.
  =String= cast is special-cased: string identity, char→string,
  then calls =rt_to_string= on the boxed value. Numeric casts emit
  LLVM trunc/zext/sext/fptosi/sitofp instructions. Type conversions
  are short-circuit: if the argument already has the target type, the
  value passes through unchanged. Errors for non-numeric source types.

[OBS id:obs.codegen-ops.gaps.closures src:codegen.c conf:high]
  *Closures are broken*: =fn=, =lambda=, =defn= have no strcmp
  dispatch handler. No closure allocation exists in the strcmp path.
  =AST_LAMBDA= only works as the value in =(define name (lambda ...))=.
  This blocks higher-order functions: =map=, =filter=, =apply=,
  =partial=.

[OBS id:obs.codegen-ops.gaps.registered-only src:codegen.c:13143-13302 conf:high]
  *Registered-only (no strcmp)*: These are =register_builtins=-only
  — reachable only from wisp-mode or when the reader maps symbol to
  defined function, not from paren-mode:
  - =concat=, =substring=, =reverse=, =nth=, =length=, =reduce=
  - =fn=, =lambda=, =->=, =→=, =λ=, =defn=
  - =cond=, =let=, =let*=, =letrec=
  - =import=, =module=, =export=
  - =match=, =pipe=

[OBS id:obs.codegen-ops.gaps.control-forms src:codegen.c conf:high]
  *Missing control forms*: =cond=, =let=, =let*=, =letrec= have no
  strcmp handler. =when=, =unless=, =do=, =with= DO have strcmp
  handlers. =cond= is treated as an ordinary variable reference.

[OBS id:obs.codegen-ops.gaps.map-runtime src:codegen.c conf:high]
  *Map/Set runtime construction*: Literals like ={:a 1}= and
  ={1 2 3}= produce garbage values without explicit =:: Map= /
  =:: Set= type annotation. Functional (=assoc=, =dissoc=, =merge=)
  always receive garbage inputs. Only =assoc! = / =find= with typed
  variables work.

[OBS id:obs.codegen-ops.gaps.list-variadic src:codegen.c:6794-6801 conf:high]
  *=(list 1 2 3)= broken*: Multi-argument =(list ...)= produces a list
  where =(car (list 1 2 3))= returns nil. Only zero-arg =(list)=
  works. =(make-list n val)= works correctly.

[OBS id:obs.codegen-ops.gaps.tail-segfault src:codegen.c conf:high]
  *=tail= on lists segfaults*: Runtime list implementation bug in
  =rt_list_cdr= or the unboxing chain. =tail= on arrays and strings
  works.

[OBS id:obs.codegen-ops.gaps.superscript src:codegen.c conf:high]
  *Superscript sugar broken*: =x²= desugars to =(^ x 2)=, but =^=
  has no strcmp dispatch handler despite being a registered builtin.
  Upper-case superscript forms similarly broken.

[OBS id:obs.codegen-ops.gaps.inline-asm src:codegen.c:13074-13101 conf:high]
  *Inline asm naked propagation*: Naked inline assembly functions
  compiled via =(define ... (asm ...)) return the first parameter
  only, not the computed sum. The raw asm output has no LLVM
  constraint wiring.

[OBS id:obs.codegen-ops.gaps.and-or-not-uc src:codegen.c:7543,7807 conf:high]

  =and=, =or=, =not= have dispatchers. NOT, XOR listed as expected
  aliases but have no implementation.

[THINK id:think.codegen-ops.strcmp-scaling conf:high]
  The strcmp dispatch architecture in codegen.c is the single largest
  maintenance liability in the compiler. At 83 strcmp handlers spanning
  ~8300 lines, plus 94 register_builtins entries, the linear-scan-if-else
  chain hits O(n) dispatch time on every expression. For a function
  call like =(+ 1 2 3)=, codegen resolves + through a strcmp chain of
  ~160 operations. This is already slow; at 300+ operations it becomes
  untenable. The current architecture works because strcmp is fast on
  short symbol names and the compiler processes one expression at a time,
  but the maintenance cost (inserting a new handler means finding the
  right alphabetical slot in the if-else chain) is the real problem.

[THINK id:think.codegen-ops.strcmp-alternatives conf:high]
  Three alternative dispatch strategies, ranked by implementation effort:
  1. Hash-based dispatch (perfect hash or djb2 on the symbol name):
     O(1) lookup, same portable C, minimal refactor of handler signatures.
     A switch on hash values with collision checking in each case.
     Estimated: 1-2 sessions to implement.
  2. Trie dispatch: prefix-tree on symbol names, O(k) where k is name
     length. More complex to set up but naturally groups related ops
     (e.g., "set-*", "get-*", "vec-*"). The trie could also store arity
     info. Estimated: 2-3 sessions.
  3. Registration table: a flat array of {name, handler_fn} pairs built
     at init time, sorted for binary search or hashed. Eliminates the
     if-else chain entirely. Estimated: 1 session.

[THINK id:think.codegen-ops.builtins-vs-strcmp-gap conf:high]
  The gap between register_builtins (94 entries) and strcmp handlers (83)
  with ~15 builtins unreachable from paren-mode is a code smell. Every
  registered builtin should have a corresponding strcmp handler, or the
  registration is dead code. The 15 orphan builtins likely correspond to
  operations that can only be reached via macro expansion or the typeclass
  dispatch hook (tc_is_method at codegen.c:8907), but this is not
  documented anywhere. A lint pass asserting "every ENV_BUILTIN has a
  strcmp handler" would catch rot early.

[THINK id:think.codegen-ops.three-layer-cost conf:high]
  The three-layer dispatch (layer 1: paren expr ASTType switch, layer 2:
  strcmp on op symbol, layer 3: switch on arg count) is architecturally
  clean but means a single operation like =(set-add s 3)= goes through:
  AST_PAREN → strcmp("set-add", 160 ops) → switch(2 args). Each layer
  is independently tested, but the composition is not — a correctness
  bug at any layer produces silent output corruption (wrong codegen).
  Property-based fuzzing (fuzz-codegen.py) that generates random
  operation + arg combinations would catch layer-boundary bugs.

[THINK id:think.codegen-ops.counts-interpretation conf:high]
  The raw counts (83 strcmp, 94 builtins) suggest the codegen dispatch
  table is larger than typical Lisp/Scheme compilers (which average
  40-60 special forms). Monad's count is inflated by:
  - First-class set/map/vector operations that other languages provide
    via libraries, not the compiler
  - Separate handlers for typed vs untyped variants of the same operation
    (set-add vs set-add/typed)
  - Inline assembly support adding ~10 operations
  - Typeclass dispatch hooks adding ~5 more
  This is a genuine cost of the "batteries-included compiler" philosophy:
  operations that would be library calls in other compilers are strcmp
  handlers in MonadC. The question is whether the boundary between
  "compiler builtin" and "library function" could be drawn differently
  to reduce the dispatch table size without losing performance.
