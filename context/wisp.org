[OBS id:obs.wisp.file src:wisp.c,wisp.h conf:high]
  Wisp is an indentation-based surface syntax that expands to standard
  S-expressions before the reader parses them. Both forms produce identical ASTs.
  Wisp and s-expr can be mixed freely within a single file. The expander is a
  pre-parse transform — it runs before the reader.

[INF id:inf.wisp.info-ref from:obs.wisp.file conf:high]
  User-facing Wisp documentation: [[file:info/wisp.org][info/wisp.org]] covers indentation
  rules, !-sugar, ?-sugar, metadata keywords, and mixing with S-expressions.

[OBS id:obs.wisp.two-pass src:wisp.c conf:high]
  Wisp expansion is a two-pass process:
  1. Tokenization: split source into tokens (parens, brackets, symbols, strings,
     numbers, etc.). Balanced brackets (=~[...]=, =[...]=, ={...}=) are preserved
     as single tokens.
  2. Layout expansion: use indentation and arity tables to insert parentheses
     where needed, converting indentation-driven blocks into S-expression trees.

[OBS id:obs.wisp.output src:wisp.c conf:high]
  The output of wisp expansion is a flat list of standard S-expressions that feed
  directly into =reader.c:'s =parse_expr=. The reader has no awareness that the
  source was originally wisp — it sees only parenthesized forms.

[OBS id:obs.wisp.arity-table src:wisp.c:120-180 conf:high]
  The arity table maps function names to parameter counts. It is used by the
  layout expander to infer correct parenthesization: a known function with arity
  N consumes the next N indented forms as its arguments. Unknown forms fall back
  to generic layout rules.

[OBS id:obs.wisp.arity-sources src:wisp.c conf:high]
  The arity table is populated from three sources:
  - Builtin functions: =wisp_register_arities_from_env= scans the env for
    registered functions and inserts their param counts.
  - FFI C headers: libclang-parsed function signatures from =include= directives
    register their arities during the Phase 0 pre-pass.
  - Layout constructors: struct field counts from =layout= declarations and FFI
    structs are registered as constructor arities.

[OBS id:obs.wisp.arity-ordering src:main.c conf:high]
  Arity registration order matters: FFI arities must be registered before the
  main wisp expansion pass so that imported C functions get correct layout
  inference. =main.c:'s Phase 0 (FFI pre-pass) handles this before Phase 1
  (wisp expansion).

[OBS id:obs.wisp.qmark-sugar src:wisp.c conf:high]
  =?-sugar= (method-call sugar): =(contains? var arg)= transforms to
  =(var contains? arg)= when =var= is an untyped symbol. This enables
  object.method() style calls in wisp. It fires only when the first argument
  is a bare identifier — literals and function calls bypass the transform.
  Workaround: bind the variable with a type annotation to prevent the sugar.

[OBS id:obs.wisp.bang-sugar src:wisp.c conf:high]
  =!-sugar= (mutation-call sugar): =(conj! var arg)= transforms to
  =(var conj! arg)= when =var= is an untyped symbol. Same mechanism and
  constraints as ?-sugar. Affects all !-suffixed builtins: =conj!=, =disj!=,
  =append!=, and any future !-suffixed functions.

[OBS id:obs.wisp.at-sugar src:wisp.c conf:high]
  =@-sugar= (indexing shorthand): array/list indexing via =@= syntax.

[OBS id:obs.wisp.infix-sugar src:wisp.c conf:high]
  Infix detection: =3 + 4= transforms to =(+ 3 4)=. Standard arithmetic and
  comparison operators are recognized as infix when they appear between two
  expressions on the same line.

[OBS id:obs.wisp.pipe-sugar src:wisp.c conf:high]
  Pipe sugar: =x |> f= transforms to =(f x)=. Pipe chains are handled by the
  wisp transformer using the arity table to determine correct grouping.

[INF id:inf.wisp.ffi-arity-flow from:obs.wisp.arity-sources conf:high]
  The FFI pre-pass (=main.c: Phase 0) parses =include= headers via libclang and
  registers every C function's arity with wisp before the main expansion pass.
  This means that =include <raylib.h>= followed by =InitWindow 1920 1080 "Hi"=
  correctly generates =(InitWindow 1920 1080 "Hi")= because wisp knows
  InitWindow takes 3 params.

[INF id:inf.wisp.builtin-arity-flow from:obs.wisp.arity-sources conf:high]
  Builtin arities are registered twice: once from a temporary env before wisp
  expansion (so forms like =define=, =if=, =until= get correct layout), and
  again from the real codegen env during Phase 3.

[OBS id:obs.wisp.env-arity-registration src:wisp.c conf:high]
  =wisp_register_arities_from_env= walks the entire env hash table and
  registers =ENV_FUNC= entries by their param count. =ENV_BUILTIN= entries
  with known arity tables are also registered.

[OBS id:obs.wisp.define-signature-clause-fixed src:wisp.c,tests/wisp/rt_wisp_define_signature_clause.mon conf:high]
  Wisp-style function signatures may take parameter names from a simple
  indented arrow clause:
  #+BEGIN_EXAMPLE
  define square :: Int -> Int
    x -> x * x
  #+END_EXAMPLE
  The expander desugars this to a named-parameter function header and a direct
  body, so =x= is bound as the function parameter rather than treated as an
  unbound pattern variable. Non-simple clauses containing =_=, literals,
  collection patterns such as =[x|xs]=, or guards keep the pattern-clause path.

[DEC id:dec.wisp.ignored-signature-param date:2026-06-27 by:codex]
  A parameter type segment written as =_= in a function signature is an
  ignored-argument contract, not a concrete type. Every pattern clause for
  that parameter position must use wildcard =_=; otherwise reader/Wisp
  expansion reports =ignored signature parameter=. After the contract is
  validated, the annotation is erased so inference accepts any caller argument
  type. Regression coverage:
  =tests.codegen.rt-wisp-ignored-signature-param= and
  =tests.reader.error.signature-ignored-param-bound=.

[OBS id:obs.wisp.typed-value-sugar-fixed src:wisp.c,tests/wisp/rt_wisp_define_typed_value_sugar.mon conf:high]
  Wisp-style =define x :: Type value= is supported as sugar for the Lisp-layer
  bracketed typed binding:
  #+BEGIN_EXAMPLE
  define x :: Int 42
  #+END_EXAMPLE
  expands like =(define [x :: Int] 42)=. The explicit bracketed forms remain
  valid for the Lisp layer and for Wisp code that wants to mirror it directly.

[OBS id:obs.wisp.parenthesized-define-header src:wisp.c,tests/wisp/rt_wisp_define_parenthesized_signature.mon conf:high]
  Bare Wisp accepts a parenthesized Lisp-style function header after =define=:
  #+BEGIN_EXAMPLE
  define (firstItem [n : Int] [xs : Arr] -> Int)
    _ []     -> 0
    n [x|xs] -> x
  #+END_EXAMPLE
  The header is kept as the grouped first argument to =define=, while the
  indented clauses remain attached as the function body. This witnesses that
  pattern clauses match the intended collection argument instead of an earlier
  scalar argument.

[OBS id:obs.wisp.char-escape-newline src:wisp.c,reader.c conf:high]
  Char literals with ='\n'= fail because wisp expands =\\n= to a real newline
  character, splitting the literal across two lines. The paragraph tokenizer
  then sees an orphan closing ='= token on the next line. This is inherent to
  wisp's line-based tokenization — escape sequences that produce whitespace or
  newlines cannot be used inside char literals in wisp mode.

[OBS id:obs.wisp.sugar-first-arg src:wisp.c conf:high]
  Both ?-sugar and !-sugar fire only when the first argument is a bare,
  untyped symbol. This means =(conj! {1 2 3} 4)= works correctly (first arg is
  a literal), while =(conj! s 3)= with untyped =s= becomes =(s conj! 3)=
  (method-call sugar). The workaround is to type-annotate the variable:
  =(define [s :: Set] ...)=.

[OBS id:obs.wisp.wisp-parse-all src:wisp.c:4696-4785 conf:high]
  =ASTList wisp_parse_all(const char *source, const char *filename)=
  Main entry point for the Wisp syntax expander. Called from main.c Phase 1
  before the reader. Algorithm:
  1. Strip comments preserving line/column positions.
  2. Run 2D fraction desugaring (desugar_fractions).
  3. Heuristic scan: if source contains no wisp-style lines (non-paren
     leading tokens), short-circuit directly to parse_all.
  4. Build a local ArityTable seeded from the global FFI arity registry.
  5. Run arity_prescan to register user-defined layout, data, class,
     and define arities from the source.
  6. Build flat WTokenStream via build_token_stream.
  7. Parse all top-level expressions into an S-expression string.
  8. Install param-kind hook and known-function hook for the reader.
  9. Call parse_all + macro_expand_all on the transformed output.
  10. Persist all discovered arities back to the global g_ffi_arities
      so that modules imported later inherit them.

[OBS id:obs.wisp.wisp-register-arity src:wisp.c:84-90 conf:high]
  =void wisp_register_arity(const char *name, int arity)=
  Registers a single function name with its arity in the global FFI arity
  table (g_ffi_arities). Called from the FFI pre-pass (main.c Phase 0) for
  every C function parsed from =include= headers. Lazily initialises the
  static table on first call.

[OBS id:obs.wisp.wisp-register-arities-from-env src:wisp.c:4468-4487 conf:high]
  =void wisp_register_arities_from_env(Env *env)=
  Walks the entire env hash table and registers =ENV_BUILTIN= and
  =ENV_FUNC= entries into the global arity table. Uses =arity_max= as the
  registered arity (-1 = variadic, 0 = fixed with arity_min, otherwise
  use arity_max). Also copies per-parameter ParamKind info for infix
  detection. Called from main.c Phase 0 (with a temporary env) and again
  from Phase 3 (after the real env is populated).

[OBS id:obs.wisp.wisp-clear-arities src:wisp.c:97-102 conf:high]
  =void wisp_clear_arities(void)=
  Frees all entries in the global FFI arity table and resets the init flag.
  Used between compilation units to prevent arity leakage across modules.

[OBS id:obs.wisp.wisp-get-arity src:wisp.c:92-95 conf:high]
  =int wisp_get_arity(const char *name)=
  Looks up a function name in the global FFI arity table. Returns -2 if
  unknown, -99 if the table was never initialised. Called by main.c and
  the reader to query arity for infix detection and layout inference.

[OBS id:obs.wisp.arity-entry-struct src:wisp.c:11-17 conf:high]
  ArityEntry stores a function name, its arity, an array of ParamKind
  values (PARAM_VALUE or PARAM_FUNC) for each parameter, and an array of
  per-parameter function arities (used when a param is itself a function).
  Entries are stored in a singly-linked list (next pointer) within a hash
  bucket.

[OBS id:obs.wisp.arity-table-struct src:wisp.c:21-23 conf:high]
  ArityTable is a fixed-size hash map with 128 buckets (ARITY_BUCKETS).
  Each bucket is a linked list of ArityEntry nodes. The hash function is
  djb2: =h = 5381; while (*s) h = ((h << 5) + h) + *s++; return h % 128=.

[OBS id:obs.wisp.arity-set src:wisp.c:31-42 conf:high]
  =arity_set(table, name, arity)= — Looks up an existing entry by name
  and updates its arity, or allocates a new ArityEntry with default
  param_kinds (PARAM_VALUE) and param_fn_arities (-1) and inserts at
  the head of the bucket. Does NOT check for duplicates.

[OBS id:obs.wisp.arity-set-with-kinds src:wisp.c:44-55 conf:high]
  =arity_set_with_kinds(table, name, arity, kinds)= — Same as arity_set
  but also copies the ParamKind array into the entry. Used when the
  caller has detailed per-parameter type info (from type signatures).

[OBS id:obs.wisp.arity-get src:wisp.c:57-62 conf:high]
  =arity_get(table, name)= — Returns the arity for a name, or -2 if
  unknown. -1 means variadic (match, etc.). Non-negative means fixed
  arity.

[OBS id:obs.wisp.arity-get-entry src:wisp.c:64-69 conf:high]
  =arity_get_entry(table, name)= — Returns the full ArityEntry pointer
  for a name, or NULL. Used by the infix promotion logic to check
  ParamaKind to distinguish function-typed params from value params.

[OBS id:obs.wisp.arity-free src:wisp.c:71-77 conf:high]
  =arity_free(table)= — Frees all ArityEntry nodes and their names,
  zeroes all buckets. Called by wisp_clear_arities and at the end of
  wisp_parse_all to clean up the local arity table.

[OBS id:obs.wisp.arity-prescan src:wisp.c:107-767 conf:high]
  =arity_prescan(table, source)= — Pre-scans the source text (before
  tokenisation) to register arities from user-defined constructs:
  bare =layout=, =data=, =type=, =class=, =define/=def= forms, and
  their s-expression counterparts. Detects bracketed parameters,
  arrow-separated type signatures, constructor fields, class method
  signatures, and function arrow counts. This pass runs inside
  wisp_parse_all after seeding from the global FFI table.

[INF id:inf.wisp.arity-internals from:obs.wisp.arity-entry-struct,obs.wisp.arity-table-struct,obs.wisp.arity-set,obs.wisp.arity-get,obs.wisp.arity-prescan conf:high]
  The arity table uses open hashing (separate chaining) with 128 buckets
  and djb2 hash. There are two instances: the global =g_ffi_arities=
  (persistent across modules) and a local table in wisp_parse_all
  (seeded from the global + builtins + prescan). At the end of
  wisp_parse_all, any newly discovered arities are merged back into the
  global table to benefit subsequent module compilation.

[THINK id:think.wisp.indentation-vs-consistency conf:high]
  Wisp's indentation-sensitive syntax is the most controversial design
  choice in the language. It gives Monad a Python-like readability that
  Lisp-family languages famously lack, but it introduces a whole class
  of whitespace bugs that don't exist in s-expression syntax. A single
  space difference can change the parse tree: =(f x y)= vs =(f x\n  y)=
  are the same in s-exprs but different in wisp (the latter becomes
  =(f x (y))). The arity table mitigates this (if f is known to take 2
  args, both parse the same), but the table is always incomplete —
  higher-order functions and macros have unknown arities. The fundamental
  tension is: wisp trades parse determinism for visual clarity, and the
  arity table is the bridge that tries to recover determinism.

[THINK id:think.wisp.arity-table-coupling conf:high]
  The coupling between wisp and the global arity table (g_ffi_arities)
  is the most architecturally interesting part of the design. The arity
  table is populated by:
  1. Phase 0: C FFI function arities from parsed headers
  2. Phase 3: env-builtin arities from register_builtins
  3. Phase 4: user-defined arities from arity_prescan inside wisp_parse_all
  4. Cross-module: arities discovered in one module persist to benefit
     subsequent modules
  This means the arity table is a shared mutable global that accumulates
  state across compilation units. It's carefully managed (wisp_clear_arities
  between units, local table seeded from global), but a bug in arity
  persistence would produce non-deterministic parses depending on
  compilation order. The fact that this hasn't surfaced as a bug suggests
  the management is correct, but it's a fragile design.

[THINK id:think.wisp.char-escape-newline-edge-case conf:medium]
  The ='\n'= char literal bug (wisp expands \\n to a real newline,
  splitting the char literal across two lines) is a deep edge case that
  reveals a fundamental limitation of wisp's line-based tokenization.
  Fixing it would require wisp to understand string/char escape sequences,
  which means parsing string syntax inside the wisp expander — effectively
  duplicating the reader's string parser. The pragmatic decision to
  document it as a known limitation (use s-expression syntax for char
  literals with escape sequences) is the right call. Not every edge case
  needs a fix; some need a workaround.

[THINK id:think.wisp.sugar-heuristic-first-arg conf:medium]
  The ?-sugar and !-sugar heuristic (fire only when the first arg is a
  bare, untyped symbol) is a compromise between usability and correctness.
  =(conj! {1 2 3} 4)= works because {1 2 3} is a literal, not a symbol.
  =(conj! s 3)= becomes =(s conj! 3) because s is a bare symbol. The
  workaround (type-annotate s) is reasonable but creates a learning
  curve: new users will be confused why (conj! my-set 3) doesn't work
  but (conj! {1 2 3} 4) does. A more robust heuristic would check if
  the symbol is bound to a set-typed variable in the env, but during
  wisp expansion the env isn't fully populated yet.

[THINK id:think.wisp.three-syntaxes-forever conf:high]
  Monad commits to supporting three syntaxes forever: s-expr, wisp, and
  mixed-mode. Each syntax doubles the parser test surface and every
  language feature must work identically in all three. This is a
  significant maintenance commitment. The payoff (widespread adoption
  by Lisp-averse programmers) is hypothetical at this stage. If the
  language gains traction, wisp will be a differentiator. If it remains
  a niche language, maintaining three syntaxes is dead weight. The
  design decision is correct per the philosophy (syntax-is-instrument),
  but it's a bet, not a sure thing.

  connects-to -> `monadc.context.reader` (wisp_parse_all produces S-exprs
  consumed by reader; wisp_register_arity feeds the reader's arity table) ·
  `monadc.context.language` (Wisp expands to canonical S-expr forms that
  correspond to language constructs: define, lambda, if, cond, let, etc.) ·
  `monadc.context.ffi` (Phase 0 in main.c uses FFI pre-pass to register C
  function arities with wisp_register_arity before wisp_parse_all runs) ·
  `monadc.context.macro` (macros are Wisp-form defines with substitution;
  macro output is re-parsed through Wisp) ·
  `monadc.context.main` (Phase 0 of compile_one registers FFI arities into

  `monadc.context.tests-arch` (wisp tests exercise Wisp expansion paths;
  arity-dependent parsing is tested extensively in Wisp codegen tests)

[OBS id:obs.wisp.old-core-coll-signatures src:wisp.c conf:high]
  Old-core style collection signatures such as [a], [Int], and [b] lower to
  boxed Coll in Wisp-generated function headers. Runtime collection helpers
  such as rt_coll_drop, rt_coll_empty, rt_coll_concat, and rt_coll_wrap operate
  on RuntimeValue*, so lowering these signatures to raw Arr creates boxed/raw
  ABI mismatches in recursive collection functions.

[OBS id:obs.wisp.old-core-function-types src:wisp.c conf:high]
  Function parameter and function return positions lower to Fn when Wisp emits
  the canonical signature. This is required for higher-order definitions such
  as predicate-based takeWhile/dropWhile/any?/all? and returned-function
  definitions.

[OBS id:obs.wisp.guard-head-tokenization src:wisp.c conf:high]
  A Wisp pattern head followed by indented guard lines beginning with `|` must
  be emitted raw as a pattern clause head. Otherwise the arity pre-scan can see
  a later expression like `p x` and incorrectly expand an earlier head
  `p [x|xs]` as a call expression.

[OBS id:obs.wisp.guard-comment-preservation src:wisp.c conf:high]
  Guard lines beginning with `|` are source syntax, not comments. The Wisp
  token stream must preserve them so old-core pattern clauses with guards
  compile.

[FIX id:fix.wisp.ignored-signature-param-contract-2026-06-27 src:wisp.c,reader.c,pmatch.c,tests/codegen/errors/reader/signature/atom.reader.error.signature.ignored-param-bound/signature-ignored-param-bound.mon,tests/codegen/branches/wisp-codegen/define-and-types/atom.codegen.80.wisp.codegen.define.and.types.rt.wisp.ignored.signature.param/rt_wisp_ignored_signature_param.mon conf:high]
  A function signature parameter written as =_= is a semantic ignored-argument
  contract. Reader/pmatch validation requires every pattern clause at that
  argument position to use wildcard =_=; binding or destructuring it is a
  compile-time error. Wisp simple-clause lowering preserves this contract before
  promotion to canonical lambda/pmatch forms.
