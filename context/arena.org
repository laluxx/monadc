[OBS id:obs.context.arena.file src:arena.h conf:high]
  This file records the full arena allocator API: a fast bump-pointer allocator
  organised as a singly-linked list of fixed-size blocks. Allocation is O(1);
  freeing the entire arena is O(blocks). Supports scratch (reset/reuse) and
  permanent (free once) usage patterns plus checkpoint/restore for stack-like
  rollback.

[OBS id:obs.arena.arena-init src:arena.h:66 conf:high]
  arena_init initialises an Arena struct. If block_size is 0, ARENA_DEFAULT_BLOCK
  (4 MiB) is used. No OS memory is allocated until the first arena_alloc call.

[OBS id:obs.arena.arena-free src:arena.h:70 conf:high]
  arena_free frees ALL memory owned by the arena by walking the block linked list
  and calling free on each block. The Arena struct itself is not freed — callers
  may reuse it with arena_init after freeing.

[OBS id:obs.arena.arena-reset src:arena.h:74 conf:high]
  arena_reset rewinds all blocks to empty (used = 0) but keeps OS memory mapped.
  This is the fast path for scratch arena reuse: zero mallocs in the common case.

[OBS id:obs.arena.arena-alloc src:arena.h:80 conf:high]
  arena_alloc bumps the pointer by `size` bytes (rounded up to ARENA_ALIGN).
  If the current block cannot satisfy the request, a new block is allocated and
  prepended to the list. Never returns NULL — calls abort() on OOM (same contract
  as xmalloc).

[OBS id:obs.arena.arena-calloc src:arena.h:83 conf:high]
  arena_calloc allocates `size` bytes and zeroes them via memset.

[OBS id:obs.arena.arena-memdup src:arena.h:86 conf:high]
  arena_memdup allocates a copy of `src` of `size` bytes.

[OBS id:obs.arena.arena-strdup src:arena.h:89 conf:high]
  arena_strdup duplicates a NUL-terminated string into the arena. Uses strlen +
  memcpy internally.

[OBS id:obs.arena.arena-strndup src:arena.h:92 conf:high]
  arena_strndup duplicates at most `n` bytes of a string. Always NUL-terminates
  the result, so the allocated buffer is at most n+1 bytes.

[OBS id:obs.arena.arena-sprintf src:arena.h:95-96 conf:high]
  arena_sprintf formats a string using printf-style varargs into arena memory.
  Returns a pointer to the NUL-terminated result. The underlying buffer is
  allocated with arena_alloc after determining the required length via vsnprintf.

  ArenaCheckpoint struct.

[OBS id:obs.arena.arena-restore src:arena.h:106 conf:high]
  arena_restore rewinds the arena to a previously saved checkpoint: restores
  the active block and its used offset, then frees any blocks that were allocated
  after the checkpoint. Only safe when no blocks were freed between save and
  restore (which is always true for this bump-pointer allocator — free is
  whole-arena only).

[THINK id:think.arena.bump-pointer-tradeoff conf:high]
  The bump-pointer arena allocator is the right choice for a compiler
  workload. Compiler data structures (AST nodes, symbol tables, type
  objects) are allocated in phases, live for the duration of a
  compilation unit, and are freed all at once. The bump allocator
  converts malloc overhead (lock, freelist search, fragmentation) into
  a single pointer increment per allocation. For a compiler processing
  tens of thousands of AST nodes, this is a measurable performance win.
  The tradeoff is memory waste: if a 4 MiB block has 3.9 MiB unused
  at the end of compilation, that memory cannot be reclaimed by other
  processes. In practice, the OS reclaims it when the compiler exits,
  so it's only a concern for long-running processes (REPL, language
  server), where checkpoint/restore provides the necessary control.

[THINK id:think.arena.checkpoint-restore-pattern conf:high]
  The checkpoint/restore mechanism (arena_checkpoint returns an index,
  arena_restore pops back to it) is a stack-based rollback that enables
  scratch allocation within a compilation phase without permanent memory
  growth. This is particularly valuable for the macro expander and HM
  inference, which allocate speculative type variables and backtrack.
  The stack discipline (you can only restore to a previous checkpoint,
  not an arbitrary point) is both a strength and a limitation: it
  guarantees O(1) restore but forbids non-LIFO allocation patterns. In
  practice, compiler algorithms are naturally stack-like (nested
  inference contexts, recursive macro expansion), so the limitation
  rarely bites.

[THINK id:think.arena.4mb-block-sizing conf:medium]
  The 4 MiB default block size with growth factor 2 is a reasonable
  heuristic but untuned. 4 MiB is large enough that most compilations
  fit in a single block (avoiding linked-list traversal for allocation),
  but small enough that a 256-block chain handles a terabyte of
  allocation before wrapping around the index counter. For a self-hosted
  compiler compiling itself, the peak allocation is likely in the
  10-50 MiB range, meaning 2-3 blocks. The growth factor of 2 means
  the second block is 8 MiB, the third 16 MiB, etc. — exponential
  growth is appropriate for compiler workloads where the next compilation
  might be arbitrarily larger.

[THINK id:think.arena.scratch-vs-permanent conf:high]
  The scratch/permanent arena distinction is a classic technique from
  compiler construction (see Wirth's Oberon compiler). The permanent
  arena holds the AST and symbol table (lifetime = whole compilation),
  while the scratch arena holds intermediate results (type variables,
  constraint sets, macro expansion intermediates). The scratch arena
  is checkpointed before each major phase and restored after, ensuring
  that temporary allocations don't leak into the permanent arena. This
  design is simple and correct but requires discipline: a developer
  working on inference must know which arena to use for each allocation.
  An arena-tagged pointer type (like Rust's arena references) would
  make this safer but is not expressible in C without macros.

[THINK id:think.arena.gc-interaction conf:medium]
  The arena allocator is used both by the compiler (for its own data
  structures) and by the runtime (for GC-managed heap objects via
  runtime.c). This dual use is convenient (one allocator for everything)
  but creates a subtle issue: compiler data allocated in the permanent
  arena persists across GC collections, but GC-managed objects must
  be movable (compacting GC) or at least traceable. If the runtime
  ever implements a compacting GC (moving objects to defragment the
  heap), arena-allocated objects cannot be moved. The current design
  avoids this by using a non-moving mark-sweep GC for runtime objects
  and arena allocation only for compiler-internal data, but the
  distinction is not enforced by the type system.

#+CAPTION: Arena Allocator Category Contract
| Field | Contract |
|-------+----------|
| Owned phase | Bump-pointer allocation for compiler data structures, scratch/permanent arena separation, checkpoint/restore for speculative work (e.g. HM inference). |
| Inputs | Allocation requests (size, alignment), arena selection (scratch vs permanent), checkpoint token. |
| Outputs | Allocated memory regions, arena checkpoint tokens for restore, freed pages on arena reset. |
| Invariants | Permanent arena is never freed during a compilation unit; scratch arena page-aligned; checkpoint/restore is LIFO; scratch arena may be reset between inference passes; runtime GC objects use a separate non-moving mark-sweep heap, not the compiler arena. |
| Test hooks | Arena stress tests (large/frequent allocations), alignment tests, checkpoint/restore round-trip tests, leak detection across reset boundaries. |
| Failure symptoms | Arena exhaustion (OOM), checkpoint mismatch (restore called without matching checkpoint), use-after-free across checkpoint boundary, accidental allocation in wrong arena. |
| Context neighbors | runtime, source-map, infer, types, reader |
