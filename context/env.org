  =Env= holds =buckets= (=EnvEntry **=), =size=, =count= for the hash table.
  =parent= (=struct Env *=) links to the enclosing scope for lexical

[THINK id:think.context.env.layout-field-resolve date:2026-06-13 by:codex]
  =env_find_layout_with_field= must linear-scan every =ENV_LAYOUT= entry
  because fields are not indexed by field name.  For large numbers of
  layouts this is O(n*m).  A field-name → layout reverse index could be
  added if profiling shows this as a bottleneck.

[THINK id:think.context.env.infer-scheme-redundancy date:2026-06-13 by:codex]
  An =EnvEntry= carries both =type= (the declared or ground type) and
  =scheme= (the quantified HM type scheme).  These are redundant for
  monomorphic bindings where =scheme= is a trivial quantifier.  The dual
  representation exists because =type= is the fast path for codegen while
  =scheme= is needed for let-polymorphism at call sites.

[DOC id:doc.env.category-contract src:context/env.org conf:high]
  This section declares the category contract for =env=. It is the
  routing target for failures and future test/context links touching the symbol
  table, scope management, and environment lookup machinery.

#+CAPTION: Symbol Table (Env) Category Contract
| Field | Contract |
|-------+----------|
| Owned phase | Environment/scope management, symbol insertion and lookup, EnvEntry storage with dual type/scheme representation, builtin registration, layout field resolution. |
| Inputs | Symbol names, EnvEntryKinds (ENV_VAR, ENV_FUNC, ENV_LAYOUT, etc.), Type/Term pointers, scope chain from parser/inference. |
| Outputs | EnvEntry lookups for symbols, scoped name resolution results, scheme retrieval for let-polymorphism, builtin function entries. |
| Invariants | Scope chain is consistent (parent pointers valid); symbols are unique within each scope; builtins are registered before any use; env_find_layout_with_field linear scan is correct though O(n*m); exported functions have is_exported flag set. |
| Test hooks | Symbol resolution tests, scope nesting tests, builtin registration order tests, layout field resolution tests. |
| Failure symptoms | Symbol not found despite being defined, ambiguous name across scopes, scope leak (symbol visible outside its scope), builtin registration collision, layout field resolution failure. |
| Context neighbors | infer, codegen-ops, typeclass, module, runtime |

[DEC id:dec.env.category-contract-owner date:2026-06-18 by:big-pickle]
  Failures matching the symptoms above should start triage in this category, then
  follow the listed neighbors only when the local contract does not explain the
  issue.
