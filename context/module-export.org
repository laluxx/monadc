[OBS id:obs.module-export.file src:module_export.h conf:high]
  This file documents the full API of module_export.h — the export manifest
  subsystem that records every exported symbol (variables and functions) with
  type information in a plain-text .mexp sidecar file. The manifest bridges
  compilation units: module A writes Foo.mexp, module B reads it to emit
  correct extern declarations for imported symbols without re-parsing the
  original source.

[OBS id:obs.module-export.mangling-convention src:module_export.h:36,84 conf:high]
  Mangled names follow the pattern =Module__symbol= (double underscore).
  The mangling in module_export.h (manifest_mangle) mirrors the LLVM symbol
  renaming in main.c Phase 9. Both must use the same convention so that
  extern declarations in the importing module reference the correct linker
  symbols.

[INF id:inf.module-export.mangling-flow from:obs.module-export.mangling-convention conf:high]
  The double-underscore convention is used for both the .mexp file records
  and the LLVM symbol names. When module B imports from module A, it reads
  A's .mexp file and emits =extern= declarations using the mangled names,
  which must match the symbols A's .o exports.

[THINK id:think.module-export.mexp-format-choice conf:high]
  The decision to use TOML for .mexp manifest files is pragmatic and
  correct. TOML is human-readable (unlike binary formats), easy to parse
  (unlike YAML, which has edge-case ambiguities), and well-supported by
  C libraries (toml.c is a single-file, MIT-licensed parser). The format
  is simple enough that the manifest can be hand-edited for debugging,
  and structured enough that the compiler can reliably parse it. JSON
  would also work but lacks comments (useful for debugging). S-expressions
  would be consistent with the language's syntax but harder to parse
  without the full reader. TOML is the pragmatic middle ground.

[THINK id:think.module-export.symbol-mangling-design conf:high]
  The name mangling scheme (package.module.symbol → flat symbol) follows
  the same approach as Java (package.Module.method) and Rust
  (module::symbol). The use of a name-mangling buffer and a character-level
  rewriter is straightforward and correct. The key design choice is
  whether mangling is reversible (can you recover the original qualified
  name from the mangled symbol?) — reversible mangling enables better
  debug info and error messages. The current scheme (flat concatenation
  with delimiters) is reversible by construction. This is the right
  choice: it costs nothing and pays dividends when debugging.

[THINK id:think.module-export.cache-invalidation-missing conf:medium]
  The .mexp file serves as a cache of the module's public interface,
  but there is no versioning mechanism. If the mangling scheme changes,
  old .mexp files produced by a previous compiler version will produce
  wrong external declarations. A compiler version stamp in the .mexp
  header would detect this: if the compiler version changes, all cached
  .mexp files are invalidated and regenerated. Without this, a
  self-hosting compiler that changes its own mangling scheme will fail
  to recompile itself from clean state (because old .mexp files conflict
  with the new output). This will surface during bootstrap.

[THINK id:think.module-export.manifest-vs-module-decl conf:medium]
  The dual declaration of exports (=module export ...= in source and
  .mexp on disk) creates a potential for divergence. The source-level
  declaration is the authoritative definition; the .mexp is a cache.
  If the .mexp is stale (e.g., from a partial build), the compiler
  silently uses stale export information. A simple timestamp comparison
  (.mexp vs .mon source) in the manifest reader would catch staleness
  and trigger a rebuild. This is the same pattern used by the build
  system for .o files and should be extended to .mexp files for
  consistency.

[THINK id:think.module-export.no-re-export-mechanism conf:medium]
  The current module export system does not support re-exports (module
  A imports symbol from B and re-exports it as if it were A's own).
  This means every module that uses a symbol from B must directly import
  B — there is no transitive dependency hiding. For small projects this
  is fine. For large projects with deep dependency trees, the lack of
  re-exports means changes to B force recompilation of every module
  that depends on B, even if the dependency is indirect. Adding re-export
  syntax (=module export { ... } from B=) would enable facade modules
  that provide stable interfaces over volatile dependencies.

[INF id:inf.module-export.mexp-role from:obs.module-export.mexp-format,obs.module-export.mangling-convention conf:high]
  The .mexp file is the serialisation bridge between compilation units.
  Unlike header files or module interfaces, .mexp is a machine-generated
  sidecar that contains exactly the symbol surface needed for cross-module
  extern declarations — no parsing required. This enables incremental
  compilation: only recompiled modules need their .mexp regenerated.
