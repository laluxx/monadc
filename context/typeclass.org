[OBS id:obs.typeclass.instance-reg.fn src:typeclass.c:258-503 conf:high]
  =tc_register_instance(reg, ast, ctx)= = — Called when =AST_INSTANCE= is
  encountered (=codegen.c:13068=). Three-step process:
  1. Compile each method: look up body in AST, fall back to default, clone lambda,
     substitute type variables with concrete types (=type_var= → =type_name=,
     =assoc_names= → =assoc_values=), parse function signature via
     =my_type_parse_fn_arrow=, annotate lambda params, compile via =codegen_expr=
     as =(define __impl_Class_Type_method lambda)=, store in =method_funcs=.
  2. Build dictionary global: get or create =__dict_Class_Type= LLVM global struct
     of =method_count= function pointers. Bitcast each method function to =i8*= and
     store at the appropriate GEP slot.
  3. Register in env: for each compiled method, insert an =ENV_FUNC= entry (2-param
     closure ABI, types unknown) so =tc_is_method= dispatch works.

  1. Check if the operation symbol (=op=) is a known method via =tc_method_class=.

[THINK id:think.typeclass.dict-passing-approach conf:high]
  The dictionary-passing implementation (instance dict struct with method
  function pointers) is the standard implementation strategy for
  typeclass dispatch, used by Haskell (GHC) and Rust (trait objects).
  MonadC's approach of storing the dict alongside the instance in the
  registry and emitting calls via GetElementPtr + BitCast follows the
  same pattern. The key difference from GHC is that MonadC's instance
  resolution happens at codegen time (strcmp dispatch checks tc_is_method),
  not at type-checking time. This means a type error in instance
  resolution produces a codegen error rather than a type error, which
  is unusual but workable — the phase ordering means type errors should
  be caught before codegen anyway.

[THINK id:think.typeclass.instance-resolution-scope conf:high]
  MonadC's instance resolution appears to be local (visible instances
  at the point of use) rather than global (all instances in scope, like
  Haskell's). This avoids the "orphan instance" problem that plagues
  Haskell (instances defined in module A for types defined in module B,
  where A and B don't know about each other). The cost is that the same
  type may have different instance resolutions in different modules,
  potentially producing different method implementations for the same
  (type, class) pair — a form of incoherence. MonadC avoids this by
  resolving instances at codegen time within a single compilation unit,
  where all imports are resolved and all instances are known.

[THINK id:think.typeclass.feature-gap conf:high]
  The current typeclass system is minimal compared to Haskell's: no
  associated types, no superclass constraints (class Eq a => Ord a),
  no default method implementations, no instance deriving. Each of
  these features adds significant complexity:
  - Associated types require type-level computation during instance
    resolution, pulling in the dependent type checker
  - Superclass constraints require the solver to build a dependency
    graph of instance dictionaries
  - Default methods require method bodies that can be overridden
  - Deriving requires code generation for standard instances (Eq, Ord,
    Show) based on the type structure
  The current minimal design is the right starting point — adding these
  features later is easier than removing overengineering.

[THINK id:think.typeclass.codegen-perf-characteristics conf:medium]
  The dictionary-passing implementation has predictable performance:
  each method call is an indirect function call through the dict (load
  dict ptr, GEP to method slot, load fn ptr, call). This is ~3 loads
  more than a direct call, which is negligible for most code but matters
  in hot loops. GHC's specialization pragmas (SPECIALIZE) generate
  direct-call variants for known concrete types. MonadC's monomorphization
  cache should naturally produce specialized variants when the concrete
  instance type is known at compile time, achieving the same effect
  without explicit pragmas. This is worth verifying: if the cache
  already specializes, the typeclass system gets zero-cost abstraction
  for free.

[THINK id:think.typeclass.strcmp-hook-implication conf:medium]
  The tc_is_method hook in the strcmp dispatch means every comparison
  operation (==, !=, <, >, <=, >=) first checks whether the symbol is
  a class method, and only falls through to the numeric/string comparison
  if it's not. This adds overhead to every comparison in the program,
  even those that don't use typeclasses. For a hot loop with integer
  comparisons, this is an unnecessary strcmp call per comparison. The
  overhead is bounded (one extra strcmp per comparison operation) and
  LLVM is likely to inline the check in optimized builds. But if
  typeclass dispatch is rare (which it will be until the standard
  library uses typeclasses heavily), the hook imposes a tax on every
  comparison for a feature most code doesn't use.

| Field | Contract |
|-------+----------|
| Owned phase | Class declaration registration, instance declaration registration, method dispatch via tc_is_method hook in strcmp dispatch, dictionary pass construction for polymorphic method calls. |

| Outputs | Instance method resolution, dispatched calls through tc_is_method hook, zero-cost abstraction via monomorphization (codegen already specializes by type). |
| Invariants | At most one instance per class+type combination; instance resolution terminates; tc_is_method hook checked before each comparison operation; monomorphization provides zero-cost dispatch without runtime dictionary overhead. |
| Test hooks | Type class test suite, method dispatch correctness tests, instance resolution tests, overlap detection tests, performance overhead benchmarks for tc_is_method path. |
| Failure symptoms | Ambiguous instance detection, missing instance for required method call, method dispatch routed to wrong instance, tc_is_method overhead on hot comparison loops, instance registration collision. |
| Context neighbors | codegen-ops, infer, env, language, tests |
