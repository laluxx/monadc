[OBS id:obs.context.types.file src:types.h,types.c conf:high]
This file documents the MonadC type representation: the =TypeKind= enum (40
values), the =Type= union-with-tag struct, supporting structs
(=FnParam=, =LayoutField=, =RefinementEntry=, =TypeAlias=), all
constructor/operation functions, refinement types, type aliases, annotation
parsing, and layout offset computation. This is the central type description
used by every compiler phase: reader, infer, codegen, and dep.

The four missing slots were never materialised: TYPE_VARIADIC (enum-only,
no constructors), and three others that remain available for future
collection-type specialisation. The enum is deliberately sparse so that new
kinds can be added without renumbering downstream switch statements.

[DOC id:doc.types.int-arbitrary-rationale date:2026-06-13 by:codex]
INT_ARBITRARY exists because the language wants to support bit-exact integer
types (e.g. =U4= for nibbles, =I1= for signed booleans) without needing 128
separate enum values. The 1–64 plus 128 constraint matches common hardware
widths. Width validation occurs in =parse_int_type=.

*** TYPE_ARR
:PROPERTIES:
:ID: monadc.context.types.kind.TYPE_ARR
:CUSTOM_ID: kind-TYPE_ARR
:CONTEXT_KIND: typekind
:CONTEXT_DESCRIPTION: Array type — fixed-size, fat pointer, or heap-growable
:CONTEXT_VERSION: 1
:SOURCE: types.h:21,types.c:493-507
:CONFIDENCE: high
:END:

[OBS id:obs.types.kind.TYPE_ARR src:types.h:21 conf:high]
Array type with three variants selected by boolean flags:

  - =arr_is_fat=false, arr_is_heap=false, arr_size>=0= → fixed-size array
    ={T data[sz]}=
  - =arr_is_fat=true= → runtime fat pointer ={T* data, i64 size}=
  - =arr_is_heap=true= → heap-growable ={T* data, i64 size, i64 cap}=

Uses =arr_element_type= for the element type. Default =arr_size=-1 for
variants where size is unknown at compile time.

*** TYPE_FN
:PROPERTIES:
:ID: monadc.context.types.kind.TYPE_FN
:CUSTOM_ID: kind-TYPE_FN

[OBS id:obs.types.kind.TYPE_FN src:types.h:22 conf:high]
Concrete function type with full parameter metadata. Stores =params= array
of =FnParam=, =param_count=, and =return_type=. Supports optional parameters
and variadic rest params ('. _'). Distinguished from =TYPE_ARROW= which is

[DOC id:doc.types.TYPE_FN-vs-ARROW date:2026-06-13 by:codex]
TYPE_FN and TYPE_ARROW serve different purposes: TYPE_FN carries full
parameter metadata (names, optional, rest) for concrete function declarations,
while TYPE_ARROW is a simple param→ret pair used during HM type inference.
Codegen treats them uniformly for closures, but inference uses only TYPE_ARROW.

*** TYPE_LAYOUT
:PROPERTIES:
:ID: monadc.context.types.kind.TYPE_LAYOUT

[OBS id:obs.types.kind.TYPE_LAYOUT src:types.h:23 conf:high]
C-struct or record type. Stores =layout_name=, =layout_fields= array,
=layout_field_count=, =layout_total_size=, =layout_packed= flag,
=layout_align= constraint, =layout_is_scalar= flag (for opaque typedefs like

Codegen uses =layout_compute_offsets= to compute field offsets.

[DOC id:doc.types.union-with-tag date:2026-06-13 by:codex]
The Type struct uses a union-with-tag pattern rather than a C union or
separate structs per kind. This means all fields coexist in one struct,
simplifying allocation (single =calloc=) and freeing (one =free=), but
requiring discipline to read only the fields relevant to =kind=. The
alternative (nested =union=) would save memory but complicate the
constructor/accessor pattern. The ~14-pointer size (~112 bytes on 64-bit)
is acceptable for the typical working set.

  - TYPE_FN frees param names, param types, the params array, and return_type.
  - TYPE_LIST frees each list_type element and the array.
  - TYPE_PTR / OPTIONAL / COLL free element_type.

[DOC id:doc.types.refinement-system date:2026-06-13 by:codex]
  Refinement types are a lightweight static contract system. A refinement type
  (e.g. "Even", "Pos") wraps a base type (e.g. "Int") with a predicate AST
  that is evaluated statically at compile time when possible. The =refinement_check_literal=
  function walks alias chains and evaluates the predicate AST against a literal
  value. Predicates support arithmetic ops (+, -, *, mod), comparison ops
  (=, !=, >, >=, <, <=), logical ops (and, or, not), and recursive refinement
  lookups via naming conventions (e.g. "Even?" look up refinement "Even").

** refinement_check_literal
:PROPERTIES:
:ID: monadc.context.types.refinement.refinement_check_literal
:CUSTOM_ID: ref-refinement_check_literal
:SOURCE: types.h:223,types.c:196-224

[OBS id:obs.types.refinement.refinement_check_literal src:types.c:196-224 conf:high]
=int refinement_check_literal(type_name, val, out_pred_src)=
Evaluates a refinement predicate against a numeric literal value. Walks the
alias chain (up to 32 steps) resolving through =TypeAlias= entries until it
finds a =RefinementEntry= with a predicate AST, then invokes
=static_eval_pred=. Returns 1 (true), 0 (false), or -1 (cannot evaluate).

=static_eval_pred= recursively evaluates the predicate AST:

  - (op lhs rhs) — arithmetic ops return the computed value; comparison ops
    return 0/1; logical and/or short-circuit via recursive evaluation.
  - (not expr) — negates.
  - (Name? arg) — strips trailing '?', looks up registered refinement,
    recurses.
  - Bare variable → truthiness of val.
  - Bare number → truthiness of number.
  - Returns -1 for anything it cannot evaluate.

=int layout_compute_offsets(fields, count, packed, elem_size_fn)=
Computes byte offsets and sizes for =LayoutField= arrays. Algorithm:

  1. If =fields[i].size= is already set (codegen path), use it directly.
  2. Else if =elem_size_fn= is provided, invoke it with =fields[i].name=.

  6. Increment offset by size.
  7. Return total struct size.

=Type *infer_literal_type(double value, const char *literal_str)=
Infers the type of a literal value. Decision order:

  1. No literal_str: if value is integer → TYPE_INT, else TYPE_FLOAT.

hex prefix. The current ordering is correct but fragile — a more robust
approach would pre-scan for the radix prefix before the slash check.

** type_from_name
:PROPERTIES:
:ID: monadc.context.types.util.type_from_name
:CUSTOM_ID: util-type_from_name
:SOURCE: types.h:213,types.c:229-409

[OBS id:obs.types.util.type_from_name src:types.c:229-409 conf:high]
=Type *type_from_name(const char *name)=
Resolves a type name string to a Type. Resolution order:

  1. Leading '*': remove prefix, resolve inner, wrap in =type_ptr=.
  2. Single lowercase letter ('a'-'z'): create TYPE_VAR with id 2000+offset.
  3. Parenthesised "(T1 T2 ...)": parse as TYPE_LIST with element types.
  4. Bracketed "[T]": parse as TYPE_COLL with element type.
  5. Trailing '?': strip, resolve inner, wrap in =type_optional=.
  6. "Fn :: (A -> B -> ...)": delegate to =type_parse_fn_arrow=.
  7. "Pointer :: T": resolve inner, wrap in =type_ptr=. Unknown → =type_layout_ref=.
  8. "Arr :: Elem :: N", "Arr :: Elem", "Arr :: N :: Elem": parse as TYPE_ARR.
  9. Named builtins: match against "Int", "Float", ..., "Heap" (36 names).
  10. =parse_int_type=: try I<n> or U<n> arbitrary-width.
  11. Alias chain: resolve through =g_aliases= up to 32 steps, recursively
      calling =type_from_name= on each target.
  12. Two-word "Constructor Arg": split on space, create TYPE_APP.
  13. Returns NULL if unresolvable.

[THINK id:think.types.TYPE_COLL-vs-concrete status:open date:2026-06-13 by:codex]
TYPE_COLL represents an abstract collection "[T]" but SET and MAP are separate
ground types. Should SET and MAP be unified under TYPE_COLL with a subtype
field? Or should TYPE_COLL gain a collection_kind enum to distinguish List vs
Set vs Map? Currently =type_from_name= for "[T]" always creates TYPE_COLL, so
there is no way to say "Set of Int" in annotation syntax. This could be
addressed by adding =collection_kind= to Type or by expanding the annotation
syntax to =#{} T= for sets.

[THINK id:think.types.TYPE_VARIADIC-first-class status:open date:2026-06-13 by:codex]
TYPE_VARIADIC is defined in the enum but never constructed. The variadic
behaviour is encoded in FnParam.rest. If the language gains higher-order
variadic functions (e.g. map with variadic args), TYPE_VARIADIC would need
a proper constructor and operations. Consider whether it should wrap a
TYPE_LIST or be a standalone type.

[THINK id:think.types.arr_is_fat-vs-heap-exclusive status:open date:2026-06-13 by:codex]
TYPE_ARR has two boolean flags: arr_is_fat and arr_is_heap. Can both be true?
In =type_arr_heap=, arr_is_fat is explicitly set to false. But the type
system doesn't enforce mutual exclusion at any other point. Could a heap array
also be a fat pointer? The runtime representation would be different — fat is
{data, size}, heap is {data, size, cap}. If both were true, which layout wins?

[DOC id:doc.types.no-nested-union date:2026-06-13 by:codex]
The Type struct deliberately avoids a C union for its variant fields. A union
would save ~80 bytes per struct but would require every field access to check
kind and would complicate =calloc=-based zero-initialisation. The current
design wastes memory (~112 bytes per Type) but eliminates an entire class of
union-related bugs. The memory overhead is acceptable because the number of
live Type structs is bounded by the AST size.

[DOC id:doc.types.arr_size-minus-one-default date:2026-06-13 by:codex]
=make_type= always sets =arr_size=-1 by default, even for non-array kinds.
This means ground types technically have =arr_size=-1, which is harmless
because =types_equal= and =type_to_string= switch on =kind=. But it's a
potential footgun if new code checks =arr_size= without checking =kind= first.
A safer design would zero-initialise and set =arr_size= only in array
constructors.

[DOC id:doc.types.layout_fields-sharing date:2026-06-13 by:codex]
TYPE_LAYOUT's =layout_fields= array is shared between the registry and Type
structs created via =type_layout=. =type_clone= does not deep-copy
layout_fields, and =type_free= does not free them. This is intentional: the
fields array is owned by the layout registry (or the struct definition
storage). Any code that creates a TYPE_LAYOUT outside the registry must manage
the fields array lifetime separately.

[DOC id:doc.types.string-format-thread-safety date:2026-06-13 by:codex]
=type_to_string= uses a 32-slot rotating buffer to avoid the classic
single-static-buffer thread-safety problem. With 32 slots, up to 32 nested or
concurrent calls are safe before reuse. This is sufficient for typical
recursive =type_to_string= calls (max depth rarely exceeds 8). However, it is
not safe for truly concurrent calls from multiple threads without external
synchronisation.

* Design Decision: 40 (Not 45) TypeKinds

[DOC id:doc.types.40-kinds date:2026-06-13 by:codex]
The TypeKind enum contains 40 values, not 45. Earlier design documents
anticipated 45 but five slots were never needed:

  1. TYPE_VARIADIC exists in the enum but has no constructor — variadic
     behaviour is encoded in FnParam.rest.
  2. Four unnamed slots were speculated for future collection-type
     specialisations but never materialised.

The enum is sparsely ordered to allow insertion of new kinds between
existing groups without renumbering. New numeric types go between the
existing numeric range; new compound types go after COLL.

The 40 kinds break into:
  - 13 ground types (general literals and primitives)
  - 13 numeric subtypes (explicit-width ints, floats)
  - 9 compound types (collections, functions, pointers)
  - 5 meta types (HM variables, arrows, unknown, application)
