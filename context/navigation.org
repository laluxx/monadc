[OBS id:obs.navigation.purpose src:context/navigation.org conf:high]
  This file is a routing table for LLMs. Given a context ID like
  =monadc.context.reader.sugar.percent=, read the prefix to find which file
  contains the node. Read this file first before writing new records.

* How to Use

1. To find a node: match the ID's second segment against the table above.
2. To add a new record: use the prefix that matches the topic's file.
3. If no prefix matches, use =context/index.org= and add a new prefix entry here.
4. For new topics: add a row to this table first, then create the file.

[THINK id:think.navigation.routing-table-overhead conf:high]
  The routing table (ID prefix → file) is the most obvious candidate for
  auto-generation. Every time a new context file is added, the routing
  table must be manually updated — and forgetting to do so produces an
  "ID prefix not found" error that is confusing for both humans and LLMs.
  The table is trivially derivable from the file listing (each file's
  =CONTEXT_ID_PREFIX= property maps to its path). An auto-generation
  script (say, =make context-nav= that runs a shell one-liner) would
  eliminate this maintenance burden entirely. The file would still be
  committed to git (for offline reference), but regenerated as a
  pre-commit hook or before each development session.

[THINK id:think.navigation.prefix-collision-risk conf:medium]
  The flat ID prefix scheme (monadc.context.<segment>) relies on segment
  uniqueness across all files. With 50+ files and a single-level prefix,
  collisions are inevitable. The prefix "types" maps to types.org, but
  "tests" maps to tests.org and "tests-arch" maps to tests-arch.org.
  If a file named "tests-meta.org" were added, its prefix would be
  ambiguous with "tests" (both start with "tests-*"). A hierarchical
  prefix scheme (monadc.context.compiler.*, monadc.context.tests.*,
  monadc.context.lang.*) would prevent ambiguity and make the routing
  table navigable by category. This is a restructing effort but would
  fix the collision problem permanently.

[THINK id:think.navigation.llm-navigation-overhead conf:high]
  For an LLM agent working with this corpus, the routing table is the
  most frequently consulted file — it answers "which file contains this
  topic?" before every read operation. The table's current flat format
  (alphabetical by prefix) is LLM-friendly: a quick scan finds the
  target prefix, and the file path is right next to it. The table's
  usefulness degrades as the corpus grows past ~50 entries (the scan
  takes longer). At ~75 entries, a categorized table (Compiler Core vs
  Language vs Tests vs Tools) would be faster to navigate than a flat
  list. The categories should match the portal sections in context.org
  for consistency.

[THINK id:think.navigation.id-first-workflow conf:medium]
  The convention of "add the routing table entry before creating the
  file" (=DEC id:dec.navigation.routing-table=) is a lightweight
  reservation system that prevents ID collisions. Creating the table
  entry first reserves the ID prefix, and creating the file with that
  prefix completes the reservation. This is exactly the right workflow
  for a context corpus where multiple agents (human + LLM) may add
  files concurrently. It is analogous to creating an interface before
  implementing it — the routing table entry is the declaration, the
  context file is the definition.

[THINK id:think.navigation.limits-of-flat-routing conf:medium]
  The current routing table assumes a bijective mapping (one prefix maps
  to one file). This works well for the current corpus but will break
  as files grow larger. Already, language.org covers multiple topics
  (syntax, constructs, types, operations, compilation pipeline) under
  a single "language" prefix. If language.org were split into
  language-syntax.org and language-types.org, the routing table would
  need a many-to-one mapping (two prefixes, one file) or the file must
  be split. The convention of one file per prefix is clean — splitting
  files when they grow too large maintains the bijection.

[DEC id:dec.navigation.routing-table date:2026-06-12 by:human+llm]
  The ID prefix→file mapping in this file is the authoritative routing table.
  Every context file's =CONTEXT_ID_PREFIX= should match its segment in
  =monadc.context.<segment>.*= IDs. When adding a new context file, add a row
  here first.
