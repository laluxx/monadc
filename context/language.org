[DEC id:dec.language.path-literals-are-values date:2026-06-12 by:human]
  Path literals are first-class =Path= values. Syntactically valid Unix path
  tokens such as =~/dir/=, =/tmp/file=, and =./file= parse as =AST_PATH=,
  infer as =TYPE_PATH=, lower to =i8*= in codegen, and print as their path
  text under =show=. Validation belongs at the reader boundary for malformed
  path syntax; valid path tokens must not be rejected merely because they are
  used as values.

[DEC id:dec.language.adjacent-strings-error date:2026-06-12 by:human]
  MonadC does not support implicit adjacent string literal concatenation. A
  connected spelling such as ="hello""world"= is a compile-time syntax error.
  Users should write one literal, for example ="helloworld"=, or use an
  explicit string operation when concatenation is intended.

[INF id:inf.language.adjacent-strings-rationale from:dec.language.adjacent-strings-error conf:high]
  Rejecting connected string literals keeps the surface language unambiguous:
  strings already support multi-line contents inside one pair of quotes, so
  C-style literal adjacency would mostly hide typos and produce confusing
  downstream type errors.

:PROPERTIES:
:ID: monadc.context.language.codegen-dispatch
:CUSTOM_ID: codegen-dispatch
:CONTEXT_KIND: component
:CONTEXT_DESCRIPTION: The three-layer codegen dispatch architecture: AST type switch, strcmp handler dispatch (83+ symbols), and register_builtins (94 entries). Only strcmp handlers are callable from paren-mode; register_builtins-only symbols are unreachable.
:CONTEXT_VERSION: 1
:CONTEXT_SCOPE: repository-local
:CONTEXT_OWNER: codex
:CONTEXT_MAINTAINER: codex
:CONTEXT_AUDIENCE: human,llm
:CONTEXT_CREATED: 2026-06-12
:CONTEXT_UPDATED: 2026-06-12
:CONTEXT_STABILITY: stable
:CONTEXT_STATUS: active
:SOURCE: codegen.c
:CONFIDENCE: high
:END:

[OBS id:obs.language.dispatch-architecture src:codegen.c conf:high]
  Codegen dispatch has three layers:
  1. =codegen_expr= switches on AST type (AST_LIST, AST_SYMBOL, AST_NUMBER, etc.)
  2. For =AST_LIST=, it strcmp's the head symbol against 83+ known symbols
     (codegen.c:3889-12492). These ARE callable from paren-mode.
  3. =register_builtins= (codegen.c:12960-13119) registers 94 entries in the
     env. These are NOT callable unless they also have a strcmp handler.

[DEC id:dec.language.dispatch-rule date:2026-06-12 by:codex]
  A function is callable from paren-mode ONLY if it has a strcmp handler in
  =codegen_expr=. =register_builtins=-only functions (=concat=, =substring=,
  =reverse=, =nth=, =length=, =reduce=, =fn=, =lambda=, =cond=, =when=, =let=,
  =let*=, =letrec=) are NOT directly callable.

[OBS id:obs.language.gap-closures src:codegen.c conf:high]
  Closures are broken: =fn=, =lambda=, =defn= have no strcmp dispatch handler.
  No closure allocation exists in codegen. This blocks higher-order functions:
  =map=, =filter=, =apply=, =partial=.

[OBS id:obs.language.gap-registered-only src:codegen.c conf:high]
  These functions are =register_builtins=-only — registered in env but lack
  strcmp dispatch and are unreachable from paren-mode: =concat=, =substring=,
  =reverse=, =nth=, =length=, =reduce=.

[OBS id:obs.language.gap-control-forms src:codegen.c conf:high]
  These control forms have no strcmp handler: =cond=, =when=, =let=, =let*=,
  =letrec=, =do=, =with=. =cond= is treated as an ordinary variable reference
  by the parser.

[OBS id:obs.language.gap-map-runtime src:runtime.c conf:high]
  Map and set runtime construction is broken for untyped contexts. Literals
  like ={:a 1}= and ={1 2 3}= produce garbage values without explicit =:: Map=
  / =:: Set= type annotation. Functional (=assoc=, =dissoc=, =merge=) always
  receive garbage inputs. Only =assoc! / find= with typed variables work.

[OBS id:obs.language.gap-list-variadic src:codegen.c:7345 conf:high]
  =(list 1 2 3)= with multiple arguments produces a list that returns =nil= for
  =(car (list 1 2 3))=. Only zero-arg =(list)= works. =(make-list n val)= works
  correctly. The bug is in runtime list construction from variadic args.

[OBS id:obs.language.gap-tail src:runtime.c conf:high]
  =tail= on lists segfaults. Runtime list implementation bug.

[OBS id:obs.language.gap-asm-naked src:asm.c conf:high]
  Inline assembly with =:naked True= compiles but does not propagate results —
  =add y %rax= returns the first parameter only, not the sum. The raw asm
  output has no LLVM constraint wiring.

[OBS id:obs.language.gap-asm-nonaked src:asm.c:346 conf:high status:superseded superseded-by:obs.language.asm-nonaked-i64-fixed]
  SUPERSEDED: Non-naked inline assembly used to fail because size suffix
  inference chose 32-bit =l= for =Int= parameters. The current implementation
  infers suffixes from LLVM operand widths and is covered by
  =tests/codegen/asm= and =rt_asm_function_nonaked.mon=.

[OBS id:obs.language.asm-nonaked-i64-fixed src:asm.c conf:high]
  Non-naked inline asm now infers bare integer instruction suffixes from LLVM
  placeholder operand widths. For =Int= parameters this emits =q=, and
  two-operand arithmetic swaps LLVM placeholders so =(asm add x y)= lowers to
  =addq $2, $1= under constraints =r,0,r=. Verified by
  =rt_asm_function_nonaked.mon= printing =30= for =(add 10 20)=.

[DEC id:dec.language.inline-asm-function-body-only date:2026-06-13 by:human]
  Meaningful inline assembly tests must use real function bodies, not top-level
  =(inline asm "...")= snippets. The supported user-facing shape is a typed
  function whose body is =(asm ...)=, for example:
  =define sys-write :: Int -> String -> Int -> Int= followed by
  =fd buf len -> (asm mov 1 %rax mov fd %rdi mov buf %rsi mov len %rdx syscall ret)=.

[OBS id:obs.language.inline-asm-parser src:reader.c:3889 conf:high]
  The reader recognizes =(asm mnemonic operand ...)= inside an S-expression and
  creates =AST_ASM=. Wisp-style definitions expand to that same form, so
  indentation syntax and parenthesized syntax share the backend.

[OBS id:obs.language.inline-asm-preprocess src:asm.c,asm.h conf:high]
  =asm.h= exposes =AsmInstruction=, =AsmContext=, =RegisterAllocator=, and
  =preprocess_asm=. =asm.c= splits a flat =AST_ASM= token stream into
  instructions by mnemonic recognition, maps parameter symbols to LLVM
  placeholders in non-naked functions, maps parameter symbols to SysV ABI
  registers in naked functions, escapes numeric immediates, and preserves string
  operands as raw text.

[OBS id:obs.language.inline-asm-codegen src:asm.c,codegen.c conf:high]
  Non-naked asm lowers through =LLVMGetInlineAsm= with return constraints
  =r= for integer-like values and =x= for floats. The first parameter is tied to
  the output with =,0=, later parameters use =,r=. Bare integer mnemonics infer
  size suffixes from LLVM integer widths or register names. Two-operand
  arithmetic/logical instructions with placeholder destinations are reordered
  for AT&T syntax.

[OBS id:obs.language.inline-asm-tests src:tests/codegen/asm conf:high]
  =tests/codegen/asm= contains 60 Wisp-style function-body asm fixtures. They
  cover syscall-shaped String operands, Int placeholder arithmetic, immediate
  escaping, multi-instruction splitting, unary inc/dec/neg/not, shifts and
  rotates with immediate counts, six-argument placeholder constraints, Char and
  Float return types, SSE scalar add/multiply, conditional moves, signed
  division, xchg, no-op placement, and syscall-shaped operands. The old
  generated top-level asm fixtures were replaced because they did not exercise
  the supported backend path.

[OBS id:obs.language.inline-asm-pointer-gap src:tests/codegen/asm/asm-syscall-write-string.mon conf:medium]
  Superseded by =obs.language.inline-asm-pointer-sugar-codegen=. The original
  gap was that the syscall example wanted =*U8= for buffers but codegen rejected
  pointer-sugar parameter types.

[OBS id:obs.language.inline-asm-pointer-sugar-codegen src:types.c,tests/codegen/asm/asm-sys-write-u8-pointer.mon conf:high]
  =type_from_name= accepts star-prefixed pointer sugar such as =*U8= and
  converts it to =Pointer :: U8=. Inline asm function definitions using the
  user-facing syscall shape now compile. String literals still type-check as
  =String=, so direct String-to-=*U8= call coercion remains a separate FFI bridge
  concern.

[OBS id:obs.language.gap-superscript src:tests/codegen conf:high]
  Superscript sugar (=x²=) desugars to =(^ x 2)=, but =^= has no strcmp
  dispatch handler despite being a registered builtin.

[OBS id:obs.language.gap-keyword-layout src:dep.c conf:high]
  Keyword-style layout construction (=(Point :x 3 :y 4)=) parses but the
  dependent type checker treats keyword tokens as positional arguments, so
  construction fails before codegen.

[OBS id:obs.language.gap-wisp-define src:wisp.c conf:high]
  Wisp-style multi-line =define= with =->= arrow syntax does not create correct
  lambda bindings. The workaround is to use s-expression form.

[OBS id:obs.language.gap-core-prelude src:core/ conf:high]
  Core prelude (=Coll.mon=, =Test.mon=) is NOT automatically imported yet. The
  desired language rule is automatic availability without explicit imports, but

  =core/prelude/Coll.mon=. Additionally, core modules have
  compilation failures: variadic =gcd= in =Math.mon=, =takeWhile= inference in

[OBS id:obs.language.array-three-forms src:types.c:449-499 conf:high]
  Arrays exist in three forms sharing one `TYPE_ARR`. `type_arr(elem, size)` is a
  compile-time-known stack array lowered to LLVM `[N x T]`. `type_arr_fat(elem)`
  is a runtime-sized fat pointer `{i8* data, i64 size}`. `type_arr_heap(elem)`
  is a heap-growable array with the same fat-pointer LLVM representation but
  `arr_is_heap=true` distinguishing it for mutation semantics. All three collapse
  to `arr.fat*` at the LLVM level when passed around.

[INF id:inf.language.array-ladder from:obs.language.array-three-forms,dec.philosophy.zero-to-metal conf:high]
  The three array forms are the zero-to-metal ladder made concrete. `[1 2 3]`
  starts as a stack array with compile-time-known size. Size sugar `[16kb I32]`
  annotates the size at the type level. `~[1 2 3]` signals heap allocation.
  From type-level set (a function `Int → elem` whose domain is known) to raw
  memory buffer, the ladder is: `type_arr → type_arr_fat → Pointer :: U8`.

[THINK id:think.language.array-why-three conf:high]
  Why three forms instead of one? Because the philosophy refuses the trade-off
  between mathematical safety and hardware control. A single uniform array
  representation would either lose the size information needed for bounds
  checking (safe but slow) or require the programmer to manage memory manually
  (fast but unsafe). Three forms let the programmer choose: `[1 2 3]` when the
  size is known and the compiler should check it; `~[1 2 3]` when the data
  outlives the stack frame; `[16kb I32]` when the layout must match a C struct.
  The cost is complexity — three constructors, three LLVM paths, three sets of
  rules about what can be done with each. The payoff is that no programmer is
  forced into a representation that fights their use case.

[OBS id:obs.language.array-type-to-llvm src:codegen.c:671-689 conf:high]
  `type_to_llvm` for `TYPE_ARR` branches on `type_arr_runtime_sized`. When true
  (fat, heap, or unknown-size), the LLVM type is a pointer to `arr.fat` struct
  `{i8*, i64}`. When false (known positive size), the LLVM type is `[N x T]`
  — a proper fixed-size array. But in practice, even stack arrays are immediately
  wrapped into `arr.fat*` via `arr_make_fat` at codegen time
  (codegen.c:3763-3789). The `[N x T]` path exists but is almost immediately
  heap-allocated.

[INF id:inf.language.array-fat-wrapping from:obs.language.array-type-to-llvm,obs.language.array-three-forms conf:high]
  The gap between the type representation and the codegen choice is significant.
  At the type level, `type_arr` preserves `arr_size`. At the codegen level,
  every array is heap-allocated into a fat pointer. The type says "I know this
  size" but the generated code treats it as runtime-sized. This means
  compile-time size information is used for type checking (bounds in indexing,
  unification) but NOT for stack allocation. The philosophical ideal of
  "zero-cost abstraction" collides with the practical reality that LLVM needs
  stack allocation decisions made at the function level, and arrays returned
  from functions must be heap-allocated.

[THINK id:think.language.array-as-function conf:high]
  Array indexing `(arr i)` is syntactic sugar for function application. In the
  semantic model, an array IS a function from `Int` to `elem`. The count
  operation is domain cardinality. The array predicate `Arr?` is a
  compile-time check of whether a type is `TYPE_ARR`. This is consistent with
  the set-theoretic model: an array is a finite function, represented
  extensionally as a block of memory. The three forms are three ways to specify
  the domain: known size (stack), unknown size (fat), mutable size (heap).
  Every other primitive follows the same pattern — there is always a
  type-level ideal and a runtime pragmatism, bridged by the codegen.

[OBS id:obs.language.set-no-param src:types.c:432,codegen.c:690-691 conf:high]
  `TYPE_SET` is a bare ground type with no element type parameter. `type_set()`
  returns a tag with no children. The LLVM type is `i8*` (opaque pointer).
  `Set?` is a compile-time type predicate checking `type->kind == TYPE_SET`.
  This means the compiler cannot distinguish `Set[Int]` from `Set[String]` at
  the codegen level — all sets are the same type.

[THINK id:think.language.set-why-untyped conf:high]
  Why does `TYPE_SET` carry no element type? The philosophical answer: because
  in the set-theoretic model, a set is just a collection of values, and values
  are all uniformly represented at runtime. The statement `(conj {1 2 3} "hi")`
  is type-correct because `conj` has scheme `∀a. Set → a → Set` — the element
  type variable `a` is instantiated but NOT stored in the result type. The
  runtime hash set stores `RuntimeValue*` pointers and uses `rt_equal_p` for
  equality. Type parameters would add a checked-boxing overhead that contradicts
  the "no overhead" principle without providing runtime safety that the hash
  table's dynamic equality doesn't already have.

[INF id:inf.language.set-grounded-in-model from:obs.language.set-no-param,dec.philosophy.sets-are-the-model,dec.philosophy.purity-is-practical conf:high]
  Sets are not merely a collection type — they are the runtime incarnation of
  the type system's semantic model. When the philosophy says "types are sets",
  it means that `TYPE_INT` and `TYPE_SET` are fundamentally the same kind of
  object: both are collections of values. `Int` is the infinite set of all
  integers; `{1 2 3}` is the finite set of three integers. The compiler should
  treat them as the same category of thing. This is why `Set?` is a compile-time
  type predicate — it's checking whether a type expression denotes a set at the
  type level. The same mechanism that checks `Int?` also checks `Set?`. The
  type system doesn't distinguish between "atomic types" and "collection types"
  at the fundamental level — both are just sets.

[OBS id:obs.language.set-two-mutability-modes src:runtime.c:1371-1383,codegen.c:2392-2415 conf:high]
  Set operations come in immutable and mutable pairs. `conj`/`disj` copy the
  set before modifying. `conj!`/`disj!` mutate in place. Both paths go through
  the same runtime codegen helper `codegen_set_op` which unboxes, calls the
  runtime function, and re-wraps. The distinction is purely a runtime choice
  — the type scheme for `conj` and `conj!` is identical: `∀a. Set → a → Set`.

[INF id:inf.language.set-functional-mutability-why from:obs.language.set-two-mutability-modes,dec.philosophy.purity-is-practical conf:high]
  The immutable/mutable distinction for sets is the "purity is practical"
  principle in microcosm. Functional `conj` returns a new set — mathematically
  pure, referentially transparent, safe to use in any context. Mutable `conj!`
  modifies in place — faster but requires the programmer to track aliasing.
  Both exist because the refusal to choose between mathematical purity and
  hardware control applies at every level, not just the grand architectural
  choices. The default is `conj` (pure). The escape hatch is `conj!` (mutable).
  The type system treats them identically — the distinction is a commitment
  about aliasing that the programmer makes, not a distinction the compiler
  enforces.

[OBS id:obs.language.set-as-function src:codegen.c:9880-9907,infer.c:1474-1477 conf:high]
  A variable of `TYPE_SET` can be used as a function: `(s 1)` returns the

  path for `ENV_VAR` with `TYPE_SET` — it skips arrow unification and calls
  `rt_set_get`. The inference side skips `TYPE_ARROW` unification when the
  function position has `TYPE_SET`.

[INF id:inf.language.set-as-function-why from:obs.language.set-as-function,dec.philosophy.sets-are-the-model conf:high]
  Set-as-function is the most direct line from philosophy to implementation. If
  types are sets and functions are relations, then a set IS a function from its
  elements to Bool. When you write `(s 1)`, you are applying the characteristic
  function of the set `s` to the argument `1`. This is not sugar — it is the
  semantic model surfacing through the syntax. The same mechanism makes arrays
  functions `(arr i)`, maps functions `(m key)`, and lists functions `(lst i)`.
  The collection-as-function coercion in `infer_unify_one` (infer.c:426-470) is
  the type system acknowledging that collections ARE functions in the
  semantic model.

[OBS id:obs.language.map-hash-brace-syntax src:reader.c:5914-5951,reader.c:1350-1353 conf:high]
  Maps use `#{k v ...}` syntax — the lexer recognizes `#{` as the two-character
  token `TOK_HASH_LBRACE` distinct from the set's `TOK_LBRACE`. The parser
  collects alternating key-value pairs into `AST_MAP.keys[]` and

[OBS id:obs.language.map-inference-no-params src:infer.c:1091-1102 conf:high]
  `infer_expr` for `AST_MAP` creates fresh type variables `key_t` and `val_t`,
  constrains all keys to `key_t` and all values to `val_t`, but returns bare
  `type_map()` with no type parameters. The inferred key/value types are
  unified at the call site but erased from the result. The HM type schemes
  for map operations (`assoc : Map -> k -> v -> Map`, `keys : Map -> List k`)
  reintroduce polymorphism at each call site through instantiation.

[THINK id:think.language.map-why-hash-brace conf:high]
  Why `#{:a 1}` instead of `{:a 1}`? The surface distinction exists because
  the set-theoretic model says maps ARE sets (of pairs), but the surface
  syntax must distinguish a map literal `{:a 1 :b 2}` from a set literal
  `{:a :b 1 2}` when both use the same brace delimiters. The `#{}` prefix is
  an explicit surface marker: "this brace block is a map, not a set." In the
  semantic model, `#{:a 1}` is `{(:a . 1)}` — a set of pairs. The surface
  syntax acknowledges that while maps and sets are the same category-theory
  object, programmers need a visual distinction for the common case.

[THINK id:think.language.map-bug-significance conf:high]
  The map garbage bug (untyped `{:a 1}` produces garbage at runtime) is the
  most philosophically significant bug in the compiler. It reveals the exact
  point where the type system's idealization (maps are uniform, everything is
  `TYPE_MAP`) collides with the runtime's requirements (boxed values need type
  information to unbox correctly). The type system says "maps are ground types,
  no parameters needed." The runtime says "I need to know the representation of
  the values I'm storing." Neither is wrong; the bridge between them was
  incomplete. The fix has two layers. Adding `case TYPE_MAP:` and `case TYPE_SET:`
  to `codegen_box` (codegen.c:1522) fixes the array-element-boxing path at
  line 1573 and the curry trampoline at line 10582. But the deeper fix is in
  `wrap_func_as_closure` (codegen.c:2554-2583) which has its own inline boxing
  logic that DOES NOT use `codegen_box` — its final `else` branch at line 2582
  passes through ALL raw pointer return types (Map, Set, List, String, Symbol,
  Keyword, Layout) as if they were already `RuntimeValue*`, when they are raw
  C pointers. This is the primary path for wrapping typed functions as closures
  (used by HOFs, storing functions in data structures, etc.), so the bug is
  wider than maps alone. The fix for `wrap_func_as_closure` requires routing
  its boxing through `codegen_box` or duplicating the type dispatch for each
  pointer-return type. This bug is a concrete demonstration that the bridge
  between abstraction and implementation has rungs missing — and finding them
  is the core engineering work of the project.

[OBS id:obs.language.map-runtime-representation src:runtime.c:864-1139 conf:high]
  The runtime map is an open-addressing hash table (`RuntimeMap`) with linear
  probing, tombstones, power-of-2 capacity starting at 8, and a load factor of
  0.7. Keys and values are stored as `RuntimeValue*` pointers. The hash
  function is FNV-1a for strings, Knuth multiplicative for integers,
  XOR-aggregate for collections. Structural equality uses `rt_equal_p`.

[INF id:inf.language.map-as-relation from:obs.language.map-ground-type,obs.language.map-as-function,dec.philosophy.sets-are-the-model conf:high]
  In the set-theoretic model, a map IS a set of pairs `(key . value)`. The
  `#{}` syntax constructs this set. `(m key)` applies the characteristic
  relation. `keys` and `vals` are projections onto the first and second
  components. `assoc` is `(conj m (pair key val))`. This is why maps share so
  much infrastructure with sets: the same codegen pattern (ground type → box →
  runtime hash table → unbox), the same type representation (`TYPE_MAP` is to
  `TYPE_SET` as `{k,v}` is to `{v}`), and the same collection-as-function
  coercion. The surface syntax distinguishes them for readability; the
  semantic model unifies them.

[OBS id:obs.language.function-dual-representation src:types.h:22-39,dep.h:122-123 conf:high]
  Functions have two type-level representations. `TYPE_ARROW` (types.h:39) is
  the HM inference arrow: a simple `param → return` chain. `TYPE_FN` (types.h:22)
  is the function descriptor with named parameters, optional args, rest args.
  `TERM_PI` (dep.h:122) is the dependent Π-type `Π(x:A).B` where the parameter
  appears in the return type. `TERM_EMBED` (dep.h:153) bridges between these:
  a `Type*` from the HM world lives as a `TERM_EMBED` term in the ITT world.

[OBS id:obs.language.function-closure-abi-dual src:codegen.c:4172-4189,codegen.c:4727-4789 conf:high]
  Closure codegen has two ABIs. The typed ABI passes all parameters as
  unboxed LLVM values — fast but requires the parameter count and types to be
  known at compile time. The closure ABI passes `(env*, i64 n, RuntimeValue** args)`
  — the environment pointer, argument count, and a boxed argument array. The
  choice is made by `use_closure_abi = is_inner || captured_count > 0`
  (codegen.c:4174). Simple top-level lambdas with no captures use the typed
  ABI. Nested lambdas that capture variables use the closure ABI.

[THINK id:think.language.function-why-dual conf:high]
  The dual function representation is THE architectural axis of the compiler,
  not a detail. Three dualities stack: HM arrow vs ITT Pi (expressiveness),
  TYPE_FN descriptor vs TYPE_ARROW chain (parameter tracking vs inference),
  typed ABI vs closure ABI (speed vs generality). Each duality serves the same
  purpose: let the common case be fast and simple while the complex case
  remains expressible. The bridges between them — `TERM_EMBED`,
  `wrap_func_as_closure`, `dep_ground_of_value_env` — are where the real
  engineering judgment lives. The design question "is this an HM function or
  an ITT function?" is not a question about the programmer's intent — it is
  a question about how much the compiler needs to know about the function to
  compile it efficiently. HM functions can be monomorphized and inlined. ITT
  functions need NbE, universe checking, and definitional equality. The
  programmer writes the same syntax either way; the compiler chooses the path.

[INF id:inf.language.function-abstraction-ladder from:obs.language.function-syntax-unification,dec.philosophy.zero-to-metal conf:high]
  The five ways to write a function are the abstraction ladder for callables.
  `λx.x` is pure lambda calculus — minimal syntax, maximal abstraction.
  `(lambda (x) x)` is Lisp. `(define (f x) x)` is named Lisp. `x -> x` in Wisp

  C-adjacent signature syntax. All compile to the same LLVM IR. The ladder is
  continuous: a programmer can start with `λx.body` in a quick experiment and
  graduate to typed signatures without changing the semantics.

[THINK id:think.language.function-dispatch-gap-significance conf:high]

  necessary. These are handled at the parser level — `(lambda ...)` is parsed

  standalone symbol `lambda` cannot be used as a higher-order function value
  (it is not callable from paren-mode). This is a bug in the completeness of
  the dispatch table, not a design choice about what functions are. The
  strcmp dispatch table registers 83+ symbols that CAN appear as call heads;
  `lambda` is missing only because the parser intercepts it. The distinction
  between "parser-level syntax" and "codegen-level callable" is an accidental
  complexity, not a principled one.

  The dependent type checker uses Normalization by Evaluation. `dep_eval`
  (dep.c:1180-1386) evaluates terms to a semantic Value domain. `dep_quote`
  (dep.c:1397-1501) reifies values back to terms. `dep_normalise` (dep.c:1503)

[OBS id:obs.language.dep-shadow-pass src:main.c:1025-1063,context/language.org:481-479 conf:high]
  The dependent type checker runs as Phase 6.5 — a "shadow pass" between
  features setup and LLVM codegen. It does not emit code. It checks types,
  elaborates surface syntax to core terms, and back-propagates ground types
  to `ast->inferred_type` via `dep_ground_of_value_env` (dep.c:2700-2782).
  The HM inferencer and LLVM codegen then use these annotated types.

[THINK id:think.language.dep-why-nbe conf:high]
  Why NbE instead of syntactic evaluation? Because NbE is the architecture
  that scales. A syntactic evaluator (reduce terms by rewriting) breaks down
  when you add η-equality, proof irrelevance, or cumulativity — each new
  feature requires new rewrite rules, and the rules interact in unpredictable
  ways. NbE with a semantic Value domain handles these uniformly: η-equality
  becomes η-expansion in `dep_quote`, cumulativity becomes subsumption in
  `dep_conv`, proof irrelevance becomes value-level erasure. The semantic
  domain is the canonical form: two terms are equal when they evaluate to the
  same value. The NbE kernel in MonadC (dep.h:304-393) is not just good
  enough — it is the right foundation for a language that aspires to full
  dependent types. It will scale to inductive families, higher inductive
  types, and cubical type theory without architectural rewrites.

[THINK id:think.language.dep-why-shadow-pass conf:high]
  Why a shadow pass instead of integrating dependent types into the main
  inference pipeline? The pragmatic answer: HM inference is O(n) in the size
  of the program, monomorphization is O(n * k) where k is the number of
  instantiation points, and NbE is O(fuel) with a hard cap of 100k. If every
  function went through the full ITT checker, the compiler would be unusably
  slow for ground-type code. The shadow pass design means: HM infers everything
  fast, the ITT pass validates dependent annotations and populates ground types
  on AST nodes, and the codegen never needs to know whether a type came from
  HM or ITT. This is the "purity is practical" principle applied to the
  compiler's own architecture — the fast path is the default, the sophisticated
  path is opt-in per expression.

[THINK id:think.language.dep-holes-significance conf:high]
  The soundness holes are not negligence — they are the project's thesis being
  tested. The claim is that an NbE kernel with a clean architecture can be
  incrementally filled in without rewrites. The stub pattern unification, the
  missing cumulativity, the Int/Char coercions — these are all places where
  the implementation says "this is where the next layer of correctness goes."
  The compiler produces working code today because the HM pipeline handles the
  common case and the ITT checker validates what it can. A program that uses
  dependent types badly (Int where Char is expected) gets caught by the HM
  type checker before the ITT pass even runs. The holes in `dep_conv` are
  only reachable if HM already passed — meaning they are refinements, not
  safety nets.

[THINK id:think.language.ground-intro conf:high]
  Ground types are the atoms of the type system. In the set-theoretic model,
  `Int` is the infinite set of all integers, `Float` is the set of all IEEE 754
  doubles, `String` is the set of all finite sequences of characters, `Bool` is
  the two-element set `{#t, #f}`, `Char` is the set of all Unicode code points,
  `Symbol` is the set of interned identifiers, `Keyword` is the set of
  keyword-flagged symbols, `Ratio` is the set of rational pairs, `Nil` is the
  singleton set containing only `nil`, and `Path` is the set of valid
  filesystem paths. These are not primitive in the sense of "simple" — an
  integer is an infinite set, which is mathematically complex — but in the
  sense of "given." The type system does not decompose them further. They are
  the Ur-elements, the ZFC atoms, the base cases of the recursion. Every
  program ultimately reduces to operations on ground types: arithmetic,
  string concatenation, symbol comparison, path resolution.

[OBS id:obs.language.ground-runtime-uniform src:runtime.h,codegen.c conf:high]
  All ground types are uniform at the LLVM level as `i8*` (opaque pointer to
  `RuntimeValue`). The runtime tag in `RTValueHeader` (runtime.h:29-31)
  distinguishes which ground type a value is. Codegen uses `rt_make_int`,
  `rt_make_float`, `rt_make_string`, `rt_make_bool`, `rt_make_char`,
  `rt_make_symbol`, `rt_make_keyword`, `rt_make_ratio`, and `rt_make_nil` to
  construct values. The type predicates (`Int?`, `Float?`, `String?`, `Bool?`,
  `Char?`, `Symbol?`, `Keyword?`, `Ratio?`, `nil?`, `Path?`) are compile-time
  checks of `type->kind` against the `TypeKind` enum after inference annotates
  AST nodes.

[INF id:inf.language.ground-grounding from:obs.language.ground-runtime-uniform,think.language.ground-intro,dec.philosophy.sets-are-the-model conf:high]
  Ground types are the crucial link between the set-theoretic philosophy and
  the generated machine code. `Int` is a set in the model and an `i8*` pointer
  in the runtime; the bridge between them is the `RuntimeValue` tag. The type
  system treats ground types as atoms (unification checks kind equality, not
  structure). The dep checker treats them as `TERM_EMBED` of a `Type*`. The
  codegen treats them as opaque pointers managed by the runtime. The
  consistency of this bridge — that `type_int()` always produces the same
  `Type*`, that `rt_make_int` always produces the same `RuntimeValue` tag,
  that the codegen path for `+` dispatches on the same kind tag — is what
  makes the compiler correct.

[OBS id:obs.language.list-carries-list-types src:types.h:86-88 conf:high]
  `TYPE_LIST` is unusual among MonadC types: it carries `list_types[]`
  (an array of `Type*` pointers, count in `list_count`), not a single
  `element_type`. This means a single list type can encode tuples:
  `(Int Float String)` is `TYPE_LIST` with three element types. The
  `list_elem` macro (types.h:118) aliases to `element_type` for backward
  compatibility with single-element lists. When `list_count == 0`, the
  list is heterogeneous (any type of element). When `list_count == 1`,
  it is a homogeneous list. When `list_count > 1`, it is a tuple.

[OBS id:obs.language.list-runtime-cons-cells src:runtime.c:1534-1667 conf:high]
  The runtime representation is linked cons cells (`RuntimeCons`):
  each cell has a `car` (RuntimeValue*) and `cdr` (RuntimeValue*,
  either another cons cell or `nil`). `nil` (TYPE_NIL) terminates the
  list. Operations: `cons`, `car`, `cdr`, `++` (append), `count`,
  `empty?`, `pair?`, `list`, `make-list`. Lists are immutable by default;
  mutable operations use `set-car!`/`set-cdr!` at the runtime level.

[OBS id:obs.language.list-as-function src:codegen.c:9880-9907,infer.c:1474-1477 conf:high]
  A list can be used as a function: `(lst i)` indexes into the list at
  position `i`, returning the `i`-th element or `nil` if out of bounds.
  This is the same collection-as-function coercion as sets, maps, and
  arrays — in the set-theoretic model, a list IS a function `Int → elem`
  whose domain is `[0, length)`.

[THINK id:think.language.list-lisp-soul conf:high]
  Lists are the Lisp soul of MonadC. The cons cell is the ur-data-structure
  from which McCarthy built the first Lisp. In MonadC, lists serve the role
  that arrays serve in other languages: the flexible, general-purpose
  sequential container. But unlike arrays, lists are lazy-capable (the
  runtime can defer cons cell construction), heterogeneous (a single list
  can mix types), and structurally recursive (a list is either nil or a
  cons of head and tail). The `list_types[]` array is the type-level shadow
  of the cons-cell structure: each element's type is tracked at compile time
  but all values are uniform `RuntimeValue*` at runtime.

[THINK id:think.language.list-computational-vs-materialized conf:high]
  Lists are the computational/streaming type; arrays are the materialized
  type. A list `(1 2 3)` is a chain of three cons cells, each allocated
  independently, taking O(n) to index. An array `[1 2 3]` is a contiguous
  block of memory, taking O(1) to index. The distinction is the same as
  linked list vs vector, or stream vs buffer. In the set-theoretic model,
  both are functions `Int → elem` — a list is a lazy function (compute
  the nth cell by walking), an array is an eager function (compute the
  nth cell by pointer arithmetic). The type system unifies them under
  collection-as-function; the programmer chooses the performance profile.

[THINK id:think.language.fixed-width-why conf:high]
  Fixed-width types exist because the type hierarchy from set-theoretic
  idealization to hardware concrete is a ladder, not a binary choice.
  `Int` is the infinite set of all integers — mathematically pure,
  compiler-independent, safe for any value range. `I32` is exactly 32
  bits, two's complement, with defined wrap-around — the hardware
  representation. `INT_ARBITRARY` (`I<n>`/`U<n>`) fills the gap for
  bit widths not in the standard set. The ladder from `Int` to `I32`
  to `I7` (7-bit signed) to `U1` (1-bit unsigned) is continuous: every
  step trades mathematical generality for hardware control. This is
  the zero-to-metal principle at the type level for numbers.

[OBS id:obs.language.fixed-width-representations src:types.h:26-36,types.h:43-44,types.h:109-111 conf:high]
  Each fixed-width type maps directly to a C primitive: `TYPE_I8` →
  `int8_t`, `TYPE_U8` → `uint8_t`, `TYPE_I16` → `int16_t`, `TYPE_U16` →
  `uint16_t`, `TYPE_I32` → `int32_t`, `TYPE_U32` → `uint32_t`, `TYPE_I64` →
  `int64_t`, `TYPE_U64` → `uint64_t`, `TYPE_I128` → either `__int128` or
  software-emulated, `TYPE_U128` → corresponding unsigned, `TYPE_F32` →
  `float` (IEEE 754 single), `TYPE_F80` → `long double` (x87 extended
  precision). `TYPE_INT_ARBITRARY` carries `numeric_width` (1–64 or 128)
  and `numeric_signed` (true for I, false for U) to represent any
  arbitrary-width integer.

  which collection type is intended, it unifies against `TYPE_COLL`;
  the specific type is resolved during monomorphization or by explicit

[OBS id:obs.language.tc-optional src:types.h:45,types.h:90 conf:high]

[THINK id:think.language.composition-patterns conf:high]
  Type composition in MonadC follows a uniform pattern: every composed
  type is a type constructor applied to type arguments. The surface
  syntax is `(Constructor Arg Arg ...)`. Each form has a specific
  `TypeKind` representation in types.h and a specific runtime behavior:
  - `(Int)` → `TYPE_LIST` with `list_count=1, list_types=[Int]` — homogeneous list
  - `(a)` → `TYPE_LIST` with `list_count=1, list_types=[Var(id)]` — polymorphic list
  - `(Int Float String)` → `TYPE_LIST` with `list_count=3` — 3-tuple
  - `()` → `TYPE_LIST` with `list_count=0` — heterogeneous list (any type)
  - `(Arr Int)` → `TYPE_ARR` with element Int — array of Ints
  - `(-> Int Int)` → `TYPE_ARROW` with param=Int, ret=Int — function type
  - `(Ptr Int)` → `TYPE_PTR` with element Int — pointer to Int
  - `(Optional Str)` → `TYPE_OPTIONAL` with element Str — nullable String
  - `(Coll (List Int))` → `TYPE_COLL` with element=List[Int] — collection constraint
  - `(Vec Bool 3)` → `TYPE_APP` with constructor="Vec", arg implies 3

[OBS id:obs.language.composition-list-as-tuple src:types.h:86-88 conf:high]
  The most general composition is `TYPE_LIST` with multiple `list_types[]`.
  `(Int Float String)` is a 3-tuple stored as a list of three cons cells.
  The compiler tracks each element's type separately at inference time but
  all values are uniform `RuntimeValue*` at runtime. This means tuple
  access `(my-tuple 0)` returns the first element with type `Int`,
  `(my-tuple 1)` returns `Float`, etc. — the type system knows the type
  of each position.

[OBS id:obs.language.composition-list-homogeneous src:types.h:86-88 conf:high]
  `(Int)` with a single type argument is a homogeneous list. The compiler
  knows every element is `Int`. The `list_count=1` path triggers the
  homogeneous list optimization path in inference: all elements unify
  against the single element type.

[OBS id:obs.language.composition-list-heterogeneous src:types.h:86-88 conf:high]
  `()` with zero type arguments is a heterogeneous list. The compiler
  knows nothing about element types at compile time. All element types
  are inferred from usage. This is the most flexible form and the least
  optimizable.

[OBS id:obs.language.composition-arr src:types.h:93-97 conf:high]
  `(Arr Int)` composes the `Arr` type constructor with element type `Int`.
  The result is `TYPE_ARR` with `arr_element_type = Int`. Size defaults
  to unknown (`arr_size = -1`) unless specified. The same pattern works
  for any ground or composed element type: `(Arr (Arr Int))` = 2D array.

[OBS id:obs.language.composition-arrow src:types.h:77-79 conf:high]
  `(-> Int Int)` composes the arrow type constructor with param `Int`
  and return `Int`. This is the HM function type `Int → Int`.
  Multi-argument functions chain: `(-> Int (-> Int Int))` = curried form.

[OBS id:obs.language.composition-ptr-optional src:types.h:90,types.h:42,types.h:45 conf:high]
  `(Ptr Int)` and `(Optional Str)` use the single-element `element_type`
  pattern. `TYPE_PTR` and `TYPE_OPTIONAL` share the same union field
  (types.h:90). The pattern is: a unary type constructor wrapping one
  inner type.

[THINK id:think.language.composition-unifying-pattern conf:high]
  The unifying pattern is: every type composition is a constructor applied
  to arguments. `TYPE_LIST` is the n-ary constructor (0 or more args).
  `TYPE_ARR`, `TYPE_PTR`, `TYPE_OPTIONAL`, `TYPE_COLL` are the unary
  constructors (1 arg each). `TYPE_ARROW` is the binary constructor
  (param + ret). `TYPE_APP` is the general n-ary constructor for future
  user-defined types (`Vec`, `Maybe`, `Either`, etc.). The type system
  treats all of these uniformly: they are Type nodes with `kind` and a
  variable number of child Type pointers. The inference engine traverses
  them structurally. The codegen dispatches on their kind. The
  set-theoretic model interprets them as sets: `(Int)` is the set of
  all lists of integers, `(-> Int Int)` is the set of all integer
  functions, `(Arr Int)` is the set of all integer arrays. Every
  composed type is a set of structured values, just like every ground
  type is a set of atomic values.

[THINK id:think.language.macros-why-set-theoretic conf:high]
  Macros in MonadC are not mere token rewriting — they are type-level
  computations in the set-theoretic model. A macro is a function from AST
  to AST that runs in Phase 4 (after import resolution, before HM inference).
  The key insight: if types are sets and functions are relations, then a
  macro is a relation between syntax trees. `define-macro` creates a new
  mapping in this relation; `is_syntax_return` detects whether an expression
  is a macro invocation by checking its residual type. This is the practical
  expression of the extensible-trait-system: macros are the mechanism by
  which the programmer extends the language's syntax within the set-theoretic
  framework, without requiring compiler modifications.

[OBS id:obs.language.macros-phase-position src:main.c,dep.c conf:high]
  Macros execute in Phase 4 of the compiler pipeline, between import
  resolution (Phase 3) and HM inference (Phase 5). The dependent type
  checker (Phase 6.5) can validate macro types retrospectively. Macro
  expansion is deterministic: given the same AST and the same macro
  definition, the output AST is always the same. The `is_syntax_return`
  predicate checks whether a function's return type synthesizes to a
  syntax type, indicating it is a compile-time macro rather than a
  runtime function.

[OBS id:obs.language.macros-extensibility src:macro.c conf:high]
  The macro system is designed as the extensible layer of what would
  otherwise be compiler built-ins. Pattern matching syntax, type class
  desugaring, data declarations, and instance deriving are all expressible
  as macros. The `define-macro` form registers a new syntax transform in
  the global macro table. Macros can inspect the AST, invoke the type
  checker, and generate new AST nodes. This is the "purity is practical"
  principle applied to language extension: the compiler defines the core
  forms, and macros define everything else in the same language.

[THINK id:think.language.runtime-value-system-why conf:high]
  The RuntimeValue is the bridge between the type system and the runtime. In
  the set-theoretic model, every value is an element of some set. In the
  runtime, every value is an `i8*` pointer to a `RuntimeValue` struct whose
  `type` tag (RuntimeValueType enum) determines which set it belongs to. This
  is the practical expression of erasure: the type system tracks fine-grained
  static types, but at runtime every value is a pointer. The tag tells the
  runtime how to interpret the data union. This design is uniform (every value
  is a pointer), extensible (new tags can be added), and compatible with C/LLVM
  (the pointer is opaque at the ABI boundary).

[OBS id:obs.language.runtime-value-tags src:runtime.h:18-34 conf:high]
  The `RuntimeValueType` enum has 14 tags: `RT_INT`, `RT_FLOAT`, `RT_CHAR`,
  `RT_STRING`, `RT_SYMBOL`, `RT_KEYWORD`, `RT_LIST`, `RT_RATIO`, `RT_ARRAY`,
  `RT_NIL`, `RT_THUNK`, `RT_BIGNUM`, `RT_SET`, `RT_MAP`, `RT_CLOSURE`. Each
  tag selects the active member of the data union in `RuntimeValue`
  (runtime.h:66-92). The LLVM type is always `i8*` — the tag is stored at
  offset 0 as a C enum (4 bytes). The codegen never needs to inspect the
  tag directly; it calls `rt_make_*` and `rt_unbox_*` functions which handle
  tag dispatch internally.

[OBS id:obs.language.runtime-value-construction src:runtime.h:258-276 conf:high]
  Values are constructed via `rt_value_*` functions: `rt_value_int(i64)`,
  `rt_value_float(double)`, `rt_value_char(char)`, `rt_value_string(const
  char*)`, `rt_value_symbol(const char*)`, `rt_value_keyword(const char*)`,
  `rt_value_list(RuntimeList*)`, `rt_value_nil()`, `rt_value_thunk(RuntimeThunk*)`,
  `rt_value_ratio(i64, i64)`, `rt_value_array(size_t)`, `rt_value_set(RuntimeSet*)`,
  `rt_value_map(RuntimeMap*)`. Hot-path types (int, float, char, list, nil,
  thunk) use arena allocation (g_eval_arena) for zero-free performance.
  Long-lived types (string, symbol, keyword, ratio, array, set, map) use
  heap allocation via `malloc` and are freed by `rt_value_free`.

[OBS id:obs.language.runtime-value-unboxing src:runtime.h:278-288 conf:high]
  Values are unboxed via `rt_unbox_*` functions: `rt_unbox_int(RuntimeValue*)`
  returns `int64_t`, `rt_unbox_float` returns `double`, `rt_unbox_char` returns
  `char`, `rt_unbox_string` returns `char*`, `rt_unbox_list` returns
  `RuntimeList*`, `rt_unbox_set` returns `RuntimeSet*`, `rt_unbox_map` returns
  `RuntimeMap*`, `rt_value_is_nil` returns `int`. These functions assert the
  correct type tag at runtime — calling `rt_unbox_int` on a `RT_FLOAT` value
  is a fatal error. The codegen inserts these calls automatically based on the
  static type information from the HM inference pass.

[OBS id:obs.language.runtime-thunk-system src:runtime.h:49-54,runtime.h:169-173 conf:high]
  Thunks implement lazy evaluation. `RuntimeThunk` (runtime.h:49-54) carries
  a function pointer `fn`, an environment `env`, a cached `value` pointer, and
  a `forced` flag. `rt_thunk_create(fn, env)` creates an unevaluated thunk.
  `rt_force(thunk)` evaluates it and caches the result. Once forced, subsequent
  calls return the cached value. Thunks are the runtime mechanism for lazy cons
  (stream programming), lazy ranges (infinite lists), and delayed evaluation
  in the dependent type checker's NbE kernel.

[OBS id:obs.language.runtime-list-representation src:runtime.h:111-126,128-135 conf:high]
  Lists are implemented as singly-linked cons cells. `ConsCell` (runtime.h:116-126)
  has inlined thunks for head and tail (both `fn`+`env`+`val`+`forced` triples)
  enabling fully lazy cons cells. A `NULL tail_val` with `tail_forced=1`
  signals end-of-list. `RuntimeList` (runtime.h:133-135) is a thin wrapper
  around `ConsCell*` — `NULL` cell means empty list. The lazy cons design
  means `(range 1 1000000)` constructs a thunk, not a million cells. Only
  accessed elements are forced.

[OBS id:obs.language.runtime-set-representation src:runtime.h:144-149 conf:high]
  RuntimeSet (runtime.h:144-149) is an open-addressing hash table with linear
  probing and tombstones. Capacity is always a power of two. Load factor
  threshold: 0.7. Operations: `rt_set_conj`, `rt_set_disj` (immutable copies),
  `rt_set_conj_mut`, `rt_set_disj_mut` (in-place mutation), `rt_set_contains`,
  `rt_set_get`. The hash function is FNV-1a for strings, Knuth multiplicative
  for integers, XOR-aggregate for collections.

[OBS id:obs.language.runtime-map-representation src:runtime.h:152-166 conf:high]
  RuntimeMap (runtime.h:156-166) mirrors RuntimeSet but stores key-value pairs
  in `RuntimeMapEntry` buckets. Open addressing, linear probing, tombstones,
  power-of-2 capacity starting at 8, load factor 0.7. Operations: `rt_map_assoc`,
  `rt_map_dissoc` (immutable), `rt_map_assoc_mut`, `rt_map_dissoc_mut` (mutable),
  `rt_map_get`, `rt_map_contains`, `rt_map_keys`, `rt_map_vals`, `rt_map_merge`.

[OBS id:obs.language.runtime-closure-representation src:runtime.h:58-64 conf:high]
  RuntimeClosure (runtime.h:58-64) carries an LLVM-compiled function pointer
  `fn_ptr`, a heap-allocated environment `env` (array of `RuntimeValue*`), the
  environment size `env_size`, the declared arity `arity`, and an optional debug
  `name`. Closures are called via `rt_closure_calln` which packs the environment
  and arguments into a flat array and invokes the compiled function through the
  closure ABI `(env*, i64 n, RuntimeValue** args)`.

[OBS id:obs.language.runtime-hashing-equality src:runtime.c conf:high]
  Structural equality is handled by `rt_equal_p` (runtime.h:255) which
  dispatches on the type tag. Bignum equality uses `mpz_cmp`. Collection
  equality (list, set, map) traverses structure recursively with cycle
  detection. Hashing (`rt_hash_value`) uses FNV-1a for strings, Knuth
  multiplicative for integers, and XOR-aggregate for collections. Hash
  consistency with equality is guaranteed: `rt_equal_p(a, b)` implies
  `rt_hash_value(a) == rt_hash_value(b)`.

[OBS id:obs.language.runtime-bignum src:runtime.h:310-317 conf:high]
  Bignums provide arbitrary-precision integer arithmetic via GMP (gmp.h).
  `rt_value_bignum_from_str` parses a decimal string. `rt_promote_to_bignum`
  upgrades an `int64_t` value to bignum on overflow. Operations: add, sub,
  mul, div. The bignum tag (`RT_BIGNUM`) is distinct from `RT_INT`, allowing
  the runtime to distinguish machine-word integers from arbitrary-precision ones.

[THINK id:think.language.pipeline-why-11-phases conf:high]
  The 11-phase pipeline is an emergent property of the compiler's architecture,
  not a designed-in plan. Each phase exists because the previous phase's output
  is needed by the next: module decls must be parsed before imports can be
  resolved, LLVM must be initialized before builtins can be registered, the
  *features* global must exist before codegen can emit expressions. The shadow
  pass (Phase 6.5) is the only deliberately inserted phase — it runs between
  HM inference and LLVM codegen because the dependent type checker must see
  the fully elaborated HM-typed AST before LLVM lowers it to a linear sequence
  of instructions. The pipeline could be refactored into fewer phases, but the
  current structure faithfully reflects the actual data dependencies.

[OBS id:obs.language.pipeline-phase1 src:main.c:855-904 conf:high]
  Phase 1 — Module + Import Declarations. Parses `module` and `import` forms
  from the top of the source file. Builds a `ModuleContext` with the module
  name, export policy, and import list. If no explicit `module` form is found,
  infers the module name from the file path. This phase must complete before
  any codegen work begins because the module name affects symbol mangling.

[OBS id:obs.language.pipeline-phase2 src:main.c:906-923 conf:high]
  Phase 2 — Verify Dependencies are Compiled. Checks the module registry for
  each imported module. If a dependency isn't found, attempts a fallback
  path-to-name resolution and recursively compiles it via `compile_one`. After
  compilation, resets the parser context to the current source file.

[OBS id:obs.language.pipeline-phase3 src:main.c:925-947 conf:high]
  Phase 3 — LLVM Setup. Calls `LLVMInitializeNativeTarget`,
  `LLVMInitializeNativeAsmPrinter`, `LLVMInitializeNativeAsmParser`.
  Initializes the `CodegenContext`, creates the LLVM module, builds the
  `DepCtx` for dependent types, registers builtins via `register_builtins`,
  registers Wisp arities, and declares runtime functions via
  `declare_runtime_functions`.

[OBS id:obs.language.pipeline-phase4 src:main.c:949 conf:high]
  Phase 4 — Declare Externals from Compiled Dependencies. Iterates through
  imported modules and declares their exported functions, globals, and types
  in the current LLVM module so codegen can reference them.

[OBS id:obs.language.pipeline-phase5 src:main.c:964 conf:high]
  Phase 5 — Create Init/Main Function and Position Builder. Creates the
  entry function (init or main depending on mode), positions the LLVM IR
  builder, and sets up the per-function allocation arena.

[OBS id:obs.language.pipeline-phase6 src:main.c:987 conf:high]
  Phase 6 — *features* Global. Emits the `*features*` global constant
  (a list of keyword symbols advertising compiler capabilities). This must
  exist before any codegen that reads features for conditional compilation.

[OBS id:obs.language.pipeline-phase6-5 src:main.c:1025-1063 conf:high]
  Phase 6.5 — Dependent Type Checking (Shadow Pass). Runs the bidirectional
  ITT type checker with NbE definitional equality. This pass does not emit
  code — it checks dependent annotations, elaborates surface syntax to core
  terms, and back-propagates ground types to `ast->inferred_type` via
  `dep_ground_of_value_env`. The HM inferencer and LLVM codegen then use
  these annotated types. This is the only deliberately inserted "shadow"
  phase (see think.language.dep-why-shadow-pass).

[OBS id:obs.language.pipeline-phase7 src:main.c:1065 conf:high]
  Phase 7 — Codegen Top-Level Expressions. The main code generation pass.
  Iterates through all top-level expressions, calls `codegen_expr` for each,
  and emits LLVM IR. This is where the 83+ strcmp dispatch handlers are
  exercised, closures are compiled, collections are constructed, and all
  runtime values are allocated.

[OBS id:obs.language.pipeline-phase8 src:main.c:1110 conf:high]
  Phase 8 — Terminate Function. Emits the return instruction for the entry
  function (returning the value of the last expression, or nil for
  module-initialization-only compilation). Runs LLVM verification on the
  generated function.

[OBS id:obs.language.pipeline-phase9 src:main.c:1122 conf:high]
  Phase 9 — Build Registry Entry + Rename LLVM Symbols. Builds a
  `CompiledModule` registry entry with the module name, LLVM module pointer,
  and exported globals. Renames LLVM symbols from their internal (reader-assigned)
  names to mangled names based on the module and export policy. This phase must

[THINK id:think.language.primitives-unified-theme conf:high]
  Ten primitives, one pattern. The set-theoretic model unifies ground types
  (atoms), collections (list, array, set, map), functions (arrow, fn, pi),
  type-level constructors (var, app, ptr, optional), macros (syntax transform),
  and dependent types (nbe kernel). Every primitive has an ideal representation
  (set, relation, function, dependent product) and a pragmatic runtime
  representation (cons cell, hash table, fat pointer, closure env, embedded
  ground term). The bridge between them is always a translation layer —
  `arr_make_fat` for arrays, `rt_value_set` for sets, `wrap_func_as_closure`
  for functions, `term_embed` for dependent types. The bridges are where the
  compiler's engineering judgment resides. They are always slightly incomplete,
  and completing them is the ongoing work of the project.

[THINK id:think.language.primitives-set-theoretic-core conf:high]
  The set-theoretic model is the unifying principle. Arrays are functions
  `Int → elem`. Sets are their own characteristic functions. Maps are sets of
  pairs which are relations which are functions. Lambda is the introduction
  form for the Pi type which is the dependent function space. Even the runtime
  representations follow the set-theoretic model: hash sets store values, hash
  maps store pairs, fat arrays store a function table `{data, size}`. The
  category theory is not decorative — it is the explicit semantic model that
  the compiler is built to realize. Every primitive, every operation, every
  codegen path traces back to the claim that types are sets and functions are
  relations.

[OBS id:obs.language.construct.data src:reader.h:59-61,reader.h:251-260 conf:high]
  `AST_DATA` represents algebraic data type declarations `(data Color Red | Green | Blue)`.
  The AST payload carries the type name, type parameters (e.g. `["a"]` for `Maybe a`),

  access via the layout system. The `typeclass.c` module generates deriving

[OBS id:obs.language.construct.class src:reader.h:60,reader.h:262-276 conf:high]

  The AST payload carries the class name, type variable, associated type names,
  method signatures (name + type string pairs), and optional default method
  implementations. The reader stores method bodies as `AST` subtrees
  (reader.h:273-274). The type class system (`typeclass.c`) resolves method
  dispatch through instance declarations at compile time.

[OBS id:obs.language.construct.instance src:reader.h:61,reader.h:278-289 conf:high]
  `AST_INSTANCE` represents type class instance declarations

  name, type name, associated type value names, and method implementations as
  lambda AST subtrees (reader.h:286-288). Instance resolution is fully static:
  the compiler resolves method calls to concrete implementations at compile
  time with no runtime dispatch overhead. Derived instances (from `data`
  deriving clauses) are generated automatically by `typeclass.c`.

[OBS id:obs.language.construct.tests src:reader.h:52,reader.h:208-211 conf:high]

  payload carries an array of assertion AST nodes. Tests are parsed but not
  compiled to the main program — they are extracted by the test runner in a
  separate pass. Each assertion within a tests block is an `(assert-eq expected actual)`
  form with its own codegen path (codegen.c:6431-6602). Test blocks use a
  distinct `AST_TESTS` tag so the compiler can skip them during normal
  compilation and extract them for test execution.

[OBS id:obs.language.construct.range src:reader.h:54,reader.h:213-219 conf:high]

[OBS id:obs.language.construct.address-of src:reader.h:53,reader.h:330 conf:high]

[OBS id:obs.language.construct.define src:codegen.c:4106-4640,codegen.c:4106-4107 conf:high]
  `(define name value)` binds a value to a name in the current scope. It accepts
  two forms: `(define name value)` for simple variable binding and
  `(define (name params ...) body)` for function definition shorthand which
  desugars to `(define name (lambda (params ...) body))`. Typed definitions
  use bracket syntax: `(define [name :: Type] value)`. When value is `AST_LAMBDA`,
  codegen performs HM inference to determine the function type scheme
  (codegen.c:4143-4144), collects free variables for closure construction, and
  selects between typed ABI and closure ABI based on capture analysis
  (codegen.c:4174). Wisp-style multi-line define with `->` arrow syntax has a
  known gap in lambda binding creation (obs.language.gap-wisp-define).

[OBS id:obs.language.construct.let src:reader.c conf:high]

  The reader desugars `let` to lambda application: `(let [x val] body)` becomes
  `((lambda (x) body) val)` (obs.reader.sugar.let-desugar). Multiple bindings
  produce nested applications. `let*` desugars to sequential `let` forms (each
  binding is in scope for subsequent bindings). `letrec` uses the Y combinator
  for recursive bindings. These forms have no strcmp dispatch handler in codegen
  (obs.language.gap-control-forms) — they are fully desugared at the reader
  level before codegen sees them.

[OBS id:obs.language.construct.set! src:codegen.c:2351,codegen.c:7471 conf:high]

  strcmp on `"set!"` (codegen.c:2351) and emits a store instruction to the
  variable's allocated storage. The compiler tracks which bindings are mutated
  to avoid unnecessary boxing. `set!` impurifies the containing function for the
  purposes of the purity tracking system. Assignment arrow sugar `x <- expr`
  desugars to `(set! x expr)` (obs.reader.sugar.assign-arrow).

  `(module Name (export ...) (import ...) ...)` declares a compilation module
  with a name, export list, and body. Module declarations are processed in

[OBS id:obs.language.construct.import src:module.c,reader.c conf:high]
  `(import ModuleName)` imports bindings from another module into the current
  scope. The import system resolves dependencies recursively: if an imported
  module has not been compiled, the compiler falls back to path-to-name
  resolution and compiles it recursively (obs.language.pipeline-phase2).
  Imports support renaming: `(import ModuleName :rename (old new))`. Core
  prelude imports (`Coll`, `Test`) are not yet automatically available
  (obs.language.gap-core-prelude).

[OBS id:obs.language.construct.export src:module_export.c,reader.c conf:high]
  `(export name1 name2 ...)` declares which bindings a module exposes to
  importers. The export list is collected during module declaration parsing
  and used by Phase 9 (registry entry + symbol renaming) to determine which
  LLVM symbols retain external linkage. Exports are the visibility boundary:
  unexported bindings are renamed to module-internal names and may be

[OBS id:obs.language.construct.with src:codegen.c:7841-7878 conf:high]
  `(with binding expr)` creates a scoped resource context. The binding is
  evaluated and bound for the scope of the body; cleanup occurs when the
  body exits. Currently has no strcmp dispatch handler — it is desugared at
  the reader level.

[OBS id:obs.language.construct.pipe src:reader.c,obs.reader.sugar.pipe conf:high]
  `(x |> f)` pipes the result of `x` as the argument to `f`. The pipe sugar
  is handled by the wisp transformer, not the reader desugarer: `x |> f`
  desugars to `(f x)`. Pipe chains `x |> f |> g` compose to `(g (f x))`.
  This is a reader-level syntactic transformation with no separate codegen path.

[OBS id:obs.language.construct.include src:codegen.c:5519-5592 conf:high]
  `(include "header.h")` or `(include <stdio.h>)` includes a C header for FFI
  access. The codegen path declares extern functions and types from the parsed
  header. The C FFI system (`ffi.c`) uses libclang to parse headers. Angle-bracket
  includes are still baseline coverage (obs.language.cffi-angle-gap). Include
  is the primary mechanism for C interop.

[OBS id:obs.language.construct.error-undefined src:codegen.c:6604-6774 conf:high]
  `(error "message")` halts execution with a runtime error. Codegen emits a
  call to `rt_error` which prints the message and calls `exit(1)`. `(undefined)`
  is a placeholder for unimplemented branches — codegen emits an LLVM
  `unreachable` instruction or a call to `rt_undefined` depending on context
  (codegen.c:6685-6774). Both are used as diagnostic and development aids.

[OBS id:obs.language.construct.assert-eq src:codegen.c:6431-6602 conf:high]
  `(assert-eq expected actual)` is the test assertion primitive. Codegen
  evaluates both arguments and calls `rt_equal_p` to compare them. If the
  comparison fails, it prints a diagnostic with `show` of both values and
  reports the failure location. This is the primary assertion form used in
  codegen test fixtures (`rt_codegen_*.mon`).

[OBS id:obs.language.construct.show src:codegen.c:5876-6364 conf:high]
  `(show value)` converts any value to its string representation. Codegen
  dispatches on the runtime type tag to call the appropriate print function.
  The show path supports all ground types, collections (list/set/map/array),
  ADTs (via the layout constructor printer), and closures. Layout-typed values
  are printed as canonical data syntax `(Ctor field0 field1 ...)`
  (obs.language.adt-show-codegen-canonical).

[THINK id:think.language.constructs-intro conf:high]
  Language constructs are the surface forms a programmer writes. Unlike types
  (which classify values) or primitives (which define semantic domains),
  constructs define the grammar of expression. Every construct has a reader
  representation (ASTType or AST_LIST head symbol), an inference path
  (type-checking rule), and a codegen path (LLVM IR lowering). The set of
  constructs is larger than the set of types — many surface forms (define, if,
  for, quote) are syntactic conventions desugared or compiled to operations on
  the type primitives documented above. This section documents each construct
  family with its AST shape, semantics, and compiler pipeline behavior.

[OBS id:obs.language.construct.number src:reader.h:40,codegen.c:3383-3384 conf:high]
  `AST_NUMBER` represents numeric literals. The reader parses integer and float
  tokens into a `double number` payload (reader.h:146) and preserves the original
  literal string in `ast->literal_str` for hex/bin/oct radix. The raw uint64
  value for literals exceeding double precision is stored in `ast->raw_int`.
  Codegen emits a boxed `rt_make_int` or `rt_make_float` call depending on
  whether the value has a fractional part. Type inference assigns `TYPE_INT`
  for integer literals and `TYPE_FLOAT` for floating-point literals. The
  predicate `Number?` accepts both.

[OBS id:obs.language.construct.symbol src:reader.h:41,reader.c conf:high]
  `AST_SYMBOL` represents symbol identifiers. The reader interns the symbol
  name as a string in `ast->symbol` (reader.h:147). Symbols are the primary
  mechanism for variable references and dispatch head detection — when a
  symbol appears as the first element of an `AST_LIST`, codegen dispatches
  on its name via strcmp. Symbols appearing in function position are resolved
  to environment bindings; standalone symbol expressions are compiled as
  variable references. The type `TYPE_SYMBOL` is a ground type; symbol names
  are interned in the global symbol table.

[OBS id:obs.language.construct.string src:reader.h:42,codegen.c:3999-4012 conf:high]
  `AST_STRING` represents string literals. The reader stores the string content
  (without surrounding quotes) in `ast->string` (reader.h:148). Codegen emits
  an LLVM `GlobalStringPtr` for the string data. String indexing `("str" i)`
  is handled inline in the codegen dispatch (codegen.c:3999-4012) by loading
  the i-th character as a `TYPE_CHAR`. Type inference assigns `TYPE_STRING`.

[OBS id:obs.language.construct.char src:reader.h:43,reader.c conf:high]
  `AST_CHAR` represents character literals. The reader parses `'a'`, `#\a`,
  and Unicode escape syntax into an 8-bit `char character` payload
  (reader.h:149). Wisp expansion handles escape sequences (`'\t'` → tab)
  before the reader sees them, with a known limitation that `'\n'` fails
  because Wisp expands `\\n` to a real newline splitting the literal across
  lines. Type inference assigns `TYPE_CHAR`.

[OBS id:obs.language.construct.keyword src:reader.h:46,reader.c conf:high]
  `AST_KEYWORD` represents keyword literals. The reader parses `:keyword` tokens
  into `ast->keyword` (reader.h:150), storing the name without the leading `:`.
  Keywords are self-evaluating — they evaluate to themselves. They are used as
  named parameters, map keys, and record field labels. Type inference assigns
  `TYPE_KEYWORD`.

[OBS id:obs.language.construct.ratio src:reader.h:47,codegen.c:4078-4099 conf:high]
  `AST_RATIO` represents ratio literals. The reader parses `1/3`, `22/7`, etc.
  into `numerator` and `denominator` (reader.h:153-156). Codegen handles ratio
  in function position specially: `(1/4 x)` compiles to `(x * 1) / 4` for
  integer arguments (codegen.c:4078-4099). Type inference assigns `TYPE_RATIO`.
  The runtime representation is a `RuntimeValue` with tag `RT_RATIO` storing
  the reduced numerator/denominator.
