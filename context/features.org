[OBS id:obs.features.detect_features src:features.h:7,features.c:15 conf:high]
  =detect_features()= → =AST *= — Scans preprocessor defines for OS, architecture,
  endianness, binary format, CPU instruction sets, and build type. Returns an
  =AST_LIST= of keyword nodes (e.g. =:LINUX=, =:X86-64=, =:ELF=, =:SSE2=,
  =:DEBUG=). Also populates an internal static =detected_features= array used by
  =has_feature=. Called during environment initialization.

[OBS id:obs.features.has_feature src:features.h:11,features.c:245 conf:high]
  =has_feature(feature_name)= → =int= — Linear search of the internal
  =detected_features= array. Strips a leading ='=':'= character if present.
  Returns 1 if the feature is detected, 0 otherwise. Called by the reader during
  conditional compilation block parsing.

[OBS id:obs.features.internal-static src:features.c:7-13 conf:high]
  =detected_features= and =feature_count= are file-static globals. =detect_features=
  frees and rebuilds them on each call. =has_feature= reads them without
  modification.

[OBS id:obs.features.reader-lexer src:reader.c:1279-1293 conf:high]
  The reader lexer recognizes =#+= as =TOK_FEATURE_BEGIN= and =#---= as
  =TOK_FEATURE_END=. Feature names follow =#+= (e.g. =#+LINUX=, =#+X86-64=).

[OBS id:obs.features.reader-parser src:reader.c:6645-6687 conf:high]
  =parse_feature_blocks= reads feature conditionals. Uses =has_feature= to check
  each feature name. Content inside a disabled feature block is skipped; content
  inside an enabled block is parsed normally. Nested =#+= opens sibling blocks at
  the same level. =is_feature_active= delegates to =has_feature=.

[OBS id:obs.features.reader-integration src:reader.c:6709-6710 conf:high]
  =parse_all= checks for =TOK_FEATURE_BEGIN= at the top level and calls
  =parse_feature_blocks= to handle conditional compilation before normal parsing.

[THINK id:think.features.conditional-syntax conf:high]
  The =#+LINUX= / =#+X86-64= / =#---= syntax for feature conditionals
  is clean and minimal. It reuses the existing comment-like prefix (#+)
  to introduce feature blocks, making it visually distinct from code
  while being easy to parse. The =#---= terminator is less obvious —
  it looks like a malformed comment rather than a deliberate end marker.
  A more explicit terminator like =#+END= or =#+ENDFEATURE= would be
  clearer, but the current syntax has the advantage of being short and
  symmetric with the opening markers. The choice matters because
  terminal-friendly feature conditionals must be easy to type.

[THINK id:think.features.platform-vs-version-flags conf:high]
  The feature system serves two distinct purposes that share the same
  syntax: platform detection (is this Linux? x86-64?) and feature
  versioning (is experimental-foo available?). Platform features are
  static per compilation (determined at configure time by detect_features).
  Version features can change as the compiler evolves. Treating them
  identically means a version feature disabled in v0.4 might produce
  a "feature not found" error in v0.3 that compiled fine — the error
  message is about the present compiler, not the past source. A
  deprecation mechanism for version features (=#+deprecated-feature= with
  a warning rather than an error) would smooth transitions.

[THINK id:think.features.detect-vs-inject conf:medium]
  The current flow (detect_features writes to env, reader checks has_feature)
  is simple but couples platform detection to the environment setup phase.
  An alternative: feature detection as a separate pre-pass that generates
  a feature bitmask, which the reader checks via a simple array lookup.
  This would decouple feature detection from environment initialization
  and allow the reader to run in a feature-feature mode without a full env.
  The current coupling is fine for the CLI compiler but would be an issue
  if the reader is extracted as a library.

[THINK id:think.features.reader-integration-point conf:medium]
  The feature block parsing happens inside the reader (parse_feature_blocks
  at reader.c:6645), which means feature conditionals are handled at the
  syntax level before macro expansion. This is the right place for
  platform conditionals (you want to exclude platform-specific code
  before any semantic analysis), but it means feature conditionals
  cannot reference macro-defined features (the macro expander hasn't
  run yet). If the user writes =#+MY_MACRO_FEATURE=, it will not be
  recognized because MY_MACRO_FEATURE hasn't been defined yet. This is
  a documented limitation of the current approach.

[THINK id:think.features.test-utility conf:medium]
  The feature system is valuable for testing: you can write test cases
  that exercise platform-specific code paths by setting feature flags
  in the test environment. The =tests/run.py= harness could support a
  =--features= flag that injects feature keywords into the test
  environment, enabling tests like "compile this Linux-specific code
  on macOS by setting the LINUX feature flag." This would dramatically
  increase test portability. Currently, platform-specific tests are
  silently skipped on the wrong platform (the feature block excludes
  them), which means the test suite covers less on non-Linux platforms.

  This section declares the category contract for =features=. It is the
  routing target for failures and future test/context links touching this area.

#+CAPTION: Features Category Contract
| Field | Contract |
|-------+----------|
| Owned phase | Platform feature detection, conditional compilation flag resolution, feature environment initialization. |
| Inputs | Preprocessor defines (=__linux__=, =__x86_64__=, etc.), feature keyword symbols (=:LINUX=, =:X86-64=). |
| Outputs | =AST_LIST= of detected feature keywords, =*features*= environment binding, boolean =has_feature= lookup. |
| Invariants | Feature detection is a compile-time, one-shot pass. =detect_features= must be called before any conditional compilation. Feature names are case-sensitive keywords. =has_feature= is read-only after initialization. |
| Test hooks | Platform-specific tests under =tests/features/=, reader conditional compilation tests exercising =#+LINUX= / =#+X86-64= blocks. |
| Failure symptoms | Feature flag not recognized, wrong platform detected, =has_feature= returns 0 for a present feature, environment has no =*features*= binding. |
| Context neighbors | reader, env, source-map, tests |

[DEC id:dec.features.category-contract-owner date:2026-06-20 by:llm]
  Failures matching the symptoms above should start triage in this category, then
  follow the listed neighbors only when the local contract does not explain the
  issue.
