[OBS id:obs.repl.file src:repl.h,repl.c conf:high]
  The interactive REPL (Read-Eval-Print Loop) provides a JIT-compiled
  environment where users enter Monad expressions and see results
  immediately. Built on top of LLVM's =ExecutionEngine= for in-process
  JIT compilation and =readline= for line editing and tab-completion.
  Header at =repl.h=, implementation in =repl.c= (≈3000 lines).

[OBS id:obs.repl.context-struct src:repl.h:11-16 conf:high]
  =REPLContext= holds all persistent REPL state:
  - =engine= (LLVMExecutionEngineRef): persistent JIT engine across expressions.
  - =cg= (CodegenContext): module/builder replaced per expression.
  - =expr_count= (unsigned int): monotonically increasing expression counter
    used to generate unique wrapper names.
  - =infer_env= (InferEnv *): persistent Hindley-Milner type environment
    accumulating definitions across REPL inputs.

  =REPLContext=, configures readline (history file,

[OBS id:obs.repl.eval-line src:repl.c:2026 conf:high]
  =bool repl_eval_line(REPLContext *ctx, const char *line)= — The core
  evaluation function for a single input line. The JIT pipeline:
  1. =parse=: parse the input string into an AST.
  2. =repl_infer= (line 2008): run HM type inference on the AST.
  3. =codegen_expr=: compile the AST to LLVM IR in a fresh module.
  4. =LLVMExecutionEngine_addModule=: JIT the module.
  5. Look up the wrapper function by name and call it via
     =LLVMGenericValueToFloat=/=LLVMRunFunction=.
  6. Print the result value.
  On parse or inference error, uses =longjmp= (via =READER_ERROR= macro or
  =g_repl_escape=) to recover without exiting the process.

[OBS id:obs.repl.eval-line-error-recovery src:repl.c:2026 conf:high]
  Error recovery in =repl_eval_line=: before each evaluation, a
  =jmp_buf= (=g_repl_escape=) is set. If the reader encounters a
  fatal error (via =READER_ERROR= macro), it =longjmp=s back to the
  REPL prompt instead of calling =exit(1)=. If inference or codegen
  fails, error flags are checked and the line is skipped gracefully.

[OBS id:obs.repl.infer src:repl.c:2008 conf:high]
  =static bool repl_infer(REPLContext *ctx, AST *ast)= — Runs HM type
  inference on the parsed AST. Returns false on failure.

[DEC id:dec.repl.jit-pipeline from:obs.repl.eval-line by:codex date:2026-06-13]
  The REPL creates a new LLVM module for every input expression, compiles
  it, adds it to the execution engine, runs it, then discards the module.
  Function definitions persist in the =infer_env= but LLVM modules are
  ephemeral — each expression is a self-contained compilation unit that
  can call previously defined functions via the execution engine's symbol
  table.

[THINK id:think.repl.module-perf from:obs.repl.eval-line conf:medium]
  Creating a new module per expression is simple but may be wasteful for
  large modules. The =compile_one= path in main.c caches compiled objects.
  If REPL performance becomes an issue, consider reusing a single module
  with =LLVMCloneModule= or accumulating declarations.

[DOC id:doc.repl.category-contract src:context/repl.org conf:high]
  This section declares the category contract for =repl=. It is the
  routing target for failures and future test/context links touching the
  interactive read-eval-print loop.

#+CAPTION: REPL Category Contract
| Field | Contract |
|-------+----------|
| Owned phase | Interactive read-eval-print loop, expression-at-a-time compilation and execution, REPL state and environment persistence across inputs. |
| Inputs | User input lines, interactive meta-commands, ReaderEnv and InferEnv seeded with builtins. |
| Outputs | Evaluated expression results via runtime printing, persistent environment across REPL expressions, error recovery without process termination. |
| Invariants | Each expression is parsed, inferred, and codegen'd independently; environment persists across expressions within a REPL session; parse errors (READER_ERROR via longjmp) are caught and reported without terminating the session. |
| Test hooks | REPL interaction test fixtures, input/output round-trip tests, error recovery tests, environment persistence tests across multiple expressions. |
| Failure symptoms | Expression parse failure in REPL context, state corruption across evaluations, longjmp crash on read error, builtin environment not seeded correctly, codegen crash on REPL-compiled expression. |
| Context neighbors | main, cli, codegen-ops, infer, reader, runtime |

[DEC id:dec.repl.category-contract-owner date:2026-06-18 by:big-pickle]
  Failures matching the symptoms above should start triage in this category, then
  follow the listed neighbors only when the local contract does not explain the
  issue.

[FIX id:fix.repl.orc-persistent-value-lifetime-2026-07-24 date:2026-07-24 by:codex src:repl.c,codegen.c,tests/test_repl.py conf:high]
  REPL environment entries must never inspect an =LLVMValueRef= after its
  module has been transferred to ORC. Persistent variables are re-declared from
  language-level environment metadata and read through exported getters backed
  by stable host storage. The built-in =*features*= binding follows that same
  path instead of creating a special cross-module data dependency.

  =fresh_module= disposes an existing non-transferred module before replacing
  it; =close_and_run= clears the module pointer when ORC takes ownership. This
  removes one abandoned LLVM module per expression. The subprocess suite runs
  the public REPL binary through pipes and EOF, covering the exact
  =show *features*= regression, repeated JIT modules, heap and scalar state,
  multiline input, recovery, commands, imports, parallel import isolation, and
  a 150-expression session.

[FIX id:fix.repl.core-prelude-and-multiline-wisp-2026-07-24 date:2026-07-24 by:codex src:repl.c,tests/test_repl.py conf:high]
  The REPL silently imports the core-owned =Function= module once during
  initialization. Core functions such as =id= therefore contribute their real
  exported symbols and arities; the compiler does not hardcode their semantics.
  =id 3= is a single Wisp application and evaluates to =3=.

  The input accumulator recognizes a known function applied to fewer operands
  than its arity as the beginning of an indentation-delimited Wisp form.
  General multiline calls now wait for their blank-line terminator in the same
  way as typed definition blocks. Parenthesized multiline forms remain
  delimiter-driven and may finish at EOF.

  The REPL integration suite discovers every =module= declaration under
  =core/=, imports all of them through one live subprocess session, rejects
  compiler/runtime diagnostics, and evaluates a final sentinel. This contract
  caught broken terminal inline assembly, stale Set folding code, debug output,
  and a source-harvesting scanner that treated parentheses inside quoted strings
  as forms. The focused suite now contains 19 subprocess tests.

[FIX id:fix.repl.terminal-protocol-and-parser-completeness-2026-07-24 date:2026-07-24 by:codex src:repl.c,wisp.c,wisp.h,reader.c,tests/test_repl.py,tests/test_repl_pty.py conf:high]
  Interactive behavior is a public protocol, not an implementation detail.
  The regression suite therefore drives the shipped binary through a real
  pseudo-terminal as well as pipes. It verifies the colored readline prompt,
  core-prelude evaluation, blank-line-terminated indented Wisp, cancellation of
  an unfinished form with Ctrl-C, recovery on the next expression, and clean
  Ctrl-D exit.

  Complete/incomplete/invalid input classification belongs to =wisp= rather
  than to a second delimiter and arity parser in =repl.c=. The REPL waits only
  on =WISP_INPUT_INCOMPLETE=; complete and invalid forms go through the normal
  parser so diagnostics retain canonical source locations. The classifier
  understands nested heterogeneous delimiters, strings, character literals,
  comments, typed definition blocks, and known fixed arities. Invalid input is
  covered through recovery to a subsequent valid form without leaking internal
  debug output.

[FIX id:fix.repl.content-addressed-cache-2026-07-24 date:2026-07-24 by:codex src:repl.c,tests/test_repl_cache.py conf:high]
  Imported native images use the Monad REPL cache under
  =$XDG_CACHE_HOME/monad/repl/v1=, falling back to
  =$HOME/.cache/monad/repl/v1=. Cache identity is derived from an explicit ABI
  version, module name, ordered object contents, runtime archive, LLVM linker
  flags, and the running compiler binary. Artifacts are immutable and published
  by atomic rename from process-private linked images.

  A per-module build lease covers object compilation and publication, preventing
  concurrent cold REPL processes from racing on shared object-cache paths.
  Abandoned leases expire, malformed/non-regular/foreign-owned artifacts are
  rejected, and corrupt images are removed and rebuilt. =MONAD_CACHE=off=
  disables both storage and leases; =MONAD_CACHE_TRACE=1= exposes
  =store=/=hit=/=repair= decisions for diagnosis and tests.

  The executable cache suite covers cold store, warm hit, corruption repair,
  six-way concurrent cold startup, XDG placement, and explicit opt-out. On the
  reference development machine, empty-session startup improves from about
  0.49 seconds cold to 0.048 seconds warm.
