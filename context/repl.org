[OBS id:obs.repl.file src:repl.h,repl.c conf:high]
  The interactive REPL (Read-Eval-Print Loop) provides a JIT-compiled
  environment where users enter Monad expressions and see results
  immediately. Built on top of LLVM's =ExecutionEngine= for in-process
  JIT compilation and =readline= for line editing and tab-completion.
  Header at =repl.h=, implementation in =repl.c= (≈3000 lines).

[OBS id:obs.repl.context-struct src:repl.h:11-16 conf:high]
  =REPLContext= holds all persistent REPL state:
  - =engine= (LLVMExecutionEngineRef): persistent JIT engine across expressions.
  - =cg= (CodegenContext): module/builder replaced per expression.
  - =expr_count= (unsigned int): monotonically increasing expression counter
    used to generate unique wrapper names.
  - =infer_env= (InferEnv *): persistent Hindley-Milner type environment
    accumulating definitions across REPL inputs.

[OBS id:obs.repl.run src:repl.c:2885 conf:high]

  =REPLContext=, configures readline (history file,
  =rl_attempted_completion_function=), and enters an interactive
  read-compile-eval-print loop. Accumulates partial input until a
  complete expression is formed (balanced parens), then calls

  the context.

[OBS id:obs.repl.eval-line src:repl.c:2026 conf:high]
  =bool repl_eval_line(REPLContext *ctx, const char *line)= — The core
  evaluation function for a single input line. The JIT pipeline:
  1. =parse=: parse the input string into an AST.
  2. =repl_infer= (line 2008): run HM type inference on the AST.
  3. =codegen_expr=: compile the AST to LLVM IR in a fresh module.
  4. =LLVMExecutionEngine_addModule=: JIT the module.
  5. Look up the wrapper function by name and call it via
     =LLVMGenericValueToFloat=/=LLVMRunFunction=.
  6. Print the result value.
  On parse or inference error, uses =longjmp= (via =READER_ERROR= macro or
  =g_repl_escape=) to recover without exiting the process.

[OBS id:obs.repl.eval-line-error-recovery src:repl.c:2026 conf:high]
  Error recovery in =repl_eval_line=: before each evaluation, a
  =jmp_buf= (=g_repl_escape=) is set. If the reader encounters a
  fatal error (via =READER_ERROR= macro), it =longjmp=s back to the
  REPL prompt instead of calling =exit(1)=. If inference or codegen
  fails, error flags are checked and the line is skipped gracefully.

[OBS id:obs.repl.infer src:repl.c:2008 conf:high]
  =static bool repl_infer(REPLContext *ctx, AST *ast)= — Runs HM type
  inference on the parsed AST. Returns false on failure.

[DEC id:dec.repl.jit-pipeline from:obs.repl.eval-line by:codex date:2026-06-13]
  The REPL creates a new LLVM module for every input expression, compiles
  it, adds it to the execution engine, runs it, then discards the module.
  Function definitions persist in the =infer_env= but LLVM modules are
  ephemeral — each expression is a self-contained compilation unit that
  can call previously defined functions via the execution engine's symbol
  table.

[THINK id:think.repl.module-perf from:obs.repl.eval-line conf:medium]
  Creating a new module per expression is simple but may be wasteful for
  large modules. The =compile_one= path in main.c caches compiled objects.
  If REPL performance becomes an issue, consider reusing a single module
  with =LLVMCloneModule= or accumulating declarations.

[DOC id:doc.repl.category-contract src:context/repl.org conf:high]
  This section declares the category contract for =repl=. It is the
  routing target for failures and future test/context links touching the
  interactive read-eval-print loop.

#+CAPTION: REPL Category Contract
| Field | Contract |
|-------+----------|
| Owned phase | Interactive read-eval-print loop, expression-at-a-time compilation and execution, REPL state and environment persistence across inputs. |
| Inputs | User input lines, interactive meta-commands, ReaderEnv and InferEnv seeded with builtins. |
| Outputs | Evaluated expression results via runtime printing, persistent environment across REPL expressions, error recovery without process termination. |
| Invariants | Each expression is parsed, inferred, and codegen'd independently; environment persists across expressions within a REPL session; parse errors (READER_ERROR via longjmp) are caught and reported without terminating the session. |
| Test hooks | REPL interaction test fixtures, input/output round-trip tests, error recovery tests, environment persistence tests across multiple expressions. |
| Failure symptoms | Expression parse failure in REPL context, state corruption across evaluations, longjmp crash on read error, builtin environment not seeded correctly, codegen crash on REPL-compiled expression. |
| Context neighbors | main, cli, codegen-ops, infer, reader, runtime |

[DEC id:dec.repl.category-contract-owner date:2026-06-18 by:big-pickle]
  Failures matching the symptoms above should start triage in this category, then
  follow the listed neighbors only when the local contract does not explain the
  issue.
