[OBS id:obs.pmatch.file src:pmatch.h,pmatch.c conf:high]
  The pattern-match compiler translates =pattern -> expr= clauses into
  equivalent =(cond ...)= chains that the generic codegen can lower. It
  lives at the boundary between the reader (which produces =AST_PMATCH= nodes)
  and codegen (which calls =pmatch_desugar= before compiling a lambda body).
  Header at =pmatch.h=, implementation in =pmatch.c= (≈1270 lines).

[INF id:inf.pmatch.file-connect from:obs.pmatch.file conf:high]
  connects-to -> `monadc.context.language.pattern-matching` (surface syntax) ·
  `monadc.context.codegen-ops` (desugared cond chains lowered by codegen) ·
  `monadc.context.reader` (AST_PMATCH / ASTPattern produced by reader)

[THINK id:think.pmatch.desugar-timing from:obs.pmatch.file conf:medium]
  Desugaring happens during codegen, not parsing, because the full type
  environment and ADT definitions are only available after inference.
  An alternative design would desugar earlier and let infer see the cond
  chains, but the current pipeline desugars right before LLVM IR emission.

[OBS id:obs.pmatch.desugar-pipeline src:pmatch.c:1050 conf:high]
  =AST *pmatch_desugar(AST *node, ASTParam *params, int param_count)= —
  Transforms an =AST_PMATCH= node into an equivalent =(cond ...)= AST.
  Called from codegen before compiling the lambda body. The returned AST
  is freshly allocated and owned by the caller. The pipeline:
  1. For each clause, generate pattern-match conditions via
     =build_pattern_conditions= (line 699) which recursively decomposes
     patterns into equality checks, type-tag checks, list-length tests,
     and nested destructuring.
  2. Wrap guards as =(and guard condition)=.
  3. Chain into =(cond (c1 body1) (c2 body2) ... (else fallback))=.

[OBS id:obs.pmatch.rename-anon-params src:pmatch.c:1037 conf:high]
  =void pmatch_rename_anon_params(AST *pm, ASTParam *params, int param_count)= —
  Walks the pattern AST and replaces anonymous parameter references with
  their generated names. Must be called before =pmatch_desugar= so that
  patterns refer to the correct variable names.

[OBS id:obs.pmatch.build-pattern-conditions src:pmatch.c:699 conf:high]
  =static void build_pattern_conditions(...)= — The core recursive pattern
  compiler. For each =PatternKind= it emits the corresponding condition
  expression:
  - =PAT_WILDCARD=: always true
  - =PAT_VAR=: always true (binds the variable)
  - =PAT_LITERAL_INT=: =(= param value)=
  - =PAT_LITERAL_FLOAT=: =(= param value)=
  - =PAT_LITERAL_STRING=: =(string= param value)=
  - =PAT_LIST_EMPTY=: =(empty? param)=
  - =PAT_LIST=: length check + element destructuring via =nth=
  - =PAT_CONSTRUCTOR=: tag check via =__adt_tag_= + field destructuring

[OBS id:obs.pmatch.helper-builders src:pmatch.c:479-672 conf:high]
  Suite of static helper constructors that build AST nodes for the
  desugared output: =sym=, =make_list=, =make_eq_int=, =make_eq_float=,
  =make_true=, =make_list_length=, =make_count=, =make_coll_wrap=,
  =make_coll_is_empty=, =make_count_eq=, =make_index_access=, =make_and=,
  =ast_substitute=, =make_let=, =rewrite_array_literals=. Each produces
  small AST fragments that combine into the final cond chain.

[OBS id:obs.pmatch.pmatch-rename-anon-params src:pmatch.c:1037 conf:high]
  Renames auto-generated anonymous parameter placeholders in pattern
  clauses to match the actual generated parameter names. Without this,
  anonymous parameters (e.g. =(lambda (Int -> Int) ...)=) would not bind
  correctly during desugaring.

[OBS id:obs.pmatch.pipeline-flow src:pmatch.c:699,1037,1050 conf:high]
  Complete desugaring pipeline:
  1. Reader calls =parse_pmatch_clauses= during parsing, producing
     =AST_PMATCH= node with =ASTPMatchClause= array.
  2. =pmatch_rename_anon_params= rewrites anonymous parameter references.
  3. Codegen calls =pmatch_desugar= which:
     a. For each clause, calls =build_pattern_conditions= to generate
        a conjunction of equality / type-tag / length / destructuring checks.
     b. Wraps guard conditions with =(and ...)=.
     c. Chains clauses into =(cond (clause1 body1) ... (else default))=.
  4. Codegen compiles the resulting =(cond ...)= via the normal dispatch path.

[INF id:inf.pmatch.pipeline-codegen from:obs.pmatch.pipeline-flow conf:high]
  connects-to -> `monadc.context.codegen-ops.cond-handler` (cond is lowered
  via the standard codegen cond dispatch) ·
  `monadc.context.language.pattern-matching` (surface pattern syntax) ·
  `monadc.context.codegen-ops.strcmp.constructor-tag` (ADT tag checks
  emitted by build_pattern_conditions for PAT_CONSTRUCTOR)

[DEC id:dec.pmatch.pipeline-timing from:obs.pmatch.pipeline-flow by:codex date:2026-06-13]
  Pattern desugaring is intentionally deferred to codegen time. Early
  desugaring (during parsing) would lose the connection to the original
  pattern structure for error messages and source mapping. Desugaring
  after inference is unnecessary because patterns contain no type-dependent
  constructs. Codegen-time desugaring is the current sweet spot.

[THINK id:think.pmatch.list-patterns-inference from:obs.pmatch.pipeline-flow conf:medium]
  List patterns desugar to =(empty? ...)= and =(nth ...)= calls. These
  are not special-cased in the inferrer; they rely on the generic
  =(empty? ...)= and =(nth ...)= type signatures registered in
  =register_builtins=. If list type inference becomes a bottleneck,
  consider adding first-class list pattern support to inference.

[DOC id:doc.pmatch.category-contract src:context/pmatch.org conf:high]
  This section declares the category contract for =pmatch=. It is the
  routing target for failures and future test/context links touching pattern
  match compilation and desugaring.

#+CAPTION: Pattern Match Compiler Category Contract
| Field | Contract |
|-------+----------|
| Owned phase | Pattern match desugaring from AST_PMATCH into nested cond/empty?/nth/car chains, pattern coverage and exhaustiveness checking. |
| Inputs | AST_PMATCH nodes with ASTPattern trees, typed scrutinee expression. |
| Outputs | Desugared cond expressions with nested empty?/nth/car guard chains, exhaustive coverage verification result. |
| Invariants | Desugaring preserves match semantics (first-match-wins); nested patterns decompose correctly into guard chains; exhaustiveness check covers all constructors; non-exhaustive matches produce a diagnostic. |
| Test hooks | Pattern match test fixtures, coverage checking tests, nested pattern decomposition tests. |
| Failure symptoms | Wrong branch taken at runtime, incomplete pattern match not detected, nested pattern desugar error, guard chain ordering inverted. |
| Context neighbors | reader, codegen-ops, infer, language, tests |

[DEC id:dec.pmatch.category-contract-owner date:2026-06-18 by:big-pickle]
  Failures matching the symptoms above should start triage in this category, then
  follow the listed neighbors only when the local contract does not explain the
  issue.
