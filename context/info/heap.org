Monad uses heap allocation for data that outlives its defining scope or has runtime-determined size. Heap-allocated objects use bump allocation (arena) or =malloc= depending on context.

For the runtime allocator implementation, see [[file:../context/runtime.org][context/runtime.org]]. For the arena-based compiler allocator, see [[file:arena.org][Arena Allocation]].

This page covers heap allocation in Monad: heap arrays, ADT allocation,
the runtime memory model, and the tradeoffs between heap and stack storage.

* 1. Heap Array Syntax

Heap arrays use the =~= tilde prefix:

#+BEGIN_SRC monad
~[1 2 3]                          ;; heap-allocated Int array
~[]                               ;; empty heap array
(define heap-arr ~[1 2 3])
#+END_SRC

#+BEGIN_SRC wisp
~[1 2 3]                          ;; heap-allocated Int array
~[]                               ;; empty heap array
define heap-arr ~[1 2 3]
#+END_SRC

See [[file:arrays.org][Arrays]] for the full array variant reference, and
[[file:performance.org][Performance Guide]] for the three array forms and their
performance characteristics.

*** Key Points
- =~[~ prefix distinguishes heap arrays from stack arrays
- Heap arrays support dynamic resizing (push/pop)
- Same element-type system as stack arrays

See also: [[file:arrays.org][Arrays]]

* 2. When Heap Allocation Happens

| Construct             | Allocation    | Reason                     |
|-----------------------+---------------+----------------------------|
| =~[1 2 3]= (heap array) | Heap (malloc) | Explicit heap request      |
| ADT constructors      | Heap (malloc) | Tagged union, dynamic size |
| Cons cells            | Heap (malloc) | Persistent linked lists    |
| Closures (planned)    | Heap          | Environment capture        |
| =[16kb I32]= (fat)      | Stack or heap | Depends on size            |
| =[1 2 3]= (stack)       | Stack frame   | Known size at compile time |

For ADT heap allocation, see [[file:data.org][Data Types]]. For cons cells and
lists, see [[file:list.org][Lists]].

*** Key Points
- Stack allocation: compile-time-known size, scope-bound lifetime
- Heap allocation: dynamic size or data that outlives its scope
- ADTs and cons cells always heap-allocate — their size is dynamic

See also: [[file:data.org][Data Types]], [[file:list.org][Lists]]

* 3. Runtime Memory Model

** 3.1 Arena Allocator ([[file:../arena.c]])

The compiler uses a bump allocator for internal allocations:

#+BEGIN_SRC c
Arena arena;                     // compiler-internal arena
arena_alloc(&arena, size);       // bump-allocate
arena_reset(&arena);             // free all at once
#+END_SRC

Full details in [[file:arena.org][Arena Allocation]] and
[[file:../context/arena.org][context/arena.org]].

*** Key Points
- Compiler-internal allocations are arena-managed, not malloc
- Batch free via =arena_reset= — no per-object tracking
- Separate arena for permanent vs scratch data

** 3.2 Runtime Allocator ([[file:../runtime.c]])

Runtime heap allocations go through the runtime's allocation layer,
which uses a dedicated eval arena and direct =malloc= for some types:

| Function / Helper       | Allocation Source | Purpose                    |
|-------------------------+-------------------+----------------------------|
| =alloc_cons_cell=       | Eval arena        | Cons cell allocation       |
| =alloc_list_wrapper=    | Eval arena        | List wrapper header        |
| =alloc_value=           | Eval arena        | RuntimeValue struct        |
| =rt_value_closure=      | =malloc=          | Closure environment        |
| =heap_cons_cell= (cold) | =malloc=          | Cold-path cons allocation  |

Internally, the runtime uses =g_eval_arena= (an =Arena= instance) for
the hot allocation path — same bump-pointer strategy as the compiler.
See [[file:../runtime.c:99][runtime.c:99 — Internal allocation helpers]] and
[[file:../context/runtime.org][context/runtime.org]].

See [[file:../context/language.org::runtime-value-system][context/language.org — Runtime Value System]] and
[[file:../context/runtime.org][context/runtime.org]].

* 4. Heap Arrays vs Stack Arrays

| Aspect     | Stack Array =[1 2 3]= | Heap Array =~[1 2 3]=   |
|------------+---------------------+-----------------------|
| Allocation | Stack frame         | Heap (malloc)         |
| Lifetime   | Scope-bound         | Manual or GC          |
| Resizable  | No                  | Yes (push/pop)        |
| C-interop  | Yes (decays to ptr) | Yes (decays to ptr)   |
| Overhead   | None                | Allocation + capacity |

For the fat-pointer array variant, see [[file:arrays.org][Arrays]]. For
C-interop considerations, see [[file:layouts.org][Layout Types]] and
[[file:ffi.org][C FFI]].

*** Key Points
- Stack arrays: zero overhead, fixed size, scope-bound lifetime
- Heap arrays: dynamic size, explicit allocation cost
- Both decay to pointers for C interop

See also: [[file:arrays.org][Arrays]], [[file:layouts.org][Layout Types]]

* 5. ADT Heap Allocation

Algebraic data types ([[file:data.org][Data Types]]) are heap-allocated tagged unions:

#+BEGIN_SRC monad
(data Tree a Leaf (Node Tree a a Tree))

;; Node allocates a heap record with tag + fields
(Node Leaf 42 Leaf)
#+END_SRC

#+BEGIN_SRC wisp
data Tree a Leaf (Node Tree a a Tree)

;; Node allocates a heap record with tag + fields
Node Leaf 42 Leaf
#+END_SRC

The tag byte is followed by constructor fields. Pattern matching
([[file:patterns.org][Pattern Matching]]) reads the tag to dispatch. See
[[file:../context/pmatch.org][context/pmatch.org]] for the pattern match compiler.

*** Key Points
- ADTs are heap-allocated as tag + fields in a contiguous block
- Tag byte enables O(1) constructor dispatch in pattern matching
- Size varies per constructor — allocation via eval arena

See also: [[file:patterns.org][Pattern Matching]]
