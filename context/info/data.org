* 1. Data Declaration Syntax

Algebraic data types are declared with the =(data ...)= form. The type name
must start with an uppercase letter (enforced — compiler error otherwise).
Constructors follow the type name; each constructor name must also start with
uppercase. Type parameters are bare symbols placed before constructors.

See also [[file:types.org][Types]] for the type system context and
[[file:layouts.org][Layouts]] for the C-interop counterpart.

** 1.1 Parenthesized (S-expression) syntax

#+BEGIN_SRC monad
(data Bool False True)                          ;; two nullary constructors
(data Option a None (Some a))                   ;; polymorphic Option
(data Tree a Leaf (Node Tree a a Tree))          ;; recursive tree
(data Point Float Float)                        ;; single-constructor shorthand
#+END_SRC

#+BEGIN_SRC wisp
data Bool False True
data Option a None (Some a)
data Tree a Leaf (Node Tree a a Tree)
data Point Float Float
#+END_SRC

The shorthand single-constructor form =(data Point Float Float)= expands to
=(data Point (Point Float Float))= — a data type whose only constructor shares
the type name. This is syntactic sugar provided by the reader; after parsing
there is no distinction between the shorthand and the explicit form.

For the Wisp syntax variant see [[file:wisp.org][Wisp]].

** 1.2 Pipe-separated syntax (=|= variants, =~= separator)

#+BEGIN_SRC monad
(data Color ~ Red | Green | Blue)
(data Maybe a = Nothing | Just a)
(data T = A | B | C)
#+END_SRC

#+BEGIN_SRC wisp
data Color ~ Red | Green | Blue
data Maybe a = Nothing | Just a
data T = A | B | C
#+END_SRC

The =~= or ==== separator sits between the type parameters and the constructors.
=|= separates alternative constructors. Both forms parse to the same =AST_DATA=
node (reader.h:59-61, reader.h:251-260). The reader normalises pipe syntax to an
internal representation before inference sees it.

See [[file:../context/reader.org][context/reader.org]] for reader internals and
[[file:../context/infer.org][context/infer.org]] for inference of ADT types.

** 1.3 Constructor naming

Constructors must start with an uppercase letter. Binding names (variables,
function parameters) must start with a lowercase letter. Constructor names are
reserved — they cannot be used as variable names in patterns where it would
cause ambiguity. This lexical distinction eliminates the need for a separate
keyword to introduce constructor patterns.

See [[file:../context/language.org][context/language.org]] for the full language
formalization and [[file:syntax.org][Syntax]] for lexical rules.

Key Points:
- Type name and all constructor names must start with uppercase
- Type parameters are bare symbols placed before constructors
- Single-constructor shorthand =(data Point Float Float)= auto-creates =(Point Point Float Float)=
- Pipe syntax (=~=, ====, =|=) normalises to the same =AST_DATA= node as S-expression form
- Uppercase vs lowercase naming resolves constructor vs variable ambiguity lexically

* 2. Deriving (Aspirational)

Deriving provides auto-generated instance implementations for common typeclasses:

#+BEGIN_SRC monad
(data Color Red Green Blue deriving Eq)
(data Suit deriving [Eq Enum])
(data T = A | B | C deriving [Bounded Enum Show])
#+END_SRC

#+BEGIN_SRC wisp
data Color Red Green Blue deriving Eq
data Suit deriving [Eq Enum]
data T = A | B | C deriving [Bounded Enum Show]
#+END_SRC

The =deriving= clause appears at the end of a data declaration. Multiple classes
use a list: =deriving [Eq Enum]=. The clause is parsed by the reader and stored
in the =AST_DATA= payload (reader.h:251-260).

See [[file:type-classes.org][Type Classes]] for the class system and
[[file:../context/typeclass.org][context/typeclass.org]] for implementation details.

** 2.1 Current state of deriving

Implementing via =typeclass.c= ([[file:../typeclass.c]]), with the following
status:

| Class   | Status                     | Behaviour                                               |
|---------+----------------------------+---------------------------------------------------------|
| =Eq=    | Working for all constructors | Structural equality comparing tags and fields           |
| =Show=  | Working for all constructors | Pretty-prints as =(Ctor field0 field1 ...)=              |
| =Enum=  | Working for nullary only     | =fromEnum= returns tag index; =toEnum= maps tag → ctor; =succ= / =pred= step through tags |
| =Bounded| Partially working           | Requires nullary constructors for bounds                 |

Deriving for constructors with fields is work in progress for =Enum= and
=Bounded=. A warning is printed if the deriving class is not defined before
use. Classes must be defined (via =(class ...)=) before they can be derived.

See [[file:../context/codegen.org][context/codegen.org]] for codegen of derived instances
and [[file:equality.org][Equality]] for =Eq= semantics.

Key Points:
- =Eq= and =Show= work for all constructors, including those with fields
- =Enum= works for nullary constructors only; field-bearing constructors are WIP
- =deriving= clause is optional; multiple classes use bracket syntax =[Eq Show]=
- A warning is emitted if the derived class is not defined before use

* 3. Show for ADTs

ADT values are displayed via the built-in =show= function. The printer routes
layout-typed values through the ADT printer when the layout has registered
constructors. The canonical output format is parenthesised constructor syntax:

#+BEGIN_SRC monad
(show (Point 3.0 4.0))      ;; prints "(Point 3.0 4.0)"
(show Red)                   ;; prints "Red"
(show (Node Leaf 1 Leaf))    ;; prints "(Node Leaf 1 Leaf)"
#+END_SRC

#+BEGIN_SRC wisp
show (Point 3.0 4.0)
show Red
show (Node Leaf 1 Leaf)
#+END_SRC

The ADT show path in codegen ([[file:../codegen.c::5876]]) checks whether the
value's layout has registered constructors. If yes, it emits canonical data
syntax; otherwise it falls through to the generic runtime value printer.

See [[file:show.org][Show]] for the full show protocol and
[[file:../context/codegen.org][context/codegen.org]] for LLVM lowering.

Key Points:
- Nullary constructors print as their bare name (e.g., =Red=)
- Constructors with fields print as =(Ctor field0 field1 ...)=
- The printer falls through to the generic runtime printer if no constructors are registered

* 4. Pattern Matching on ADTs

ADT constructors are matched in pattern clauses using
=PatternKind::PAT_CONSTRUCTOR=:

#+BEGIN_SRC monad
(define (maybe->int [m :: (Option Int)] -> Int)
  (match m with
    (None)     -> 0
    (Some x)   -> x))

(define (tree-depth [t :: (Tree a)] -> Int)
  (match t with
    (Leaf)           -> 0
    (Node l _ r)     -> (+ 1 (max (tree-depth l) (tree-depth r)))))
#+END_SRC

#+BEGIN_SRC wisp
define maybe->int [m :: (Option Int)] -> Int
  match m with
    (None)     -> 0
    (Some x)   -> x

define tree-depth [t :: (Tree a)] -> Int
  match t with
    (Leaf)           -> 0
    (Node l _ r)     -> + 1 (max (tree-depth l) (tree-depth r))
#+END_SRC

Pattern match compilation ([[file:../context/pmatch.org][context/pmatch.org]]) desugars constructor
patterns to tag checks via =__adt_tag_= prefix convention
([[file:../codegen.h::16]]). The =build_pattern_conditions= function at
pmatch.c:699 emits tag-equality checks for =PAT_CONSTRUCTOR= patterns and
generates field destructuring via =nth= / =index-access=. Desugaring is deferred
to codegen time (pmatch.c:1050) so the full type environment and ADT definitions
are available — inference must run first to populate the type environment.

Constructor coverage: if the final clause is not an explicit wildcard (=_=), the
compiler emits an =(undefined)= fallback. Exhaustiveness-aware fallback removal
(where the compiler statically proves all branches are covered) is future work.

See [[file:patterns.org][Pattern Matching]] for full pattern language reference and
[[file:../context/language.org][context/language.org]] for working dispatch.

Key Points:
- Constructor patterns use =PAT_CONSTRUCTOR= with =__adt_tag_= tag checks
- Desugaring happens after inference so ADT definitions are available
- Always include a wildcard =_= fallback to avoid =(undefined)= at runtime
- Field destructuring uses =nth= / =index-access= in the generated condition chain

* 5. Memory Representation

Each ADT is a tagged union on the heap:

| Offset | Content                          |
|--------+----------------------------------|
| +0     | =i32= tag (constructor index)    |
| +8     | =i64= payload slot 0             |
| +16    | =i64= payload slot 1             |
| ...    | Up to max fields                 |

Each constructor compiles to a function that =malloc=s the struct, stores tag +
fields. Nullary constructors also get a global singleton so they are never
heap-allocated at call sites — the global is reused for every occurrence. The
tag is computed by =adt_constructor_tag= (codegen.h:16) which derives a stable
integer from the constructor name via a hash.

See [[file:heap.org][Heap Allocation]] for allocation details and
[[file:arena.org][Arena]] for arena-based allocation. Also see
[[file:../context/codegen.org][context/codegen.org]] for LLVM IR lowering of ADTs.

Key Points:
- Tag is an =i32= at offset 0; payload fields follow as =i64= slots
- Nullary constructors use global singletons (no heap allocation at call sites)
- Tags are stable integers derived from constructor name via hash
- Field count is implicit in the constructor definition; unused slots are zeroed

* 6. Comparison with =layout= types

| Aspect     | =data= (ADT)                 | =layout= (C interop)            |
|------------+------------------------------+---------------------------------|
| Tag        | Automatic =i32= tag          | No tag (untagged)               |
| Allocation | Heap (=malloc=)              | Stack or heap (explicit)        |
| Fields     | Named by position            | Named fields with offsets       |
| C compat   | No                           | Yes (struct-compatible)         |
| Use case   | Algebraic sum types          | C FFI, precise memory layout    |

The =layout= type ([[file:layouts.org]]) is the C-interop counterpart. ADTs
use runtime tag dispatch; layouts use positional field access. Both can be
printed via =show=, but only ADTs get the canonical =(Ctor field ...)= format.

See [[file:../context/ffi.org][context/ffi.org]] for FFI details and
[[file:../context/types.org][context/types.org]] for type representation.

Key Points:
- ADTs have automatic tag dispatch; layouts are untagged and C-compatible
- ADTs always heap-allocate; layouts can be stack-allocated
- Choose =data= for sum types, =layout= for precise memory control and C interop

* 7. Runnable Example

#+BEGIN_SRC monad
;; run with: monadc data-demo.mon

(data Bool False True)
(data Maybe a Nothing (Just a))
(data Tree a Leaf (Node Tree a a Tree))

(define (maybe->int [m :: (Maybe Int)] -> Int)
  (match m with
    (Nothing) -> 0
    (Just x)  -> x))

(define (tree-depth [t :: (Tree a)] -> Int)
  (match t with
    (Leaf)           -> 0
    (Node l _ r)     -> (+ 1 (max (tree-depth l) (tree-depth r)))))

(show (maybe->int (Just 42)))           ;; prints 42
(show (tree-depth (Node Leaf 1 (Node Leaf 2 Leaf))))  ;; prints 2
#+END_SRC

* 8. Gotchas

Common mistakes and surprising behaviours with ADTs:

- **Unused slots are zeroed** — the memory representation reserves =i64= slots for all fields. Constructors with fewer fields than the maximum leave unused slots zeroed but still-allocated.

- **Tag ordering is unstable** — tag values are derived from constructor name hashes (=adt_constructor_tag= at codegen.h:16). Never hardcode tag numbers; always use pattern matching.

- **Recursive types must use constructor fields** — bare recursion (=data T T=) is not allowed; recursion must go through a constructor field (e.g., =(Node Tree a a Tree)=).

- **Derived classes must be pre-defined** — =(class ...)= must be evaluated before =deriving= references it. The compiler emits a warning but does not error, which can lead to silent codegen failures.

- **=Enum= deriving is nullary-only** — deriving =Enum= for constructors with fields produces a warning and may misbehave at runtime.

- **Nullary constructors are singletons** — =(eq? Red Red)= compares pointer identity. This is usually fine but can surprise if you mutate singletons (which you should not).
