The CLI argument parser maps command-line arguments to a =CompilerFlags=
struct and dispatches to the appropriate command handler. Implementation:
[[file:../cli.h][cli.h]] (type definitions) and [[file:../cli.c][cli.c]] (~730 lines, all =cmd_*= functions).

For full implementation context, see [[file:../context/cli.org][context/cli.org]].
For the build info resolution and monad.json schema, see
[[file:../context/buildsystem.org][context/buildsystem.org]].

This page covers the Monad compiler CLI: command modes, compiler flags,
emit options, build info resolution, project scaffolding, and core path
resolution.

1. CommandMode Enum

The =CommandMode= enum (defined in =cli.h:5-14=) selects the compiler's
operating mode:

| Mode | Invocation | Description |
|-----------+----------------------------+-------------------------------------------------|
| =CMD_COMPILE= | =monad <file.mon>= | Compile a single source file (default with arg) |
| =CMD_REPL= | =monad= (no args) or =monad -i= | Launch interactive REPL (see [[file:repl.org][REPL]]) |
| =CMD_NEW= | =monad new <name>= | Scaffold a new project directory (see [[file:package-dev.org][Package Development]]) |
| =CMD_BUILD= | =monad build= | Build project from =monad.json= (see [[file:build.org][Build System]]) |
| =CMD_RUN= | =monad run [-- args...]= | Build and execute the project binary |
| =CMD_CLEAN= | =monad clean= | Remove build artifacts |
| =CMD_INSTALL= | =monad install= | Build and copy binary to =~/.local/bin= |
| =CMD_TEST= | =monad test [file]= | Compile with tests, run, delete binary (see [[file:testing.org][Testing]]) |

Command dispatch happens in =parse_flags= (cli.c:193) and is executed by =main=
in [[file:../main.c][main.c]].

*** Key Points
- Eight command modes covering compile, build, test, and project management
- Default mode is =CMD_COMPILE= when a file argument is present
- =CMD_REPL= activates when no arguments are given

See also: [[file:build.org][Build System]]

2. CompilerFlags Struct

The =CompilerFlags= struct (cli.h:16-29) aggregates all parsed CLI options:

| Field | Type | Set By | Purpose |
|------------------+---------------+---------------+-------------------------------|
| =mode= | CommandMode | subcommand argument | Operating mode |
| =emit_ir= | bool | =--emit-ir= | Emit LLVM IR (.ll) |
| =emit_bc= | bool | =--emit-bc= | Emit LLVM bitcode (.bc) |
| =emit_asm= | bool | =--emit-asm= | Emit assembly (.s) |
| =emit_obj= | bool | =--emit-obj= (default when compiling) | Emit object file (.o) |
| =emit_json= | bool | =--emit-json= | Emit JSON AST dump (.json) |
| =emit_typst= | bool | =--emit-typst= | Emit Typst output (.typ + PDF) |
| =start_repl= | bool | =-i= | Enter REPL after compilation |
| =test_mode= | bool | =--test= or =--test-mode= | Emit test blocks (see [[file:testing.org][Testing]]) |
| =test_run= | bool | =monad test= subcommand | Run tests, delete after |
| =output_name= | char * | =-o= or =--output= | Output binary or file path |
| =input_file= | char * | positional argument | Input source file |
| =package_name= | char * | =monad new <name>= positional | Package name for scaffolding |

For diagnostic output flags and error format, see [[file:errors.org][Diagnostics]].
For runtime error handling patterns, see [[file:error-handling.org][Error Handling]].

*** Key Points
- =CompilerFlags= is the single struct aggregating all CLI state
- Multiple emit flags can be combined (e.g., =--emit-ir --emit-asm=)
- Default output is an object file when compiling

See also: [[file:errors.org][Diagnostics]]

3. Emit Flags

Emit flags control what output the compiler produces. Multiple flags
can be combined. When no flag is given, =emit_obj= is the default.

| Flag | Ext | Description |
|-------------+-------+-------------------------------|
| =--emit-ir= | =.ll= | LLVM IR (human-readable text) |
| =--emit-bc= | =.bc= | LLVM bitcode (binary) |
| =--emit-asm= | =.s= | Assembly text |
| =--emit-obj= | =.o= | Object file (default) |
| =--emit-json= | =.json= | AST dump in JSON |
| =--emit-typst= | =.typ= | Typst source + compiled PDF |

Other flags:

| Flag | Description |
|----------------------+---------------------------------------|
| =-o <file>= | Output binary path |
| =--test= or =--test-mode= | Compile with embedded test blocks |
| =-i= | Enter REPL after compiling input file |
| =-Wall=, =-Wextra= | Accepted but currently no-ops |

For details on how emit flags control codegen, see
[[file:../context/codegen-ops.org][context/codegen-ops.org]].
For assembly output details, see [[file:asm.org][Assembly]].

*** Key Points
- Six emit formats: IR, bitcode, asm, object, JSON, Typst
- Default emit is =--emit-obj= (object file)
- =-Wall= and =-Wextra= are accepted but have no effect yet

See also: [[file:asm.org][Assembly]]

4. Build Info Resolution

The =cmd_build=, =cmd_run=, =cmd_clean=, =cmd_install=, and =cmd_test= commands
call =resolve_build_info= (cli.c:415), which reads =monad.json= from the
current directory and parses:

- Source file lists (via =yaml_list=)
- Executable definitions (via =yaml_exe_field= and =yaml_first_exe=)
- Dependencies and compiler options

The returned =BuildInfo= struct is consumed by =do_build= (cli.c:505), which
iterates over executables and compiles each source file via =compile_one=
(in [[file:../main.c][main.c]]), then links the resulting objects with the Monad runtime.

For the =monad.json= schema and the build system architecture, see
[[file:../context/buildsystem.org][context/buildsystem.org]].
For the package manifest (=package.yaml=) format, see [[file:package-dev.org][Package Development]].

*** Key Points
- Build commands read =monad.json= from the current directory
- =do_build= compiles each source, links with Monad runtime
- =resolve_build_info= parses YAML fields for source lists and executables

See also: [[file:build.org][Build System]]

5. Project Scaffolding

=CMD_NEW= (via =cmd_new= at cli.c:292) creates a new Monad project:

- Creates directory structure: =src/=, =lib/=
- Writes a =package.yaml= manifest with metadata from =git config=
  (author name via =git config user.name=)
- Writes a default =src/main.mon= entry point
- Uses static helpers: =make_dir=, =write_file=, =git_config=, =current_year=

For the full =package.yaml= reference, see [[file:package-dev.org][Package Development]].
For module resolution in scaffolded projects, see [[file:modules.org][Module System]].

*** Key Points
- =monad new <name>= creates a ready-to-build project skeleton
- Reads author name from =git config= automatically
- Creates =src/=, =lib/=, =package.yaml=, and =src/main.mon=

See also: [[file:package-dev.org][Package Development]]

6. Core Path Resolution

Core module resolution order (for =import= statements):

1. Local =./ModuleName.mon= (project source directory)
2. =$MONAD_CORE= environment variable, if set
3. =/usr/local/lib/monad/core/= (system installation path)

For module import syntax, resolution semantics, and export manifests,
see [[file:modules.org][Module System]].

*** Key Points
- Resolution: local → =$MONAD_CORE= env → system install path
- Allows per-project overrides via the =MONAD_CORE= environment variable
- System path is =/usr/local/lib/monad/core/= by default

See also: [[file:modules.org][Module System]]

7. Usage Examples

Basic compile (produces object file):

#+BEGIN_SRC monad
monad hello.mon
#+END_SRC

View help:

#+BEGIN_SRC monad
monad --help
#+END_SRC

Emit LLVM IR and assembly alongside the object file:

#+BEGIN_SRC monad
monad file.mon --emit-ir --emit-asm
#+END_SRC

Simplify by using Wisp syntax as the file extension; the compiler detects the
syntax from the file extension (.mon vs .wisp):

#+BEGIN_SRC wisp
monad program.wisp --emit-asm
#+END_SRC

Launch the interactive REPL:

#+BEGIN_SRC monad
monad
#+END_SRC

Build a project from =package.yaml=:

#+BEGIN_SRC monad
monad build
#+END_SRC

Scaffold a new project:

#+BEGIN_SRC monad
monad new my-project
#+END_SRC

Compile and run tests, then delete the binary:

#+BEGIN_SRC monad
monad test tests/test-suite.mon
#+END_SRC

For REPL interaction examples, see [[file:repl.org][REPL]].
For build system commands, see [[file:build.org][Build System]].
For property-based fuzzing via CLI, see [[file:fuzzing.org][Fuzzing]].
