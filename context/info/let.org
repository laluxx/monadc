Test coverage: =let=, =let*=, =letrec= are exercised by codegen
fixtures in =rt_control_all.mon= and =rt_let_*.mon=.

=let= forms create local bindings — they introduce new variable scopes
within an expression. All =let= forms are expressions: they evaluate to
the body's return value. Unlike [[file:define.org][=define=]] (which is
module-level), =let= bindings are lexically scoped to their body.

* 1. let

Sequential bindings, evaluated in parallel (no dependency between
bindings):

#+BEGIN_SRC monad
(let [x 1 y 2] (+ x y))            ;; 3
(let [name "world"]
  (++ "hello " name))               ;; "hello world"
#+END_SRC

#+BEGIN_SRC wisp
let [x 1 y 2] + x y
let [name "world"]
  ++ "hello " name
#+END_SRC

Each binding in the vector sees the outer scope, not earlier bindings
in the same =let=. Reader-desugared into nested lambda applications
([[file:../context/reader.org::sugar-registry][context/reader.org — Sugar Registry]]).

#+BEGIN_QUOTE
Key Points:
- Bindings are evaluated in parallel (no cross-reference)
- Desugars to =((lambda [x y] body) 1 2)=
- Returns the body's value
- No strcmp dispatch handler — use Wisp or let desugar only
#+END_QUOTE

* 2. let*

Sequential bindings, evaluated left-to-right:

#+BEGIN_SRC monad
(let* [x 1 y (+ x 1)] (+ x y))     ;; 3 — y uses x
(let* [a 10 b (* a 2) c (+ a b)] c) ;; 30
#+END_SRC

#+BEGIN_SRC wisp
let* [x 1 y + x 1] + x y
let* [a 10 b * a 2 c + a b] c
#+END_SRC

Each binding sees all previous bindings. Desugars to nested =let=
forms (each with a single binding). See
[[file:../context/reader.org::sugar-registry][context/reader.org — Sugar Registry (let desugar)]].

* 3. letrec

Recursive bindings — allows self-reference and mutual recursion within
a local scope:

#+BEGIN_SRC monad
(letrec [fact (lambda [n]
                (if (<= n 1) 1
                  (* n (fact (- n 1)))))]
  (fact 5))                         ;; 120

(letrec [even? (lambda [n] (if (= n 0) True (odd? (- n 1))))
         odd?  (lambda [n] (if (= n 0) False (even? (- n 1))))]
  (even? 4))                        ;; True
#+END_SRC

#+BEGIN_SRC wisp
letrec [fact lambda [n]
         if (<= n 1) 1
           * n (fact (- n 1))]
  fact 5

letrec [even? lambda [n] if (= n 0) True (odd? (- n 1))
        odd?  lambda [n] if (= n 0) False (even? (- n 1))]
  even? 4
#+END_SRC

Reader-desugared — no separate strcmp dispatch handler exists
([[file:../context/language.org::codegen-gaps][context/language.org — Known Gaps]]).
See [[file:recursion.org][Recursion]] for mutual recursion patterns.

** 3.1 letrec with List Recursion

#+BEGIN_SRC monad
(letrec [sum (lambda [xs]
               (match xs
                 [] -> 0
                 [x|rest] -> (+ x (sum rest))))]
  (sum [1 2 3 4 5]))               ;; 15
#+END_SRC

#+BEGIN_SRC wisp
letrec [sum lambda [xs]
         match xs
           [] -> 0
           [x|rest] -> + x (sum rest)]
  sum [1 2 3 4 5]
#+END_SRC

#+BEGIN_QUOTE
Key Points:
- =letrec= enables local recursion and mutual recursion
- All bindings are in scope for each other's bodies
- No strcmp dispatch handler (reader-desugared)
- Essential for local recursive helpers without top-level =define=
#+END_QUOTE

* 4. Typed let Bindings

All =let= forms support type annotations on bindings using the same
bracket syntax as [[file:define.org][=define=]]:

#+BEGIN_SRC monad
(let [[x :: Int] 1 [y :: Int] 2] (+ x y))
(let* [[acc :: Int] 0 [n :: Int] 10] (+ acc n))
#+END_SRC

#+BEGIN_SRC wisp
let [[x :: Int] 1 [y :: Int] 2] + x y
let* [[acc :: Int] 0 [n :: Int] 10] + acc n
#+END_SRC

See [[file:types.org][Types]] for type annotation syntax and
[[file:type-inference.org][Type Inference]] for inference rules.

* 5. Scope Rules

| Form    | Evaluation Order | Can Refer to Earlier? | Can Be Recursive? |
|---------+------------------+-----------------------+-------------------|
| let     | Parallel         | No                    | No                |
| let*    | Sequential       | Yes                   | No                |
| letrec  | Lazy             | Yes                   | Yes               |

All three use bracket vector syntax: =[name value ...]=.

* 6. Desugaring

All =let= forms are reader-level desugaring — they produce =AST_LAMBDA=
and =AST_APP= nodes before codegen:

- =(let [x 1 y 2] body)= → =((lambda [x y] body) 1 2)=
- =(let* [x 1 y (+ x 1)] body)= → =(let [x 1] (let [y (+ x 1)] body))=
- =(letrec [f ...] body)= → inserted directly as =AST_LETREC= node

Known gap: =let=, =let*=, =letrec= have no strcmp dispatch handler in
codegen ([[file:../context/language.org::codegen-gaps][context/language.org — Known Gaps]]).
They are =register_builtins=-only symbols
([[file:../codegen.c::12960][codegen.c:12960]]) — =(let ...)= in
paren-mode is treated as an ordinary variable reference. Wisp syntax
for =let= works because the reader processes it first.

See [[file:../context/reader.org::sugar-registry][context/reader.org — Sugar Registry]]
and [[file:../context/reader.org::reader-lambda-params][context/reader.org — Lambda Param Syntax]]
for reader-level desugaring details.

* 7. with (Sequential Evaluation)

=(with [x 1 y 2] body)= — a sugar for sequential =let= bindings with
codegen dispatch (works in paren-mode):

#+BEGIN_SRC monad
(with [x 1 y (+ x 1)] (+ x y))     ;; 3
#+END_SRC

#+BEGIN_SRC wisp
with [x 1 y + x 1] + x y
#+END_SRC

=with= has a strcmp handler in codegen ([[file:../codegen.c::7841][codegen.c:7841]]),
unlike =let=. See [[file:sequencing.org][Sequencing]] for more on =with=.

** 7.1 with vs let Comparison

| Aspect       | let                              | with                             |
|--------------+----------------------------------+----------------------------------|
| Evaluation   | Parallel                         | Sequential                       |
| Paren-mode   | No strcmp handler                | Has strcmp handler               |
| Desugaring   | Lambda application               | Direct codegen                   |
| Use case     | Independent bindings             | Dependent bindings               |

* 8. Nested let Forms

#+BEGIN_SRC monad
(let [x 1]
  (let [y 2]
    (+ x y)))                       ;; 3
#+END_SRC

#+BEGIN_SRC wisp
let [x 1]
  let [y 2]
    + x y
#+END_SRC

Nesting is the standard way to combine independent binding scopes.
Each inner =let= captures the outer bindings via lexical scoping.

* 9. Implementation Notes

Key source locations for let bindings:

| Aspect                | Location                                                              |
|-----------------------+-----------------------------------------------------------------------|
| Sugar registry        | [[file:../context/reader.org::sugar-registry][context/reader.org — Sugar Registry]]          |
| Lambda param syntax   | [[file:../context/reader.org::reader-lambda-params][context/reader.org — Lambda Param Syntax]] |
| Binding constructs    | [[file:../context/language.org::binding-constructs][context/language.org — Binding Constructs]] |
| Codegen gaps          | [[file:../context/language.org::codegen-gaps][context/language.org — Known Gaps]]           |
| =with= handler        | [[file:../codegen.c::7841][codegen.c:7841]] — with codegen                   |
| Builtins registration | [[file:../codegen.c::12960][codegen.c:12960]] — register_builtins             |

All =let= variants are processed by the reader into core AST nodes
before codegen sees them. The =with= form is the only sequential
binding with a direct codegen handler, making it the practical choice
when paren-mode dispatch is required.

#+BEGIN_QUOTE
Key Points:
- =let=: parallel, non-recursive, no strcmp
- =let*=: sequential, non-recursive, no strcmp
- =letrec=: lazy, recursive, no strcmp
- =with=: sequential, has strcmp, works in paren-mode
- Prefer =with= when you need paren-mode dispatch with sequential bindings
#+END_QUOTE

* 10. Runnable Example

#+BEGIN_SRC monad
;; run with: monadc
(define (sum-to [n :: Int] -> Int)
  (letrec [loop (lambda [acc i]
                  (if (> i n) acc
                    (loop (+ acc i) (+ i 1))))]
    (loop 0 1)))
(show (sum-to 10))  ;; 55
#+END_SRC

* 11. Gotchas

- **No strcmp handler**: =let=, =let*=, =letrec= have no codegen
  dispatch in paren-mode ([[file:../codegen.c::12960][codegen.c:12960]]).
  They are =register_builtins=-only — =(let ...)= in paren-mode is
  treated as a variable reference. Use Wisp syntax or =with= instead.
- **Parallel evaluation**: =let= bindings cannot refer to each other.
  Use =let*= (=let* [x 1 y (+ x 1)]=) when you need sequential deps.
- **No local recursion from =let=/=let*=**: Only =letrec= supports
  recursive bindings. =let= and =let*= fail if a binding refers to
  itself.
- **=with= is the practical choice**: Unlike =let=, =with= has a strcmp
  handler and is callable from paren-mode with sequential semantics.
