Test coverage: purity is inherent in the design — all codegen fixtures
verify that functions return the same value for the same arguments
unless mutation is used.

Monad functions are pure by default: given the same arguments, they
return the same value and produce no side effects. This is enforced by
the type system and the immutability of bindings.

For the design rationale, see [[file:../context/philosophy.org][context/philosophy.org]] and
[[file:functional-programming.org][Functional Programming]].

* 1. Pure by Default

All =define= and =let= bindings are immutable unless explicitly mutated:

#+BEGIN_SRC monad
(define x 42)                    ;; immutable — always 42
(define (square n) (* n n))     ;; pure — same input, same output
#+END_SRC

#+BEGIN_SRC wisp
define x 42                    ;; immutable — always 42
define (square n) (* n n)     ;; pure — same input, same output
#+END_SRC

A function that does not call =set!= or =<-= is guaranteed pure. See
[[file:define.org][Define]] and [[file:let.org][Let Bindings]].

*** Key Points
- Bindings are immutable by default — no =const/=let=var distinction
- Purity follows from immutability: no =set!= → pure function
- The compiler can rely on this for optimisation (e.g., CSE, inlining)

See also: [[file:define.org][Define]], [[file:let.org][Let Bindings]]

* 2. Referential Transparency

An expression is referentially transparent when it can be replaced by
its value without changing program behavior:

#+BEGIN_SRC monad
;; These are equivalent:
(+ (square 3) (square 3))        ;; 9 + 9 = 18
(+ 9 9)                          ;; same result — square is pure
#+END_SRC

#+BEGIN_SRC wisp
;; These are equivalent:
+ (square 3) (square 3)        ;; 9 + 9 = 18
+ 9 9                          ;; same result — square is pure
#+END_SRC

=set!= breaks referential transparency for the mutated binding. See
[[file:mutation.org][Mutation]].

*** Key Points
- Referential transparency enables local reasoning
- Pure expression = safe to inline, reorder, cache, or common-subexpression-eliminate
- Only =set!= and =<-= break transparency — everything else is safe

See also: [[file:functional-programming.org][Functional Programming]]

* 3. Mutation Impurifies

Using =set!= or =<-= inside a function makes it impure:

#+BEGIN_SRC monad
(define counter 0)
(define (increment)
  (set! counter (+ counter 1))
  counter                          ;; impure — different result each call)
#+END_SRC

#+BEGIN_SRC wisp
define counter 0
define increment
  set! counter (+ counter 1)
  counter                        ;; impure — different result each call
#+END_SRC

The compiler tracks which bindings are mutated to avoid unnecessary
boxing. Future versions will track impurity for optimization. See
[[file:../context/language.org][context/language.org]] for purity tracking details.

*** Key Points
- Mutation is opt-in: =set!= and =<-= are the only impure operations
- Impurity is localised: only the containing function is affected
- The compiler tracks mutation to guide boxing decisions

See also: [[file:mutation.org][Mutation]]

* 4. Benefits of Purity

| Benefit          | Description                                    |
|------------------+------------------------------------------------|
| Testability      | Pure functions are trivial to test             |
| Composition      | Output depends only on input                   |
| Caching          | Memoization is safe                            |
| Parallelism      | No shared mutable state                        |
| Reasoning        | Local reasoning — no hidden effects            |

See [[file:testing.org][Testing]] for testing pure functions and
[[file:functional-programming.org][Functional Programming]] for composition patterns.

*** Key Points
- Purity makes every function a mathematical function — input → output
- Testing, caching, and parallelism all become simpler
- Local reasoning: you never need to check what else might have changed state

See also: [[file:testing.org][Testing]], [[file:functional-programming.org][Functional Programming]]

* 5. Immutable Data Structures

Collection operations return modified copies by default:

#+BEGIN_SRC monad
(assoc map key val)              ;; returns new map, original unchanged
(conj set elem)                 ;; returns new set, original unchanged
(cons elem list)               ;; returns new list, original unchanged
#+END_SRC

#+BEGIN_SRC wisp
assoc map key val              ;; returns new map, original unchanged
conj set elem                 ;; returns new set, original unchanged
cons elem list               ;; returns new list, original unchanged
#+END_SRC

Mutable variants (=assoc!=, =conj!=, =append!=) exist for performance
but are opt-in. See [[file:collections.org][Collections]], [[file:map.org][Maps]],
[[file:set.org][Sets]], and [[file:list.org][Lists]].

*** Key Points
- Immutable by default — mutable variants are explicit opt-in (=name!= convention)
- Persistent data structures enable efficient immutable updates
- Mutable variants exist for hot paths where allocation cost matters

See also: [[file:collections.org][Collections]]
