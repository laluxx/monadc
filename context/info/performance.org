This guide covers performance considerations when writing Monad
programs. The compiler's architecture is designed for zero-cost
abstraction: arena allocation replaces GC, monomorphization eliminates
boxing, and layout types provide C-compatible struct layout.

Verified by production use in Main.mon (1244 lines) and how_to/
examples. LLVM IR output can be inspected via =--emit-ir= to verify
codegen quality. The arena allocator ([[file:../arena.c]]) is tested
through the entire compiler pipeline.

For detailed compiler internals see [[file:../context/codegen.org][context/codegen.org]] and
[[file:../context/arena.org][context/arena.org]].

* 1. Arena Allocation

The primary allocator ([[file:../arena.c]], [[file:../arena.h]]) is a bump-pointer
arena with 4 MiB (4,194,304 byte) blocks (=ARENA_DEFAULT_BLOCK= at
arena.h:36). Allocations are O(1): the pointer is bumped by =size=
bytes, rounded up to =ARENA_ALIGN= (8 bytes). When a block is full, a
new block is allocated (doubling in size, growth factor ~2) and
prepended to the singly-linked list.

Free is batch-only: =arena_free= walks the block chain and frees all
OS memory. =arena_reset= rewinds all blocks (used = 0) but keeps
memory mapped — zero mallocs in the common fast path.
=ArenaCheckpoint= saves (block, offset); =arena_restore= rewinds to
that point and frees any blocks allocated after it. This is a
stack-based rollback (LIFO only).

** 1.1 Tradeoffs

- *Good for*: batch-freed workloads like compiler passes. AST nodes,
  symbol tables, type objects — allocated in phases, freed all at
  once. Bump allocation converts malloc overhead (lock, freelist
  search, fragmentation) into a single pointer increment.
- *Bad for*: long-lived small allocations that are not batch-freed. If a
  4 MiB block has 3.9 MiB unused, that memory cannot be reclaimed by
  other processes until the compiler exits. For the REPL or language
  server, checkpoint/restore provides the necessary control.
- The compiler uses two arenas: permanent (AST, symbol table — lifetime
  = whole compilation) and scratch (type variables, constraint sets —
  checkpointed between phases).

See [[file:../context/arena.org][context/arena.org]] for full details and
[[file:arena.org][Arena Allocation]] for the user-facing documentation.

*** Key Points
- O(1) bump allocation — replaces malloc for the hot path
- Batch free via =arena_reset= — no per-object tracking
- Two-arena pattern avoids waste across compilation phases
- Checkpoint/restore controls lifetime for REPL and incremental use

See also: [[file:arena.org][Arena Allocation]]

* 2. Monomorphization

Polymorphic functions (=id x = x=, =const x y = x=) are
monomorphized: each concrete instantiation generates separate
optimized machine code. For example, =id 3= and =id "hello"= produce
two distinct LLVM functions with =Int= and =String= type-specialized
bodies.

- *Payoff*: zero-cost polymorphism — no boxing, no vtable dispatch, no
  runtime type tags. Each instantiation gets LLVM's full optimizer.
- *Cost*: code size blowup. Each unique type combination at a call
  site creates a new function body. The =MonoCache= in
  =CodegenContext= (context/codegen.org:141) deduplicates: repeated
  =(id 3)= calls reuse the same =id_Int= specialization.

See [[file:../context/language.org::hm-polymorphism][context/language.org — HM polymorphism]] and
[[file:type-inference.org][Type Inference]].

*** Key Points
- Zero-cost polymorphism — no boxing or vtable dispatch
- Each unique type instantiation generates separate machine code
- =MonoCache= deduplicates repeated instantiations of the same type
- Code size grows linearly with distinct type combinations used

See also: [[file:type-inference.org][Type Inference]]

* 3. Three Array Forms

Arrays share one type tag (=TYPE_ARR=) but have three representations
with different performance characteristics (types.c:493-507):

| Form                      | Constructor         | LLVM Type            | Use Case                               |
|---------------------------+---------------------+----------------------+----------------------------------------|
| Stack (fixed-size)        | =type_arr(elem, sz)=| =[N x T]=            | Compile-time-known size, stack storage |
| Fat pointer               | =type_arr_fat(elem)=| ={i8*, i64}=         | Runtime-sized, heap-backed             |
| Heap-growable             | =type_arr_heap(elem)=| ={i8*, i64, i64}=    | Mutable, push/pop semantics            |

Stack arrays are proper LLVM =[N x T]= arrays — contiguous, cache-friendly,
zero-overhead. Fat pointer and heap-growable both collapse to
=arr.fat*= at the LLVM level (=arr_make_fat= at codegen.c:3763-3789).
In practice, even stack arrays are heap-allocated immediately because
LLVM stack allocation decisions are function-level and returned arrays
must be heap-backed.

Surface syntax: =[1 2 3]= (stack), =~[1 2 3]= (heap), =[16kb I32]=
(size-annotated). See [[file:types.org][Types]], [[file:arrays.org][Arrays]],
[[file:heap.org][Heap Allocation]], and [[file:../context/types.org][context/types.org]].

*** Key Points
- Three representations share one type tag; selection depends on use case
- Stack arrays: zero-overhead LLVM =[N x T]= when size is known at compile time
- Fat pointer and heap-growable unify at the LLVM level (=arr.fat*=)
- Returned arrays must be heap-backed regardless of surface syntax

See also: [[file:arrays.org][Arrays]]

* 4. Layout Types for Cache-Friendly Data

Layout types ([[file:layouts.org]]) are C11-compatible struct types with
no hidden padding (unless =:packed= is specified), no pointer chasing,
and LLVM-native representation:

- Fields are laid out at known byte offsets (=LayoutField.offset=).
- No boxing or heap indirection — a layout value is a contiguous byte
  block.
- Access through dot notation (=(p.x)=) compiles to a fixed-offset
  load from an LLVM struct pointer.

This makes layout types ideal for:
- C interop (identical ABI)
- Cache-friendly hot paths (sequential memory access)
- SIMD-friendly data (aligned, contiguous)

See [[file:layouts.org]], [[file:../how_to/Layouts.mon]], and
[[file:ffi.org][C FFI]].

*** Key Points
- C-compatible ABI with no hidden padding (unless =:packed=)
- Contiguous byte blocks — no pointer chasing, no boxing
- Dot notation compiles to fixed-offset LLVM loads
- Ideal for FFI, hot paths, and SIMD data

See also: [[file:layouts.org][Layout Types]], [[file:ffi.org][C FFI]]

* 5. Inline Assembly for Hot Paths

Inline assembly ([[file:asm.org]], [[file:../context/asm.org]]) provides
zero-overhead access to the CPU when the compiler's codegen is not
optimal. Naked functions (=:naked True=) emit no prologue/epilogue:

#+BEGIN_SRC monad
(define (add-fast [x :: Int] -> [y :: Int] -> Int)
  :naked True
  (asm
    mov x %rax
    add y %rax
    ret))
#+END_SRC

#+BEGIN_SRC wisp
define add-fast [x :: Int] -> [y :: Int] -> Int
  :naked True
  asm
    mov x %rax
    add y %rax
    ret
#+END_SRC

The register allocator maps parameters to SysV AMD64 ABI registers
(%rdi-%r9 for ints, %xmm0-%xmm7 for floats). Non-naked asm uses LLVM
inline asm constraints (=r= for ints, =x= for floats). Size suffixes
(=q=, =l=) are inferred from operand widths.

60 asm test fixtures in =tests/codegen/asm= cover syscall-shaped
operands, multi-instruction splitting, shifts, SSE, conditional moves,
and more. See [[file:../context/asm.org]].

*** Key Points
- Naked functions: no prologue/epilogue — full control over register state
- Parameter mapping follows SysV AMD64 ABI
- Operand size suffixes inferred from type widths
- 60+ test fixtures verify correctness across instruction families

* 6. Type Annotations Help Codegen

Explicit type annotations improve codegen quality:

#+BEGIN_SRC monad
;; Without annotation — inference must work harder
(define x (+ 1 2))

;; With annotation — codegen knows immediately
(define [x :: Int] (+ 1 2))
#+END_SRC

#+BEGIN_SRC wisp
;; Without annotation — inference must work harder
define x
  + 1 2

;; With annotation — codegen knows immediately
define [x :: Int]
  + 1 2
#+END_SRC

This is especially important for polymorphic code, where the
annotated instantiation triggers monomorphization earlier and avoids
speculative type variable overhead. See [[file:types.org][Types]] and
[[file:type-inference.org][Type Inference]].

*** Key Points
- Annotations remove speculation from type inference
- Earlier monomorphization = better optimization
- Most impactful on polymorphic code and hot paths

See also: [[file:types.org][Types]]

* 7. Avoid Unnecessary Consing

Each =cons= call allocates (lists are linked cells). For tight loops,
prefer mutation (=set!=, =<-=) over allocation, and tail recursion
over non-tail recursion.

#+BEGIN_SRC monad
;; Allocation-heavy
(define (range n)
  (if (= n 0) nil
    (cons n (range (- n 1)))))
#+END_SRC

#+BEGIN_SRC wisp
;; Allocation-heavy
define range n
  if (= n 0) nil
    cons n (range (- n 1))
#+END_SRC

See [[file:mutation.org][Mutation]] and [[file:recursion.org][Recursion]] for alternatives.

*** Key Points
- Each =cons= allocates a new cell on the heap
- For tight loops: prefer mutation or tail recursion
- Profile before optimizing — most code does not need micro-optimisation

See also: [[file:mutation.org][Mutation]]

* 8. What the Compiler Does NOT Optimize (Yet)

- Tail-call elimination (not yet implemented)
- Closure allocation (closures not yet optimized)
- Inlining (not yet implemented)
- List fusion
- Automatic vectorization

See [[file:../context/language.org::codegen-gaps][context/language.org — Known codegen gaps]] for
the current limitations and planned work.

*** Key Points
- Tail calls, inlining, and closure optimisation are not yet implemented
- Manual transformation may be needed for tight loops and HOF patterns
- Check known gaps before assuming the compiler will optimise

* 9. Profiling Tips

1. **Start simple** — write clear code first, optimize later
2. **Use =show= to measure** — print timestamps around operations
3. **Inspect LLVM IR** — use =--emit-ir= to see what codegen produces
4. **Check assembly** — use =--emit-asm= for the final output
5. **Profile at the system level** — use =perf=, =valgrind=, etc.

See [[file:show.org][Show and Display]] and [[file:cli.org][CLI Reference]].

*** Key Points
- Prefer readability first, then measure, then optimize
- LLVM IR inspection (=--emit-ir=) reveals codegen quality directly
- System profilers (=perf=) give ground truth for hot spots

See also: [[file:show.org][Show and Display]], [[file:cli.org][CLI Reference]]
