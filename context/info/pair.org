A =Pair= is a product type holding two values of possibly different
types. It is the simplest compound data structure, analogous to tuples
in other languages. Pairs form the foundation of lists (cons cells)
and map entries (key-value pairs).

See [[file:data.org][Algebraic Data Types]] for the ADT system and
[[file:types.org][Types]] for the type system.

* 1. Type

=Pair= is defined as a regular algebraic data type:

#+BEGIN_SRC monad
(data Pair a b
  (Pair a b))
#+END_SRC

#+BEGIN_SRC wisp
data Pair a b
  Pair a b
#+END_SRC

The constructor shares the type name. The type parameters =a= and =b=
allow pairs of any two types: =(Pair Int String)=, =(Pair Float (List
Bool))=, etc.

See [[file:../context/types.org][context/types.org]] for TYPE_APP representation
and [[file:../context/codegen.org][context/codegen.org]] for Pair lowering.

Key Points:
- =Pair= is a regular ADT with two type parameters
- Constructor name matches the type name
- Fully polymorphic: =(Pair a b)= for any types =a=, =b=

* 2. Construction

#+BEGIN_SRC monad
(Pair 1 "one")                  ;; Pair Int String
(Pair "hello" 42)              ;; Pair String Int
(Pair x y)                     ;; from variables
#+END_SRC

#+BEGIN_SRC wisp
Pair 1 "one"
Pair "hello" 42
Pair x y
#+END_SRC

Pairs are created with the =Pair= constructor in both S-expression and
Wisp syntax. There is no special literal syntax — pairs use the standard
ADT constructor call.

See [[file:wisp.org][Wisp]] for Wisp syntax reference and
[[file:data.org][Data Types]] for constructor semantics.

Key Points:
- Construction uses standard ADT constructor syntax: =(Pair val1 val2)=
- No special literal syntax (unlike lists ='()= or maps =#{}=)
- Type is inferred from the values

* 3. fst / snd

#+BEGIN_SRC monad
(define p (Pair 1 "one"))
(fst p)                        ;; 1
(snd p)                        ;; "one"
#+END_SRC

#+BEGIN_SRC wisp
define p (Pair 1 "one")
fst p
snd p
#+END_SRC

Pattern matching on =Pair=:

#+BEGIN_SRC monad
(define (swap [p :: (Pair a b)] -> (Pair b a))
  (match p with
    (Pair x y) -> (Pair y x)))
#+END_SRC

#+BEGIN_SRC wisp
define swap [p :: (Pair a b)] -> (Pair b a)
  match p with
    (Pair x y) -> Pair y x
#+END_SRC

The =fst= and =snd= functions are builtins that extract the first and
second fields respectively. Pattern matching is the idiomatic
alternative for destructuring.

See [[file:patterns.org][Pattern Matching]] for Pair destructuring patterns
and [[file:../context/language.org][context/language.org]] for fst/snd dispatch.

Key Points:
- =fst= extracts the first element; =snd= extracts the second
- Pattern matching =(Pair x y)= is the idiomatic destructuring form
- =swap= reorders the two fields

* 4. Nesting

Pairs can be nested for multi-field structures:

#+BEGIN_SRC monad
(Pair 1 (Pair 2 3))            ;; Pair Int (Pair Int Int)
(Pair (Pair "a" 1) (Pair "b" 2))
#+END_SRC

#+BEGIN_SRC wisp
Pair 1 (Pair 2 3)
Pair (Pair "a" 1) (Pair "b" 2)
#+END_SRC

For more than two fields, consider =data= types
([[file:data.org][Data Types]]) or =layout= types
([[file:layouts.org][Layout Types]]). Nested pairs become unwieldy
beyond two levels.

See [[file:layouts.org][Layout Types]] for named-field alternatives and
[[file:ffi.org][FFI]] for C-interop struct layouts.

Key Points:
- Pairs nest naturally for multi-value structures
- Beyond 2-3 levels, prefer =data= or =layout= types
- Nested pairs still O(1) access via chained =fst=/=snd=

* 5. Pair vs Layout vs Data

| Aspect    | Pair                       | data                    | layout          |
|-----------+----------------------------+-------------------------+-----------------|
| Fields    | 2, named by position        | 0..N named by position  | Named           |
| Tag       | None (no sum type)          | Automatic i32 tag       | None (untagged) |
| C compat  | No                          | No                      | Yes             |
| Pattern   | =(Pair x y)=                | =(Ctor x y)=            | =(=struct= f v)= |

See [[file:data.org][Data Types]] for ADTs,
[[file:layouts.org][Layout Types]] for C-interop types, and
[[file:../context/types.org][context/types.org]] for type representation.

Key Points:
- Pair: 2 positional fields, no tag, no C compat
- data: 0..N positional fields, automatic tag
- layout: named fields, no tag, C-compatible

* 6. Examples

** 6.1 Zip two lists into pairs

#+BEGIN_SRC monad
(define (zip [xs :: [a]] [ys :: [b]] -> [(Pair a b)])
  (match (Pair xs ys) with
    ((Pair [] _))          -> []
    ((Pair _ []))          -> []
    ((Pair [x|xs] [y|ys])) -> (cons (Pair x y) (zip xs ys))))
#+END_SRC

#+BEGIN_SRC wisp
define zip [xs :: [a]] [ys :: [b]] -> [(Pair a b)]
  match (Pair xs ys) with
    (Pair [] _)          -> []
    (Pair _ [])          -> []
    (Pair [x|xs] [y|ys]) -> cons (Pair x y) (zip xs ys)
#+END_SRC

** 6.2 Unzip pairs into two lists

#+BEGIN_SRC monad
(define (unzip [ps :: [(Pair a b)]] -> (Pair [a] [b]))
  (reduce (lambda [(Pair acc-a acc-b) (Pair a b)]
            (Pair (cons a acc-a) (cons b acc-b)))
          (Pair [] [])
          ps))
#+END_SRC

#+BEGIN_SRC wisp
define unzip [ps :: [(Pair a b)]] -> (Pair [a] [b])
  reduce (lambda [(Pair acc-a acc-b) (Pair a b)]
            Pair (cons a acc-a) (cons b acc-b))
          (Pair [] [])
          ps
#+END_SRC

See [[file:list.org][Lists]] for list operations and
[[file:recursion.org][Recursion]] for recursive patterns.

Key Points:
- =zip= combines two lists into a list of pairs, truncating to the shorter input
- =unzip= separates a list of pairs into two lists using =reduce=
- Both are standard functional programming idioms

* 7. Runnable Example

#+BEGIN_SRC monad
;; run with: monadc pair-demo.mon

;; Construction
(define p (Pair "hello" 42))
(show p)                          ;; prints (Pair "hello" 42)

;; Accessors
(show (fst p))                    ;; prints "hello"
(show (snd p))                    ;; prints 42

;; Pattern matching
(define (swap [p :: (Pair a b)] -> (Pair b a))
  (match p with
    (Pair x y) -> (Pair y x)))

(show (swap p))                   ;; prints (Pair 42 "hello")

;; Nested pairs
(define nested (Pair 1 (Pair 2 3)))
(show (fst nested))               ;; prints 1
(show (fst (snd nested)))         ;; prints 2

;; Zip two lists into pairs
(define (zip [xs :: [a]] [ys :: [b]] -> [(Pair a b)])
  (match (Pair xs ys) with
    ((Pair [] _))          -> []
    ((Pair _ []))          -> []
    ((Pair [x|xs] [y|ys])) -> (cons (Pair x y) (zip xs ys))))

(show (zip '(1 2 3) '("a" "b" "c")))  ;; prints ((Pair 1 "a") (Pair 2 "b") (Pair 3 "c"))
#+END_SRC

* 8. Gotchas

Common mistakes and surprising behaviours with pairs:

- **No special literal syntax** — use =(Pair a b)= constructor. Unlike lists (='()=) and maps (=#{}=), there is no dedicated pair literal.

- **Pattern syntax is =(Pair x y)=, not =[x y]=** — the constructor must match the ADT pattern syntax, not list/vector destructuring.

- **=fst=/=snd= on non-pair crashes** — these are untyped builtins; applying them to a non-pair value causes a runtime crash rather than a type error.

- **Nested pairs beyond 2-3 levels are unwieldy** — for structured data with more fields, use =data= or =layout= types instead.

- **Pairs are heap-allocated** — like all ADT constructors, =(Pair a b)= allocates on the heap. There is no stack-allocated tuple.

- **Lists ARE pairs** — a cons cell =(Pair x xs)= is a list cell. Operations that work on pairs also work on list cells (and vice versa), which can cause confusion when =fst= returns the head and =snd= returns the tail.
