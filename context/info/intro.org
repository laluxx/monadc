The Monad language: a compiled Lisp-family functional language with dual syntax, Hindley-Milner type inference, dependent types, arena allocation, and C FFI. This page covers the language's design, features, quick start, compiler pipeline, and tooling.

Test coverage: introductory examples are tested as part of =rt_tutorial_*.mon= fixtures in [[file:../tests/codegen/branches/][tests/codegen/branches/]]. The reader conformance corpus is at [[file:../tests/reader-atoms.tsv][tests/reader-atoms.tsv]]. See [[file:../context/tests-arch.org][context/tests-arch.org]] for the full test architecture.

* 1. What is Monad?

Monad is a compiled Lisp-family functional language with a /dual-surface
syntax/: traditional S-expressions (parenthesized prefix notation) and
indentation-based Wisp (modeled after Python-style block structure). Both
syntaxes produce identical ASTs and mix freely in the same file.

The compiler targets x86-64 Linux via LLVM 19+ and is written in C
(~29,000+ lines across 30+ source files, see
[[file:../context/build.org][context/build.org]]). The compiler produces native binaries
linked against a static runtime library (=libmonad.a= compiled from
[[file:../runtime.c][runtime.c]]).

The language draws inspiration from three traditions:

- *From Lisp* ([[file:from-lisp.org][from-lisp.org]]): code-as-data, macros as functions,
  S-expression syntax, interactive REPL-driven development
- *From Haskell* ([[file:from-haskell.org][from-haskell.org]]): Hindley-Milner type inference,
  purity by default, type classes, algebraic data types
- *From C* ([[file:from-c.org][from-c.org]]): explicit memory management (arenas), inline
  assembly, C FFI via =#include=, predictable performance

For a deeper look at design motivations, see
[[file:../context/philosophy.org][context/philosophy.org]].

** 1.2 Key Design Points

- *Type system*: Hindley-Milner inference (constraint-based, with
  generalization, instantiation, and monomorphization) for ground types,
  plus a full Intuitionistic Type Theory (ITT) dependent type checker
  with Normalization by Evaluation (NbE), bidirectional checking, and
  a =TERM_EMBED= bridge between the two systems.

- *Memory*: Arena allocation with bump-pointer semantics, 4 MiB blocks,
  and checkpoint/restore. No general-purpose garbage collector — the
  arena is reset at predictable boundaries.

- *FFI*: C interop via =(include "header.h")= powered by libclang header
  parsing. No binding generators, IDL files, or wrapper code needed.

- *Macros*: Ordinary functions that return =Syntax= values and operate
  at compile time. No separate =macro= keyword. Full hygiene with
  rename-pair tracking, fixpoint expansion, and variadic splice.

- *Dual syntax*: S-expressions (paren-mode) and Wisp (indentation-based)
  are freely mixable within a single file and produce identical ASTs.

- *Self-hosting-ready*: =Main.mon= (1244 lines, verified at
  [[file:../Main.mon][Main.mon]]) serves as the standard library and primary compiler test
  surface. Core library modules exist under =core/= covering lists, sets,
  math, text processing, and more (see [[file:../context/core-libs.org][context/core-libs.org]]).

- *Test suite*: ~1305 test assets across codegen runtime fixtures
  (423 + 60 ASM), TSV-driven parser corpora (1243 rows), property-based
  fuzzing (41 algebraic laws) — see [[file:../context/tests-arch.org][context/tests-arch.org]] and
  [[file:testing.org][Testing]].

* 2. Getting Started

The fastest way to get Monad running on your machine:

1. *Build the compiler* — see [[file:build.org][build.org]] for dependencies and
   build instructions (currently Linux x86-64 only, LLVM 19+ required).
2. *Command-line reference* — see [[file:cli.org][cli.org]] (or [[file:../context/cli.org][context/cli.org]])
   for all =monad= subcommands and flags.
3. *Follow the tutorial* — [[file:tutorial.org][tutorial.org]] walks you through your
   first programs step by step.
4. *Explore the REPL* — =monad -i= launches a JIT-compiled interactive
   session (see [[file:repl.org][repl.org]]).

** 2.1 Quick Start: Hello World

The canonical "hello world" in both syntaxes:

#+BEGIN_SRC monad
(module Main)
(show "hello world")
#+END_SRC

#+BEGIN_SRC wisp
module Main
show "hello world"
#+END_SRC

Compile and run:

#+BEGIN_SRC sh
make && ./monad hello.mon -o hello && ./hello
#+END_SRC

Notice how both syntaxes express the same program. The Wisp version drops
parentheses and uses indentation to show nesting. Both produce the same
AST and the same compiled binary.

** 2.2 Key Concepts Introduced

- =(module Main)= declares the entry module — top-level forms execute in order
- =(show)= prints any value to stdout — see [[file:show.org][Show]]
- Both S-expression and Wisp syntax are valid, freely mixable — see [[file:syntax.org][Syntax]]
- Compilation produces native binaries via LLVM — see [[file:../context/codegen.org][context/codegen.org]]
- The REPL ([[file:repl.org]]) provides interactive experimentation without compiling

* 3. Quick Tour

A brief walk through Monad's surface in both syntaxes. Each example shows
the S-expression and Wisp form side by side.

** 3.1 Bindings and Functions

Bind a value with =define=, then call a function:

#+BEGIN_SRC monad
(define x 42)
(define (greet name)
  (++ "hello, " name))
(show (greet "world"))
#+END_SRC

#+BEGIN_SRC wisp
define x 42
define greet name
  ++ "hello, " name
show greet "world"
#+END_SRC

The =define= form is the primary binding mechanism. In Wisp, the function
body is indented under the definition. See [[file:define.org][define.org]] for all binding forms.

** 3.2 Arithmetic

Prefix notation for all operators ([[file:arithmetic.org]]), including recursion:

#+BEGIN_SRC monad
(define (fact n)
  (if (<= n 1)
    1
    (* n (fact (- n 1)))))
(show (fact 10))  ; 3628800
#+END_SRC

Wisp uses the same prefix operators without parentheses:

#+BEGIN_SRC wisp
define fact n
  if <= n 1
    1
    * n (fact (- n 1))
show fact 10
#+END_SRC

See [[file:arithmetic.org][arithmetic.org]], [[file:numeric.org][Numeric Types]].

** 3.3 Lists and Collections

Lists are linked cons cells using quoted syntax:

#+BEGIN_SRC monad
(define xs '(1 2 3 4 5))
(show (map (lambda (x) (* x x)) xs))
#+END_SRC

#+BEGIN_SRC wisp
define xs '(1 2 3 4 5)
show map (lambda (x) (* x x)) xs
#+END_SRC

The =quote= (='=) prevents evaluation, creating a literal list. See
[[file:list.org][list.org]], [[file:collections.org][collections.org]].

** 3.4 Pattern Matching

Destructure values with =match=:

#+BEGIN_SRC monad
(match '(1 2 3)
  ((list a b c) (show (+ a b c))))
#+END_SRC

#+BEGIN_SRC wisp
match '(1 2 3)
  list a b c -> show + a b c
#+END_SRC

Pattern matching supports destructuring, guards, and nested patterns.
See [[file:patterns.org][patterns.org]].

** 3.5 Dependent Types (Advanced)

Monad supports dependent types for compile-time invariant checking.
A length-indexed vector:

#+BEGIN_SRC monad
;; A vector of length n
(data (Vec :type -> Nat -> type)
  (Nil  (Vec A 0))
  (Cons A (Vec A n) (Vec A (Succ n))))
#+END_SRC

This declares =Vec A n= as a vector of element type =A= and length =n=.
The dependent checker ([[file:../context/dep.org]]) verifies length arithmetic at
compile time. See [[file:data.org][data.org]], [[file:dependent-types.org][dependent-types.org]], [[file:types.org][types.org]].

** 3.6 Quick Tour Summary

| Concept             | S-expr Form         | Wisp Form                |
|---------------------+---------------------+--------------------------|
| Value binding       | =(define x 42)=       | =define x 42=              |
| Function definition | =(define (f x) body)= | =define f x\n  body=       |
| Arithmetic          | =(+ 1 2 3)=           | =+ 1 2 3=                  |
| List literal        | ='(1 2 3)=            | ='(1 2 3)=                 |
| Lambda              | =(lambda (x) body)=   | =lambda x\n  body=         |
| Pattern match       | =(match x (p e) ...)= | =match x\n  p -> e\n  ...= |
| Conditional         | =(if cond t f)=       | =if cond t f=              |
| Data type           | =(data T (C ...))=    | =data T\n  C\n  ...=       |

** Complete example:

#+BEGIN_SRC monad
;; Save as hello.mon && monadc hello.mon && ./hello
(module Main)

(define (greet [name :: String] -> String)
  (++ "Hello, " name "!"))

(define (fact [n :: Int] -> Int)
  (if (<= n 1)
    1
    (* n (fact (- n 1)))))

(show (greet "Monad"))
(show "10! = " (fact 10))    ;; 3628800
#+END_SRC

* 4. Compiler Pipeline

The MonadC compiler transforms source to native code through six stages:

1. *Reader* ([[file:../reader.c][reader.c]]) — Tokenizes and parses source text into an AST.
   Recognizes both S-expression syntax and Wisp indentation, producing
   identical AST nodes regardless of surface syntax. The reader is
   tested against a conformance corpus (=tests/reader-atoms.tsv=,
   [[file:../context/language.org::syntax-corpus][context/language.org — Syntax Corpus]])
   covering every grammar production: numeric literals, strings, chars,
   paths, keywords, symbols, arrays (stack/fat/heap), sets, maps,
   ranges, lambdas, data declarations, type annotations, refinements,
   layout types, and inline assembly forms.

2. *Macro expansion* ([[file:../context/macro.org]]) — Macro functions (ordinary
   =define= returning =Syntax=) are expanded at compile time. Expansion is
   fixpoint — the compiler re-expands until the term stabilizes
   ([[file:../context/macro.org::macro-fixpoint][context/macro.org — Fixpoint]]). Hygiene is
   maintained through rename-pair tracking that prevents unintended
   variable capture ([[file:../context/macro.org::macro-hygiene][context/macro.org — Hygiene]]).

3. *Wisp expander* ([[file:../wisp.c][wisp.c]]) — If the source uses Wisp syntax, the
   expander desugars indentation into S-expressions
   ([[file:../context/wisp.org::wisp-architecture][context/wisp.org — Architecture]]).
   Wisp !-suffix and ?-suffix sugar are applied at this stage.

4. *Type inference / checking* — The dual type system runs:
   - *HM inference* (=infer.c=, [[file:../context/infer.org]]) handles ground types with
     constraint generation, unification, generalization, and
     monomorphization. Fast path for non-dependent code.
   - *ITT dependent checker* (=dep.c=, [[file:../context/dep.org]]) handles Pi-types,
     Sigma-types, universe levels, equality/refl/subst, Nat and
     Nat-elim, metavariables, and bidirectional infer/check entry
     points ([[file:../context/dep.org::dep-nbe-kernel][context/dep.org — NbE Kernel]]) — all normalized via NbE.

5. *Codegen* ([[file:../codegen.c][codegen.c]]) — Lowers typed AST to LLVM IR. Dispatch
   uses a three-layer architecture: AST type switch → strcmp dispatch
   for 83+ known symbols → =register_builtins= for 94 env entries
   ([[file:../context/codegen-ops.org::dispatch-architecture][context/codegen-ops.org — Dispatch Architecture]]).
   Polymorphic functions are monomorphized — each concrete type
   instantiation generates separate machine code.

6. *Linking* — The compiler invokes LLVM's optimization pipeline, emits
   object code, and links against =libmonad.a= (the runtime) to produce
   a native executable.

See [[file:../context/codegen.org][context/codegen.org]] and [[file:../context/infer.org][context/infer.org]] for deeper
compiler internals documentation.

* 5. Key Ideas

Monad's design rests on several foundational concepts:

- *Types are sets* — =Int= is the set of 63-bit integers. A refinement
  type like ={ x ∈ Int | x > 0 }= is a subset. Functions are relations.
  The compiler materializes infinite sets as values you can inspect.

- *Immutable by default* — Mutation is explicit: =set!= for variables,
  =conj!/disj!== for collections, =<-= for array elements. The language
  encourages pure functional style but provides controlled mutation.

- *Code is data* — Macros are just functions that return =Syntax=
  values. The quote operator (= '(...)=) and quasi-quote (= `(...)=)
  construct syntax objects at runtime, and macro expansion transforms
  them at compile time ([[file:../context/macro.org::macro-syntax-convention][context/macro.org — Syntax Convention]]).

- *No trade-off between safety and control* — The same language gives
  you dependent types AND inline assembly, refinement types AND naked
  functions, lazy infinite lists AND C-compatible arrays — all without
  binding generators, FFI wrappers, or runtime overhead.

- *Continuous abstraction ladder* — Start with =show "Hello!"=, add
  type annotations, introduce polymorphic functions, prove invariants
  with dependent types, drop to layout-based structs for performance,
  and write inline assembly for hardware access — all in the same file
  and the same function.

** 5.1 Key Points

- Monad combines Lisp code-as-data, Haskell-style types, and C-level control
- Two syntaxes (S-expr + Wisp) produce identical ASTs — freely mixable
- Memory uses arena allocation: no GC, predictable performance
- Type system: HM inference + dependent types (ITT/NbE) with a =TERM_EMBED= bridge
- Macros are ordinary functions, not a separate language
- All features (dependent types, inline ASM, FFI) work together in one file

* 6. Philosophy

Monad is built on five core principles:

1. *No trade-off between safety and control* — You should not have to
   choose between dependent types and inline assembly, between
   mathematical guarantees and hardware control.

2. *Purity is practical* — Mathematical purity enables compile-time
   predicate checking, exhaustive pattern coverage verification, and
   zero-cost type abstraction through monomorphization.

3. *Syntax is instrument* — Two syntaxes (S-expressions and Wisp) for
   two tasks: programmatic generation and macro processing with
   S-expressions, readability in large codebases with Wisp. Both work
   for every construct and mix freely.

4. *Zero-to-metal abstraction ladder* — The ladder from =show "Hello!"=
   to inline assembly must be continuous, with no language boundary,
   binding layer, or runtime switch.

5. *Sets are the model* — Types are sets. Functions are relations.
   Refinement types are subsets. Universes are sets of sets. This is
   not metaphor — it is the compiler's semantic model.

For a full exposition of the design philosophy, see
[[file:../context/philosophy.org][context/philosophy.org]].

** 6.1 Key Points

- Purity enables compile-time verification and zero-cost abstraction
- Dual syntax serves different use cases without compromising expressiveness
- The abstraction ladder from simple to systems-level is continuous
- Set-theoretic semantics unify the type system from ground types to universes

* 8. Project Status

MonadC is under active development. The compiler core is functional:
parsing, type inference, and codegen produce working binaries for a
large subset of the language. Known limitations include:

- Closures (=fn=, =lambda=, =defn=) lack a codegen strcmp handler —
  higher-order functions like =map= and =filter= are blocked.
- Several control forms (=cond=, =when=, =let=, =let*=, =letrec=) are
  registered builtins but lack strcmp dispatch.
- Map and set runtime construction is broken for untyped contexts
  (needs explicit =:: Map= / =:: Set= annotation).
- Core prelude modules are not yet automatically imported.

See [[file:../context/language.org][context/language.org]] for the full codegen gap analysis.

* Common Pitfalls

- **Map/set literals need type annotation**: `#{:a 1}` or `{1 2 3}` without a
  `:: Map` or `:: Set` annotation causes broken runtime construction. Always
  annotate: `(define [m :: Map] #{:a 1})`.
- **cond, when, let, let*, letrec have no codegen dispatch**: These are
  registered as builtins but lack strcmp handlers. They work as special forms
  but NOT via paren-mode function call.
- **Closures/fn/lambda/defn lack strcmp handler**: Higher-order functions
  like =map= and =filter= are blocked until codegen dispatch is added.
- **Module prelude not auto-imported**: Core prelude modules must be
  imported explicitly — they are not loaded by default.
- **Feature conditionals are parsed but inert**: `#+linux` / `#+darwin`
  blocks parse correctly but have no codegen effect yet.
- **REPL + inline assembly edge cases**: The REPL JIT pipeline may not
  handle inline assembly or certain FFI calls correctly.

See [[file:../context/language.org][context/language.org]] for the full codegen gap analysis.
