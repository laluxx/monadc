Test coverage: the =rt_codegen_err_*.mon= diagnostic fixture pattern
(43+ fixtures) covers expected compiler errors for every stable error
path. The =??= null-coalescing operator is verified by =rt_null_coalesce.mon=.
See [[file:../tests/codegen/rt_codegen_err_*.mon][tests/codegen/rt_codegen_err_*.mon]].

This page covers runtime error handling in Monad: the =Error= type,
=error= builtin, =undefined=, the =??= operator, Option/Result patterns,
and =assert-eq=.

Monad provides several mechanisms for handling and signalling errors, from
compiler diagnostics to runtime error values.

For compiler error message format and diagnostic patterns, see [[file:errors.org][Diagnostics]].

1. Compiler Error Diagnostics

Compiler diagnostics are treated as semantic relations between invalid source
forms and explicit compiler errors
([[file:../context/language.org::codegen-diagnostics][context/language.org]]).
Each stable codegen error path should have a focused expected-failure fixture.

1.1 Diagnostic fixture pattern

The =tests/codegen/rt_codegen_err_*.mon= files are the diagnostic fixture
pattern. Each fixture covers one compiler error per file:

- Unbound variables
- Unknown functions
- Quote/quasiquote arity errors
- Doc/code reflection misuse
- Sized-array mismatches
- Layout declaration/access errors
- Address-of errors
- Mutation lvalue/arity errors
- Cast arity
- Refinement naming and base-type errors
- Empty list forms
- Non-callable number heads

Extended diagnostic coverage also includes codegen guard branches for:
- Runtime collection operations (=head=, =tail=, =pair?=, =++=, =count=)
- Control forms (=if=, =unless=, =while=, =for=)
- Arithmetic/bitwise arity checking
- Array-as-function arity
- Scalar variables in call position
- Top-level =undefined=
- Empty-array inference
- Quoted-list literal bounds checks
- Malformed non-array =for= bindings

[[file:../tests/codegen/rt_codegen_err_*.mon][tests/codegen/rt_codegen_err_*.mon]]

For the full list of error patterns by compiler layer, see [[file:errors.org][Diagnostics # Common Error Patterns]].

1.2 Parser error prefix

The parser uses =Error:= prefix for user-facing messages and =[internal]= for
internal compiler errors. See [[file:errors.org][Diagnostics]] for the full
list of error codes.

2. Runtime Error Handling

2.1 Error (type)

=(data Error String)= — a simple error type carrying a message string.

2.2 error (builtin)

=(error "message")= — halts the program at runtime. Use for unrecoverable
errors. Codegen emits a call to =rt_error= which prints the message and calls
=exit(1)= ([[file:../codegen.c::6604]]).

#+BEGIN_SRC monad
(error "fatal: disk full")
#+END_SRC

#+BEGIN_SRC wisp
error "fatal: disk full"
#+END_SRC

2.3 undefined

=(undefined)= — placeholder for unimplemented branches. Codegen emits an LLVM
=unreachable= instruction or a call to =rt_undefined= depending on context
([[file:../codegen.c::6685]]). Non-exhaustive pattern matches emit =(undefined)=
as their fallback.

#+BEGIN_SRC monad
(define (fib n)
  (match n
    (0 1)
    (1 1)
    (_ (undefined))))  ;; placeholder
#+END_SRC

#+BEGIN_SRC wisp
define fib n
  match n
    0 1
    1 1
    _ undefined  ;; placeholder
#+END_SRC

3. ?? (null-coalescing)

=(?? expr default)= ([[file:../codegen.c::5763]]) returns =expr= if not =nil=,
=default= otherwise.

#+BEGIN_SRC monad
(?? (lookup key dict) "default")
(?? (safe-div a b) 0)
(?? (scores "Sam") 0)           ;; map lookup with default
#+END_SRC

#+BEGIN_SRC wisp
?? (lookup key dict) "default"
?? (safe-div a b) 0
?? (scores "Sam") 0
#+END_SRC

Implementation: evaluates =lhs=, checks nil via =rt_value_is_nil=, branches to
=rhs= or returns =lhs= unwrapped. Scalar types (Int, Float, Bool, Char) are
never nil and skip the check entirely.

For the conditionals reference (if, unless, etc.), see [[file:conditionals.org][Conditionals]].
For the ?? operator in the context of optional values, see [[file:maybe.org][Maybe (Option)]].

Key Points:
- =(error "msg")= halts the program unconditionally = use for unrecoverable failures
- =(undefined)= is a placeholder for unimplemented branches, emits LLVM =unreachable=
- =(?? expr default)= returns =expr= if not =nil=, =default= otherwise
- Debug mode uses fat pointers for nil tracing; release mode is a single null check
- =(assert-eq expected actual)= is the primary test assertion, used in codegen fixtures

See also: [[file:conditionals.org][Conditionals]]

4. Pattern: Option type with match

The =(Option a)= type provides a safe return type for fallible
operations. See [[file:maybe.org][Maybe (Option)]] for the full reference.

#+BEGIN_SRC monad
(define (safe-div a b -> (Option Int))
  (if (= b 0) None
    (Some (/ a b))))
#+END_SRC

#+BEGIN_SRC wisp
define safe-div a b -> (Option Int)
  if (= b 0) None
    Some (/ a b)
#+END_SRC

For pattern matching syntax, see [[file:patterns.org][Patterns]].

5. Pattern: Result pattern

Use =(Error String)= for operations that need error messages:

#+BEGIN_SRC monad
(define (parse-int s -> (Result Int String))
  ...)
#+END_SRC

#+BEGIN_SRC wisp
define parse-int s -> (Result Int String)
  ...
#+END_SRC

See [[file:result.org][Result]] for the full reference.

6. Debug vs Release nil behaviour

- **Debug (fat pointer):** raw pointer + static string table baked into the
  binary. =nil= carries a source-location tag, enabling trace output without
  heap allocation.
- **Release (thin pointer):** =??= is a single null check; the string table is
  stripped entirely. No trace, no overhead.

6.1 Nil Bubble Tracing (design in progress)

In debug mode, a nil value propagating through chained calls carries an origin
trace. When caught by =??=, the trace can report where the nil came from:

#+BEGIN_SRC text
?? caught nil:
  old-safe-last [] -> nil               (File.mon:12) <- origin
  └─ safe-last [_|xs] (File.mon:8 )
     └─ process-file filename           (File.mon:34)
        └─ handled: result ?? "default" (File.mon:67)
#+END_SRC

The trace is recorded via source-location tags in the fat pointer metadata.

For debug vs release build modes, see [[file:build.org][Build System # Three Build Modes]].
For the runtime value system, see [[file:../context/language.org::runtime-value-system][context/language.org — Runtime Value System]].

See also: [[file:build.org][Build System]]

7. assert-eq (test assertion)

=(assert-eq expected actual)= — the primary test assertion primitive. Codegen
evaluates both arguments and calls =rt_equal_p= to compare them. If comparison
fails, it prints a diagnostic with =show= of both values and reports the
failure location ([[file:../codegen.c::6431]]). Used extensively in codegen
test fixtures.

#+BEGIN_SRC monad
(assert-eq 4 (* 2 2))
#+END_SRC

#+BEGIN_SRC wisp
assert-eq 4 (* 2 2)
#+END_SRC

For the embedded test block syntax, see [[file:testing.org][Testing # Embedded Tests Blocks]].

See also: [[file:testing.org][Testing]]
