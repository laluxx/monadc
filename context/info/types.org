Monad's type system combines Hindley-Milner inference for the surface language with a dependent type checker based on Intuitionistic Type Theory (ITT). This page covers ground types, compound types, type annotations, type inference, and all advanced type features.

* Quick Reference

| Form                   | Description                   |                 |
|------------------------+-------------------------------+-----------------|
| Type :: Base           | Ground type (e.g. =Int=, =Float=) |                 |
| =(define [x :: Int] 42)= | Typed value binding           |                 |
| =A -> B=                 | Function type                 |                 |
| =(data T (A ...))=       | Algebraic data type           |                 |
| =(type Name Base)=       | Type alias                    |                 |
| =(class C a (meth ...))= | Type class declaration        |                 |
| ={ x ∈ T               | pred }=                       | Refinement type |

The Monad type system is /dual/: it combines Hindley-Milner (HM) inference
for the surface language with a full dependent type checker based on
Intuitionistic Type Theory (ITT) for advanced use. The two systems are
bridged by a =TERM_EMBED= node type, allowing mixed programs.

* 2. Primitive Types

Monad has a dual type system: Hindley-Milner (HM) inference
([[file:../infer.c]]) for the surface language, and a dependent type
checker ([[file:../dep.c]]) based on Intuitionistic Type Theory (ITT)
for advanced use. The two systems are bridged by a =TERM_EMBED= node
type (see [[file:../context/dep.org::dep-interop-bridge][context/dep.org — HM/ITT Bridge]]).

The =TypeKind= enum ([[file:../types.h::8]]) defines 40 type categories
documented in [[file:../context/types.org][context/types.org]].

* 4. Type Annotations

** 4.1 The =::= (double colon) syntax

Type annotations use the =::= operator to declare the expected type of a
binding, parameter, or expression. The =::= syntax is desugared to a
single colon in the AST — both forms are accepted:

#+BEGIN_SRC monad
(define [x :: Int] 42)    ;; identical to
(define [x : Int] 42)
#+END_SRC

#+BEGIN_SRC wisp
define x :: Int = 42    ;; identical to
define x : Int = 42
#+END_SRC

** 4.2 Typed vs Untyped Code

Annotations are optional — Hindley-Milner inference fills in types
automatically. Compare untyped and typed styles:

#+BEGIN_SRC monad
;; Untyped — types inferred by HM
(define x 42)
(define (add a b) (+ a b))

;; Typed — explicit annotations
(define [x :: Int] 42)
(define [add :: Int -> Int -> Int]
  (lambda [a :: Int] [b :: Int] (+ a b)))
#+END_SRC

#+BEGIN_SRC wisp
;; Untyped — types inferred by HM
define x 42
define add a b
  + a b

;; Typed — explicit annotations
define x :: Int = 42
define add :: Int -> Int -> Int
  a :: Int  b :: Int -> + a b
#+END_SRC

The inference engine generates type constraints and unifies them
([[file:../context/infer.org::unification][context/infer.org — Unification]]). Annotations add
additional constraints that guide or constrain the inference.

** 4.3 Binding Syntax

Bind variables with type annotations using square brackets:

#+BEGIN_SRC monad
;; Typed value definition
(define [x :: Int] 42)

;; Typed function parameter
(lambda [x :: Int] (+ x 1))

;; Function with typed signature
define square :: Int -> Int
  x -> x * x
#+END_SRC

#+BEGIN_SRC wisp
;; Typed value definition
define x :: Int = 42

;; Typed function parameter
lambda x :: Int
  + x 1

;; Function with typed signature
define square :: Int -> Int
  x -> x * x
#+END_SRC

** 4.4 Annotations in Compound Types

Type annotations compose with compound types:

#+BEGIN_SRC monad
(define [xs :: List :: Int] '(1 2 3))
(define [lookup :: String -> Map :: String :: Int -> (Option Int)]
  ...)
#+END_SRC

#+BEGIN_SRC wisp
define xs :: List :: Int = '(1 2 3)
define lookup :: String -> Map :: String :: Int -> Option Int
  ...
#+END_SRC

See [[file:type-inference.org][Type Inference]] for details on how annotations interact with
inference.

** 4.5 Type Annotations Key Points

- =::= and =:= are equivalent in the AST
- Annotations are optional — HM inference handles untyped code
- Square brackets surround typed bindings: =[x :: Int]=
- Annotations compose through compound types

* 5. Type Inference

The =infer= pass ([[file:../infer.c]]) handles surface-level type
inference: constraint generation, unification (Robinson's algorithm),
generalization, and instantiation (let-polymorphism).

MonadC uses Hindley-Milner (HM) type inference as its primary type
system — the same core algorithm as ML, Haskell, and OCaml. Inference
is O(n) for ground types and runs before codegen.

** 5.1 Key Concepts

| Concept          | Description                                         | Context Link                              |
|------------------+-----------------------------------------------------+-------------------------------------------|
| Algorithm W      | Damas-Milner constraint generation + unification    | [[file:../context/infer.org::constraint-generation][context/infer.org — Constraint Generation]] |
| Let-polymorphism | =let=-bound names are generalized; lambda params mono | [[file:../context/infer.org::generalization-instantiation][context/infer.org — Generalization]]        |
| Monomorphization | Each polymorphic instantiation → separate LLVM fn   | [[file:../context/codegen.org::codegen-mono-cache][context/codegen.org — Mono Cache]]          |
| Type variables   | Lowercase identifiers: =∀a. a → a=                    | [[file:../context/infer.org::type-scheme][context/infer.org — Type Scheme]]           |
| Unification      | Robinson's algorithm with occurs check              | [[file:../context/infer.org::unification][context/infer.org — Unification]]           |
| Constraint gen   | Walk AST, emit equality constraints                 | [[file:../context/infer.org::constraint-generation][context/infer.org — Constraint Generation]] |
| Zonking          | Substitute solved variables for concrete types      | [[file:../context/infer.org::zonking][context/infer.org — Zonking]]               |

** 5.2 Example: Polymorphic Identity

#+BEGIN_SRC monad
(define [id :: a -> a] x -> x)       ;; generalized at top-level
(let [f (lambda [x] x)]              ;; generalized in let
  (show (f 42))                       ;; instantiates Int
  (show (f "hello")))                 ;; instantiates String
#+END_SRC

#+BEGIN_SRC wisp
define id :: a -> a
  x -> x       ;; generalized at top-level
let f = lambda x
        x       ;; generalized in let
  show (f 42)                       ;; instantiates Int
  show (f "hello")                   ;; instantiates String
#+END_SRC

The =id= function works at any type because it is generalized at the
=define= level. Each use (=f 42= and =f "hello"=) instantiates a
different monomorphic copy via monomorphization.

See [[file:type-inference.org][Type Inference]] for full reference including unification
details, polymorphism, and known limitations.

** 6.1 Dependent Types (ITT)

The =dep= pass ([[file:../dep.c]]) implements dependent type checking
with Normalization by Evaluation (NbE)
([[file:../context/dep.org::dep-nbe-kernel][context/dep.org — NbE Kernel]]).

| Term       | Syntax         | Meaning                  |
|------------+----------------+--------------------------|
| Universe   | =(Type n)=       | Type universe at level n |
| Pi type    | =((x : A) -> B)= | Dependent function space |
| Sigma type | =((x : A) × B)=  | Dependent pair type      |
| Equality   | =(Eq a x y)=     | Propositional equality   |
| Nat        | =Nat=            | Natural numbers          |

** 6.2 HM / ITT Bridge

The two type systems coexist through a =TERM_EMBED= bridge
([[file:../context/dep.org::dep-interop-bridge][context/dep.org — HM/ITT Bridge]]). Ground
types inferred by HM are embedded as terms in the ITT kernel when
dependent code references them. This allows mixed programs where a
function has an HM-polymorphic outer signature and a dependent inner
contract:

#+BEGIN_SRC monad
(define [vec-append :: Vec a n -> Vec a m -> Vec a (+ n m)]
  ...)
#+END_SRC

The HM path handles the polymorphic =a= and the outer scheme; the ITT
path handles the arithmetic over =n= and =m= in the return type. The
bridge flow:

1. HM inference runs first, producing a typed term with embedded
   =TERM_EMBED= nodes where the user wrote dependent annotations.
2. The dep checker descends into those nodes and runs
   bidirectional NbE.
3. If both pass, the term is fully checked. If the dep checker
   encounters HM-only types, it uses =TERM_EMBED= as an opaque
   placeholder and checks definitional equality at the HM level.

** 6.3 NbE Kernel

The dependent checker uses Normalization by Evaluation with:
- A semantic =Value= domain supporting closures, environments, and
  stuck neutral terms
  ([[file:../context/dep.org::dep-value-struct][context/dep.org — Value Struct]]).
- Bidirectional =infer/=check= entry points
  ([[file:../context/dep.org::dep-bidirectional-checker][context/dep.org — Bidirectional Checker]]).
- Metavariables (holes) for partial type inference in dependent
  contexts ([[file:../context/dep.org::dep-metactx][context/dep.org — Meta Context]]).
- Definitional equality via NbE: two terms are equal if they
  normalize to the same semantic value.

This foundation will scale to inductive families, higher universes,
and cubical extensions without architectural rewrites.

** Complete example:

#+BEGIN_SRC monad

(module Main)

;; All major type features in one program

  (Node 5 (Node 3 Leaf Leaf) (Node 7 Leaf Leaf)))

;; Type alias
(type Count Int)

;; Refinement type
(type Positive { n :: Int | (> n 0) })
(define [n :: Positive] 10)

(show (id "hello"))
(show (+ x 1))
#+END_SRC

** Common Pitfalls

- **:: and : are equivalent, not different operators**: Both desugar
  to the same AST node. There is no semantic difference — pick one.
- **Type annotations are optional but recommended for top-level**: HM
  inference is complete, but omitting annotations on top-level bindings
  can produce confusing error messages when inference fails downstream.
- **Fixed-width types do NOT auto-promote**: `I8` + `I16` is a type
  error — use explicit [[file:casting.org][Casting]].
- **Refinement types with non-literal arguments defer to runtime**: The
  compiler only checks literals at compile time; variables fall through
  to a runtime predicate check (or silently pass without one).
- **Type variables are implicitly quantified — explicit ∀ is needed
  only for constraints**: Surface syntax `(define (id x) x)` is the same
  as `(define [id :: a -> a] x -> x)`.

([[file:../context/types.org::types-summary-table][context/types.org — TypeKind Enum]]):

| Category | Count | Types                                                                         |
|----------+-------+-------------------------------------------------------------------------------|
| Ground   |    13 | INT, FLOAT, CHAR, STRING, SYMBOL, BOOL, HEX, BIN, OCT, KEYWORD, RATIO, NIL, PATH  |
| Numeric  |    13 | F32, F80, I8, U8, I16, U16, I32, U32, I64, U64, I128, U128, INT_ARBITRARY         |
| Compound |     9 | LIST, ARR, FN, LAYOUT, SET, MAP, PTR, OPTIONAL, COLL                              |
| Meta     |     5 | UNKNOWN, VAR, ARROW, VARIADIC, APP                                                |

Each category serves a distinct role:
- *Ground*: primitive value types that carry data
- *Numeric*: fixed-width and numeric representation types
- *Compound*: types built from other types (collections, functions)
- *Meta*: compiler-internal types for inference and type construction

See [[file:../context/types.org][context/types.org]] for full per-kind documentation.
