Test coverage: the arena allocator ([[file:../arena.c]]) is tested
through the entire compiler pipeline — every codegen fixture exercises
arena allocation.

Monad does not have a garbage collector. All runtime values are
allocated in compiler-managed arena regions — bump-pointer allocation
with deterministic lifetime and no GC pauses.

For the internal design, see [[file:../context/arena.org][context/arena.org]]. For the runtime
memory model, see [[file:../context/runtime.org][context/runtime.org]].

This page covers the arena allocator used throughout the Monad
compiler: bump-pointer allocation, checkpoint/restore, lifetime model,
and tradeoffs vs garbage collection.

* 1. Bump-Pointer Arena

The primary allocator ([[file:../arena.c]], [[file:../arena.h]]) is a
bump-pointer arena:

- Blocks are 4 MiB (=ARENA_DEFAULT_BLOCK=, arena.h:36)
- Allocation is O(1): pointer bumped by =size= bytes, rounded to
  =ARENA_ALIGN= (8 bytes)
- When a block is full, a new block is allocated (doubling in size,
  growth factor ~2) and prepended to the singly-linked list

#+BEGIN_SRC text
Arena blocks:
[4MB block] -> [8MB block] -> [16MB block] -> ...
^                                    ^
bump pointer (allocates here)        next block (overflow)
#+END_SRC

The arena is used throughout the compiler: the parser ([[file:../reader.c]])
allocates AST nodes, the macro expander ([[file:../macro.c]]) allocates
expanded forms, the type inferencer ([[file:../infer.c]]) allocates type
trees, and the codegen ([[file:../codegen.c]]) consumes the arena. See
[[file:../context/codegen.org][context/codegen.org]] for the codegen pipeline.

*** Key Points
- O(1) bump-pointer allocation — no malloc overhead per allocation
- 4 MiB default blocks with doubling growth factor
- Every compiler phase uses the same arena: parse, expand, infer, codegen
- No GC, no reference counting, no per-object free

See also: [[file:performance.org][Performance]]

* 2. Free / Reset

| Operation        | Behavior                                       |
|------------------+------------------------------------------------|
| =arena_free=     | Walks block chain, frees all OS memory         |
| =arena_reset=    | Rewinds all blocks (used=0), keeps memory mapped |
| =arena_checkpoint=| Saves (block, offset) snapshot                  |
| =arena_restore=  | Rewinds to checkpoint, frees later blocks       |

=arena_reset= is the common fast path — zero mallocs.

The two-arena strategy (permanent + scratch) is documented in
[[file:performance.org][Performance Guide]] and
[[file:../context/arena.org][context/arena.org]].

*** Key Points
- =arena_reset= keeps memory mapped — fastest reuse, no syscalls
- =arena_free= returns everything to the OS (cleanup on exit)
- Two-arena pattern: permanent = whole compilation, scratch = per-phase

See also: [[file:performance.org][Performance]]

* 3. Checkpoint / Restore

=ArenaCheckpoint= saves (block, offset). =arena_restore= rewinds to
that point and frees any blocks allocated after it:

#+BEGIN_SRC C
ArenaCheckpoint cp = arena_checkpoint(&arena);
// ... allocate temporary values ...
arena_restore(&arena, &cp);    // free everything after checkpoint
#+END_SRC

This is a stack-based rollback (LIFO only). Checkpoint/restore is
essential for the REPL ([[file:repl.org][REPL]]) and the language server,
where compilation phases are interleaved with user interaction. See
[[file:../context/repl.org][context/repl.org]].

*** Key Points
- LIFO stack discipline only — non-LIFO patterns require the heap
- Zero runtime cost when no restore is needed
- Enables REPL incremental compilation without leaking memory

See also: [[file:repl.org][REPL]]

* 4. Tradeoffs vs GC

| Aspect         | Arena Allocation                | Garbage Collector              |
|----------------+---------------------------------+---------------------------------|
| Allocation     | O(1) bump-pointer               | O(1) bump + GC pauses later     |
| Deallocation   | Batch (arena reset/free)        | Incremental/mark-sweep          |
| Pauses         | None (deterministic)            | Stop-the-world for GC cycles    |
| Fragmentation  | None (linear bump)              | Can fragment over time          |
| Flexibility    | LIFO only (stack discipline)    | Any order (cycle collection)    |
| Memory return  | Keeps mapped (arena_reset)      | Returns to OS lazily            |

The tradeoff is acceptable for a compiler workload: allocations follow
a stack discipline (parse → infer → codegen → reset). For runtime data
that outlives the compiler pass, see [[file:heap.org][Heap Allocation]].

*** Key Points
- Arena wins on determinism, allocation speed, and cache behaviour
- GC wins on flexibility (non-LIFO lifetimes, cyclic data)
- Compiler workloads fit the stack-discipline sweet spot perfectly

See also: [[file:heap.org][Heap Allocation]]

* 5. Lifetime Model

1. Compilation begins: arena is fresh
2. Parser allocates AST nodes in the arena ([[file:../reader.c]])
3. Macro expansion ([[file:../macro.c]]) allocates new nodes
4. Type inference ([[file:../infer.c]]) allocates type trees
5. Codegen ([[file:../codegen.c]]) consumes arena — output is LLVM IR (arena-independent)
6. Compilation ends: =arena_free= — all memory reclaimed at once

No individual =free= calls. No reference counting. No GC.

For the compilation pipeline, see [[file:build.org][Build System]] and
[[file:../context/build.org][context/build.org]].

*** Key Points
- Single pass: allocate throughout, consume at codegen, free all at once
- LLVM IR output is arena-independent — no dangling pointer risk
- Entire compiler uses exactly one memory management strategy

See also: [[file:build.org][Build System]]
