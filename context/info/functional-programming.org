* 1. Pure Functions

Functions in Monad are pure by default: given the same arguments, they return
the same value and produce no side effects. Mutation via =set!= impurifies the
containing function for the purposes of the purity tracking system.

#+BEGIN_SRC monad
(define (square [x :: Int] -> Int)
  (* x x))

(define (multf [x :: Float] -> [y :: Float] -> Float)
  "Multiply X and Y as Floats"
  (* x y))
#+END_SRC

#+BEGIN_SRC wisp
define (square [x :: Int] -> Int)
  * x x

define (multf [x :: Float] -> [y :: Float] -> Float)
  "Multiply X and Y as Floats"
  * x y
#+END_SRC

Functions are defined with typed signatures using bracket syntax:
=[param :: Type]= parameters and =-> ReturnType= return annotation.

See [[file:purity.org][Purity]], [[file:define.org][Define]], and [[file:types.org][Types]].

*** Key Points
- Pure by default: same input → same output, no side effects
- =set!= and =<-= break purity locally
- Typed signatures use =[param :: Type]= and =-> ReturnType= syntax

See also: [[file:purity.org][Purity]]

* 2. Lambda

#+BEGIN_SRC monad
(define increment (lambda [x] (+ x 1)))
(define apply (lambda [f x] (f x)))
#+END_SRC

#+BEGIN_SRC wisp
define increment (lambda [x] (+ x 1))
define apply (lambda [f x] (f x))
#+END_SRC

No =fn= shorthand — use =lambda= ([[file:../codegen.c]]). All surface syntax
forms for functions (=(lambda [x] body)=, =(define (f x) body)=, Wisp
=x -> body=) are unified by the reader into =AST_LAMBDA= nodes
([[file:../reader.c::3244]]). The parser does not distinguish between surface
forms — the distinction is purely syntactic.

See [[file:lambda.org][Lambda]] for the full reference and [[file:wisp.org][Wisp Syntax]].

*** Key Points
- =lambda= is the only anonymous function syntax (no =fn= shortcut)
- All function forms converge to =AST_LAMBDA= in the reader
- Non-capturing lambdas compile to statically-typed ABI functions

See also: [[file:lambda.org][Lambda]]

* 3. Recursion

Named =define= is the standard form for recursion. For local recursion:

#+BEGIN_SRC monad
(letrec [fact (lambda [n]
                (if (<= n 1) 1
                  (* n (fact (- n 1)))))])
  (fact 5))

;; Top-level recursive function:
(define (fib [n :: Int] -> Int)
  (if (<= n 2) 1
    (+ (fib (- n 1)) (fib (- n 2)))))
#+END_SRC

#+BEGIN_SRC wisp
letrec [fact (lambda [n]
               (if (<= n 1) 1
                 (* n (fact (- n 1)))))])
  fact 5

;; Top-level recursive function:
define (fib [n :: Int] -> Int)
  if (<= n 2) 1
    + (fib (- n 1)) (fib (- n 2))
#+END_SRC

=letrec= is reader-desugared (no strcmp dispatch handler) — see
[[file:control.org][Control Flow]] for known gaps and [[file:recursion.org][Recursion]]
for recursion patterns.

*** Key Points
- Top-level recursion via =define=; local recursion via =letrec=
- =letrec= is desugared by the reader, not codegen
- Tail-call elimination is not yet implemented — recursive calls consume stack

See also: [[file:recursion.org][Recursion]]

* 4. Referential Transparency

Expressions can be replaced with their values without changing program behavior.
Variables bound with =define= or =let= are immutable unless mutated with =set!=
or =<-=.

See [[file:purity.org][Purity]] and [[file:mutation.org][Mutation]].

*** Key Points
- Immutable bindings guarantee referential transparency
- Safe to inline, reorder, cache, or CSE pure expressions
- Only =set!= and =<-= break transparency

See also: [[file:purity.org][Purity]]

* 5. Hindley-Milner Polymorphism

MonadC uses Hindley-Milner type inference with type variables (= 'a =, ='b=,
etc.). Polymorphic functions like identity instantiate correctly at multiple
concrete types via monomorphization:

#+BEGIN_SRC monad
(define [id :: a -> a]
  (lambda [x] x))

(show (id 42))          ;; id_Int — specialized to Int
(show (id "hello"))     ;; id_String — specialized to String
#+END_SRC

#+BEGIN_SRC wisp
define [id :: a -> a]
  lambda [x] x

show (id 42)          ;; id_Int — specialized to Int
show (id "hello")     ;; id_String — specialized to String
#+END_SRC

The compiler generates a separate LLVM function for each concrete type
instantiation. Unannotated parameters are NOT boxed objects — each concrete
call generates zero-overhead native code
([[file:../context/language.org::hm-polymorphism][context/language.org]]).

The type system surface supports:
- Polymorphic schemes: =∀a. a → a=
- Type applications: =Maybe Float=, =Result Int String=
- Constrained polymorphism via class constraints (aspirational)

See [[file:type-inference.org][Type Inference]] and [[file:types.org][Types]].

*** Key Points
- Hindley-Milner inference with monomorphization at codegen
- Each concrete type instantiation generates separate, optimized machine code
- No boxing, no vtable dispatch — zero-cost polymorphism
- Type annotations help inference and trigger earlier monomorphization

See also: [[file:type-inference.org][Type Inference]]

* 6. Higher-Order Functions (Aspirational)

Functions are first-class values at the semantic level. However, closures have
a known codegen gap: =fn=, =lambda=, and =defn= have no strcmp dispatch handler
([[file:../context/language.org::codegen-gaps][context/language.org]]). No
closure allocation exists in the current codegen. This blocks:

- =map=, =filter=, =foldl=, =foldr= (not implemented)
- =apply=, =partial=, =compose= (not implemented)
- Pipe operator (=|>=) (reader-level sugar only)
- Storing functions in data structures (segfaults)

The closure dispatch gap is historically contingent, not principled — the parser
intercepts =(lambda ...)= and produces =AST_LAMBDA= before codegen sees it, but
the symbol =lambda= itself is not callable from paren-mode. The typed ABI (for
non-capturing lambdas) works: partial application of a typed binary function
returns a callable closure
([[file:../context/language.org::hm-polymorphism][context/language.org]]).

** 6.1 Closure ABI

When closures are implemented, they will use a dual ABI:
- **Typed ABI** (no captures): passes parameters as unboxed LLVM values — fast,
  zero overhead.
- **Closure ABI** (captures): passes =(env*, i64 n, RuntimeValue** args)= —
  the environment pointer, argument count, and a boxed argument array.

The choice is made at compile time: simple top-level lambdas with no captures
use the typed ABI; nested lambdas that capture variables use the closure ABI
([[file:../context/language.org::primitive-function][context/language.org]]).

** 6.2 Workaround for higher-order patterns

Use manual recursion and nested calls for iteration. Define concrete typed
functions instead of passing lambdas as arguments:

#+BEGIN_SRC monad
;; Instead of (map f xs), recurse explicitly:
(define map-add1 :: [Int] -> [Int]
  []     -> []
  [x|xs] -> (++ (+ x 1) (map-add1 xs)))
#+END_SRC

#+BEGIN_SRC wisp
;; Instead of (map f xs), recurse explicitly:
define map-add1 :: [Int] -> [Int]
  []     -> []
  [x|xs] -> (++ (+ x 1) (map-add1 xs))
#+END_SRC

See [[file:recursion.org][Recursion]] and [[file:iteration.org][Iteration]] for
alternative patterns.

*** Key Points
- Closures are a known codegen gap — no =map=/=filter=/=fold= yet
- Non-capturing lambdas work correctly via the typed ABI
- Workaround: manual recursion with concrete typed functions
- Future closure ABI will use dual dispatch (typed vs environment-based)

See also: [[file:lambda.org][Lambda]]
