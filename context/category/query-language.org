[DOC id:doc.category.query-language.purpose src:context/category/query-language.org conf:high]
  The category query language is the keyboard-native interface to Ctx.  It is
  deliberately not SQL: it is a small glyph-and-word language for exploring a
  typed category in a live terminal prompt.  Plain words find objects; symbolic
  operators transform the current selection through morphisms, projections, and
  bounded path composition.

[DEC id:dec.category.query-language.stackish date:2026-06-16 by:llm]
  The language is stack/array inspired rather than relational.  A query denotes
  a stream of object views.  Filters narrow the stream.  Graph operators map the
  stream through Hom-sets.  Projection operators forget details intentionally,
  acting as functors from Ctx to simpler categories such as Kind, File, or
  Concept.

[DOC id:doc.category.query-language.prompt src:context/category/query-language.org conf:high]
  The canonical prompt is =(Ctx) ◗=.  A narrower interface may replace =Ctx=
  with a current projection, for example =(superset) ◗= when the user is
  navigating a superset/category slice.  The right triangle =◗= means "push this
  expression through the current context projection".

| =>= | outgoing codomain map | =#reader >= |
| =>kind= | outgoing by edge kind | =?x >supports= |
| =<= | incoming domain map | =?x <= |
| =~= | one-hop neighborhood | =?x ~= |
| =*N= | radius for next graph op | =?x *2 ~= |

| =ƒkind= | projection/functor | =typed define ƒkind= |
| =!N= | result limit | =wisp !50= |

[OBS id:obs.category.query-language.non-sql src:context/category/query-language.org conf:high]
  The query is read left-to-right as transformations on a selection, not as a
  declarative table expression.  This keeps the command line short enough for a
  TUI while still exposing category-theoretic operations.

[DOC id:doc.category.query-language.semantics src:context/category/query-language.org conf:high]
  Let Ctx be the free category generated by graph G.  A query expression denotes
  a finite ranked subset of Obj(Ctx) or a virtual view derived from Obj(Ctx),
  such as a path view or projection group.  The outgoing operator =>= maps a set
  S to ={ cod(f) | f ∈ Gen(G), dom(f) ∈ S }=.  The incoming operator =<= maps S
  to ={ dom(f) | f ∈ Gen(G), cod(f) ∈ S }=.  The neighborhood operator =~= is
  the union of the identity image, outgoing image, and incoming image.

[INF id:inf.category.query-language.functor-projection from:doc.category.query-language.semantics conf:high]
  =ƒkind= is a functor-like projection because it preserves only object kind and
  forgets individual identity and most morphism detail.  It is intentionally
  lossy; the UI must display that information has been forgotten.

[DOC id:doc.category.query-language.information src:context/category/query-language.org conf:high supports:doc.category.info.search]
  Each query step should reduce uncertainty.  Plain search reduces lexical
  entropy; kind/path filters reduce type/source entropy; graph operators trade
  lexical precision for semantic adjacency; projections deliberately compress a
  result set.  A good TUI shows these effects as counts, kinds, Hom-set sizes,
  and path summaries.
