This page covers contributing to Monad: building from source, code
structure, the context-first workflow, commit format, and TDD pattern.

This page covers how to contribute to the MonadC compiler. For the full
commit message format specification (types, scopes, examples), see
[[file:../context/commit-format.org][context/commit-format.org]]. For the complete
project guidelines (hard constraints and soft preferences), see
[[file:../context/guidelines.org][context/guidelines.org]].

The context-first workflow is the single most important contribution
pattern: read context/ before work, extend context/ after work. Every
session should leave a trace in the context corpus.

Test-driven development: add a codegen fixture (=rt_<category>_<atom>.mon=)
before implementing the feature, then run =python3 tests/run.py= to verify.

See also: [[file:testing.org][Testing]]

2. Building from Source

Requires LLVM 19+, C11 compiler (=gcc= or =clang=), make:

#+BEGIN_SRC sh
git clone <repo>
cd monad
make
#+END_SRC

Build variants: =make= (debug), =make debug= (assertions and debug
output), =make release= (optimized, LTO). Clean with =make clean=.

See [[file:build.org][Build System]] for full build documentation and
[[file:../context/build.org][context/build.org]] for build system internals.
For system requirements, see [[file:../context/env.org][context/env.org]].

4. Context-First Workflow

Before starting any work, read the relevant context files in
=context/=. The context corpus records design decisions, invariants,
and rationale that the source code alone does not capture.

After completing work, extend the context corpus: record new
discoveries, decisions, verification results, and follow-up TODOs.
Every session should leave a trace in the context. See
[[file:../context/guidelines.org][context/guidelines.org]] for full rules and
[[file:../context/workflow.org][context/workflow.org]] for workflow examples.

Example workflow:

#+BEGIN_SRC sh
# Before work: read context
cat context/reader.org context/types.org

# Implement feature in compiler
vim reader.c types.c

# After work: extend context
cat >> context/reader.org << EOF
** discovered: ...
EOF
#+END_SRC

Key Points:
- Read =context/= before starting work, extend it after completing
- Every session should leave a trace in the context corpus
- Use TDD: add a codegen fixture first, implement, then verify with =python3 tests/run.py=
- The context corpus records design decisions the source code alone does not capture

5. Commit Message Format

Every commit follows this structure:

#+BEGIN_EXAMPLE
<type>(<scope>): <summary>

<body>
#+END_EXAMPLE

  5.1 Allowed Types

| Type      | When to use                                                       |
|-----------+-------------------------------------------------------------------|
| =feat=      | New language feature, compiler pass, or user-visible capability   |
| =fix=       | Bug fix                                                           |
| =refactor=  | Code restructuring, no behaviour change                           |
| =test=      | New or updated tests, corpus rows, or test infrastructure         |
| =context=   | Changes only to =context/= Org files or the visualizer              |
| =build=     | Makefile, CI, install, or toolchain changes                       |
| =docs=      | Documentation outside the context corpus (READMEs, man pages)     |
| =perf=      | Measurable performance improvement                                |
| =chore=     | Maintenance with no semantic effect (renaming, moves, cleanup)    |

  5.2 Allowed Scopes

| Scope       | Covers                                      |
|-------------+---------------------------------------------|
| =reader=      | reader.h, reader.c, lexer, parser, AST      |
| =wisp=        | wisp.c, Wisp surface-syntax layer           |
| =types=       | types.h, types.c, type system               |
| =infer=       | infer.c, type inference                     |
| =codegen=     | codegen.c, LLVM code generation             |
| =dep=         | dep.c, dependency and term elaboration      |
| =tests=       | tests/run.py, tests/*.mon, corpus TSV       |
| =context=     | context/ Org files                          |
| =visualizer=  | context-visualizer.py, context/visualizer/  |
| =build=       | Makefile, install scripts                   |

  5.3 Rules

- First line: 72-character hard limit, no trailing period
- Imperative mood throughout (e.g. "Add path literals", not "Added")
- Active voice, present tense for invariants
- Body wrapped at 72 characters
- Exact symbol names in =monospace=
- Prose, never bullet lists
- No footer section — everything in body prose
- Section order mirrors compilation pipeline: wisp → reader → types →
  infer → codegen → tests → context

See [[file:../context/commit-format.org][context/commit-format.org]] for full format specification and
examples.

6. Code Style Conventions

- **Naming**: Types =PascalCase=, values =camelCase= or =kebab-case=,
  predicates suffix with =?= (e.g. =zero?=, =empty?=), mutating
  operations suffix with =!= (e.g. =set!=, =reset!=
- **Indentation**: 2 spaces, no tabs
- **Comments**: =;;= for line comments, =;| |;= for block comments
- **Hybrid C-Lisp style**: function signatures use =[param :: Type]=
  declarations while bodies remain S-expressions, common in low-level/
  graphics code
- **Pure Wisp style**: indentation-based with equation definitions,
  type signatures, and pattern clauses, used in =core/prelude=

Example of code style conventions:

#+BEGIN_SRC monad
;; Naming: PascalCase for types, kebab-case for values
(define Color Int)
(define (zero? n) (== n 0))
(define (reset! state) ...)

;; Comments: ;; for line, ;| |; for block
;; Indentation: 2 spaces, close parens at end of line
(define (greet name)
  (if (zero? (length name))
    (show "Hello, World")
    (show (++ "Hello, " name))))
#+END_SRC

#+BEGIN_SRC wisp
# Naming: PascalCase for types, kebab-case for values
define Color Int
define zero? n
  == n 0
define reset! state ...

# Comments: ;; for line, ;| |; for block
# Indentation: 2 spaces
define greet name
  if (zero? (length name))
    show "Hello, World"
    show (++ "Hello, " name)
#+END_SRC

See [[file:style.org][Style Guide]] for the complete conventions and
[[file:../context/style.org][context/style.org]] for the full style taxonomy.

7. Test-Driven Development Pattern

When adding a new feature:

1. **Context** — Read the relevant context files in =context/= to
   understand the subsystem architecture and design rationale.
2. **Test** — Add a codegen fixture in =tests/codegen/= following the
   =rt_<category>_<atom>.mon= naming convention with appropriate
   =TEST-*= header metadata.
3. **Implementation** — Implement the feature in the compiler.
4. **Verification** — Run =python3 tests/run.py= and verify all tests
   pass. Add golden =.stdout= (and optionally =.json=, =.desugar=)
   sidecars as needed.

For parser-level features, add TSV rows to the appropriate
=reader-*.tsv= corpus file instead of (or in addition to) a codegen
fixture.

Example of a codegen fixture:

#+BEGIN_SRC monad
;; tests/codegen/rt_arith_add.mon
;; TEST-ARGS: -O0
;; TEST-NAME: arithmetic add

(define (main)
  (show (+ 1 2)))
#+END_SRC

#+BEGIN_SRC wisp
# tests/codegen/rt_arith_add.mon
# TEST-ARGS: -O0
# TEST-NAME: arithmetic add

define main
  show (+ 1 2)
#+END_SRC

See [[file:testing.org][Testing]] for test infrastructure and
[[file:fuzzing.org][Fuzzing]] for property-based testing. See
[[file:../context/tests.org][context/tests.org]] for test architecture
and [[file:../context/tests-arch.org][context/tests-arch.org]] for the test
corpus design.

8. Running Tests

#+BEGIN_SRC sh
python3 tests/run.py              # All tests (codegen, TSV, ASM)
python3 tests/run.py --ci         # CI mode (verbose failure output)
python3 tests/run.py codegen      # Codegen tests only
make fuzzing                      # Property-based fuzzing (41 .fuzz laws)
#+END_SRC

Core/prelude module tests: =python3 tests/run_core.py=

For test-writing conventions, see [[file:testing.org][Testing]] and
[[file:../context/test-contracts.org][context/test-contracts.org]].

See also: [[file:fuzzing.org][Fuzzing]]

Key Points:
- =python3 tests/run.py= runs all tests (codegen, TSV, ASM)
- =python3 tests/run.py --ci= for verbose CI failure output
- =python3 tests/run.py codegen= runs only codegen fixtures
- =make fuzzing= runs property-based testing (41 .fuzz laws)
- =python3 tests/run_core.py= runs core/prelude module tests

9. Before Contributing

1. Read relevant context docs in =context/= for the subsystem
2. Match the project's C coding style (arena allocator, no GC, C11)
3. Add tests for new features (codegen fixture or TSV row)
4. Verify existing tests pass: =python3 tests/run.py=
5. Use the commit format: =type(scope): description=
6. After contributing, extend the context corpus with what was done

For the C coding style conventions, see [[file:../context/style.org][context/style.org]]
and [[file:../context/arena.org][context/arena.org]] for the arena allocator pattern.

See also: [[file:style.org][Style Guide]]
