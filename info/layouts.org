Layout types provide C-compatible memory layouts for struct-like data.
Internally they are represented as =TYPE_LAYOUT= (types.h:23) with fields
stored in =LayoutField= structs (types.h:62-67): each field has a =name=,
=type=, =offset= (computed by =layout_compute_offsets= in types.c), and
=size=.

Test coverage: layout type construction, field access, and methods are
tested by =rt_layout_*.mon= fixtures (see
[[file:../tests/codegen/][tests/codegen/]]). FFI struct layout mapping via libclang is
covered by =rt_cffi_*.mon= fixtures. Examples: [[file:../how_to/Layouts.mon][how_to/Layouts.mon]].

** Declaration

#+BEGIN_SRC monad
(layout Point [x :: Float] [y :: Float])
(layout Matrix3x3
  [m11 :: Float] [m12 :: Float] [m13 :: Float]
  [m21 :: Float] [m22 :: Float] [m23 :: Float]
  [m31 :: Float] [m32 :: Float] [m33 :: Float])
#+END_SRC

#+BEGIN_SRC wisp
layout Point [x :: Float] [y :: Float]
layout Matrix3x3
  [m11 :: Float] [m12 :: Float] [m13 :: Float]
  [m21 :: Float] [m22 :: Float] [m23 :: Float]
  [m31 :: Float] [m32 :: Float] [m33 :: Float]
#+END_SRC

** Construction

#+BEGIN_SRC monad
(define p (Point 3.0 4.0))
#+END_SRC

#+BEGIN_SRC wisp
define p
  Point 3.0 4.0
#+END_SRC

** Field Access

Dot notation: =(p.x)= returns 3.0, =(p.y)= returns 4.0 (merged symbol
access handled by =codegen_dot_chain= in codegen.c). Access compiles
to a fixed-offset load from an LLVM struct pointer.

** Field Mutation

#+BEGIN_SRC monad
(set! p.x 5.0)     ;; mutates field in-place
(<- p.x 5.0)       ;; sugar for set!
#+END_SRC

#+BEGIN_SRC wisp
set! p.x 5.0     ;; mutates field in-place
<- p.x 5.0       ;; sugar for set!
#+END_SRC

** Field Options

Each field can carry options that modify its layout behavior:

| Option   | Description                                   |
|----------+-----------------------------------------------|
| :packed  | Remove padding between fields (no alignment) |
| :align N | Override field alignment to N bytes          |

#+BEGIN_SRC monad
(layout Packet [flags :: U8 :packed] [len :: Int :align 8])
#+END_SRC

#+BEGIN_SRC wisp
layout Packet [flags :: U8 :packed] [len :: Int :align 8]
#+END_SRC

Additional field-level flags stored on the =LayoutField= (types.c):

- *Inline fields*: nested layouts stored directly (not as pointers) —
  =layout_is_inline= flag on the parent.
- *Pointer fields*: fields storing a pointer to another layout.
- *Array fields*: fields that are fixed-size arrays of a layout type.

** Nested Layouts

Layout types can be embedded as fields of other layouts. The inner
layout is stored in the outer layout at the computed offset, not as
a separate heap allocation. The =layout_is_inline= flag on the inner
field marks this (types.h:602). For opaque C typedefs (like
=VkInstance=), =layout_is_scalar=true= treats the layout as a
single-word typedef with no accessible fields.

** Layout Offset Computation

Field offsets are computed by =layout_compute_offsets= (types.c) which
processes the =LayoutField= array:

1. Starting offset = 0 (adjusted for =layout_align=).
2. For each field: align to field's natural alignment (or =:align N=
   override), assign offset, advance by field size.
3. If =layout_packed=, skip alignment — fields are tightly packed.
4. Final =layout_total_size= is the last offset + size, rounded up to
   the struct alignment.

See [[file:../context/types.org][context/types.org]] for detailed field structures and offset rules.

** Array-of-Layouts

Create arrays of layout types using standard array syntax with the
layout as the element type:

#+BEGIN_SRC monad
(layout Point [x :: Float] [y :: Float])

;; Fixed-size array of 10 Points
(define [points :: Arr :: Point :: 10] [])

;; Heap array of Points
(define [points :: Arr :: Point] ~[])
#+END_SRC

#+BEGIN_SRC wisp
layout Point [x :: Float] [y :: Float]

;; Fixed-size array of 10 Points
define [points :: Arr :: Point :: 10] []

;; Heap array of Points
define [points :: Arr :: Point] ~[]
#+END_SRC

Each Point element is stored contiguously in the array, matching C
struct array layout.

** Methods (=where= clause)

The =where= and =method= clauses inside =layout= define methods on the
layout type. Methods receive =Self= as their first argument and are
accessed with arrow syntax:

#+BEGIN_SRC monad
(layout Vec3
  [x :: Float]
  [y :: Float]
  [z :: Float]
  where
    method ->array :: Self -> Array
      s -> [s.x s.y s.z])

(define v (Vec3 1.0 2.0 3.0))
(show v->array)   ; [1.0 2.0 3.0]
#+END_SRC

#+BEGIN_SRC wisp
layout Vec3
  [x :: Float]
  [y :: Float]
  [z :: Float]
  where
    method ->array :: Self -> Array
      s -> [s.x s.y s.z]

define v
  Vec3 1.0 2.0 3.0
show v->array   ; [1.0 2.0 3.0]
#+END_SRC

Method dispatch is tested in =rt_layout_*.mon= fixtures. The
=where= block can contain multiple =method= forms.

** Comparison: =layout= vs =data=

| Aspect       | =data= (ADT)            | =layout=               |
|--------------+-----------------------+----------------------|
| Memory       | Heap-allocated (cons) | Stack or inline      |
| C compat     | No                    | Yes                  |
| Polymorphism | Yes (type params)     | No                   |
| Field access | Pattern matching      | Dot notation         |
| Mutation     | Immutable             | =set!= / =<-=            |
| Layout       | Linked list / tagged  | C struct, contiguous |

Layout types trade polymorphism and immutability for C-compatible
contiguous memory, predictable offsets, and zero-copy FFI.
