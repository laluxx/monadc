Monad is a compiled Lisp-family functional language with a dual-surface
syntax (S-expressions and indentation-based Wisp). It targets x86-64
Linux via LLVM 19+ and is written in C (~27,000+ lines). The compiler
produces native binaries linked against a static runtime library
(=libmonad.a=).

- *Type system*: Hindley-Milner inference (constraint-based, with
  generalization, instantiation, and monomorphization) for ground types,
  plus a full Intuitionistic Type Theory (ITT) dependent type checker
  with Normalization by Evaluation (NbE), bidirectional checking, and
  a =TERM_EMBED= bridge between the two systems.

- *Memory*: Arena allocation with bump-pointer semantics, 4 MiB blocks,
  and checkpoint/restore. No general-purpose garbage collector — the
  arena is reset at predictable boundaries.

- *FFI*: C interop via =(include "header.h")= powered by libclang header
  parsing. No binding generators, IDL files, or wrapper code needed.

- *Macros*: Ordinary functions that return =Syntax= values and operate
  at compile time. No separate =macro= keyword. Full hygiene with
  rename-pair tracking, fixpoint expansion, and variadic splice.

- *Dual syntax*: S-expressions (paren-mode) and Wisp (indentation-based)
  are freely mixable within a single file and produce identical ASTs.

- *Self-hosting-ready*: =Main.mon= (1244 lines) serves as the standard
  library and primary compiler test surface. Core library modules exist
  under =core/= covering lists, sets, math, text processing, and more.

* Quick Start

#+BEGIN_SRC monad
(module Main)
(show "hello world")
#+END_SRC

Compile and run:

#+BEGIN_SRC sh
make && ./monad hello.mon -o hello && ./hello
#+END_SRC

Or with the Wisp syntax variant:

#+BEGIN_SRC monad
module Main
show "hello from wisp"
#+END_SRC

* Compiler Pipeline

The MonadC compiler transforms source to native code through these
stages:

1. *Reader* (reader.c) — Tokenizes and parses source text into an AST.
   Recognizes both S-expression syntax and Wisp indentation, producing
   identical AST nodes regardless of surface syntax. The reader is
   tested against a conformance corpus (=tests/reader-atoms.tsv=)
   covering every grammar production: numeric literals, strings, chars,
   paths, keywords, symbols, arrays (stack/fat/heap), sets, maps,
   ranges, lambdas, data declarations, type annotations, refinements,
   layout types, and inline assembly forms.

2. *Macro expansion* — Macro functions (ordinary =define= returning
   =Syntax=) are expanded at compile time. Expansion is fixpoint —
   the compiler re-expands until the term stabilizes. Hygiene is
   maintained through rename-pair tracking that prevents unintended
   variable capture.

3. *Wisp expander* (wisp.c) — If the source uses Wisp syntax, the
   expander desugars indentation into S-expressions. Wisp !-suffix
   and ?-suffix sugar are applied at this stage.

4. *Type inference / checking* — The dual type system runs:
   - *HM inference* (=infer.c=, =infer.h=) handles ground types with
     constraint generation, unification, generalization, and
     monomorphization. Fast path for non-dependent code.
   - *ITT dependent checker* (=dep.c=, =dep.h=) handles Pi-types,
     Sigma-types, universe levels, equality/refl/subst, Nat and
     Nat-elim, metavariables, and bidirectional infer/check entry
     points — all normalized via NbE (Normalization by Evaluation).

5. *Codegen* (=codegen.c=) — Lowers typed AST to LLVM IR. Dispatch
   uses a three-layer architecture: AST type switch → strcmp dispatch
   for 83+ known symbols → =register_builtins= for 94 env entries.
   Polymorphic functions are monomorphized — each concrete type
   instantiation generates separate machine code.

6. *Linking* — The compiler invokes LLVM's optimization pipeline, emits
   object code, and links against =libmonad.a= (the runtime) to produce
   a native executable.

* Key Ideas

- *Types are sets* — =Int= is the set of 63-bit integers. A refinement
  type like ={ x ∈ Int | x > 0 }= is a subset. Functions are relations.
  The compiler materializes infinite sets as values you can inspect.

- *Immutable by default* — Mutation is explicit: =set!= for variables,
  =conj!/disj!== for collections, =<-= for array elements. The language
  encourages pure functional style but provides controlled mutation.

- *Code is data* — Macros are just functions that return =Syntax=
  values. The quote operator (= '(...)=) and quasi-quote (= `(...)=)
  construct syntax objects at runtime, and macro expansion transforms
  them at compile time.

- *No trade-off between safety and control* — The same language gives
  you dependent types AND inline assembly, refinement types AND naked
  functions, lazy infinite lists AND C-compatible arrays — all without
  binding generators, FFI wrappers, or runtime overhead.

- *Continuous abstraction ladder* — Start with =show "Hello!"=, add
  type annotations, introduce polymorphic functions, prove invariants
  with dependent types, drop to layout-based structs for performance,
  and write inline assembly for hardware access — all in the same file
  and the same function.

* Philosophy

Monad is built on five core principles:

1. *No trade-off between safety and control* — You should not have to
   choose between dependent types and inline assembly, between
   mathematical guarantees and hardware control.

2. *Purity is practical* — Mathematical purity enables compile-time
   predicate checking, exhaustive pattern coverage verification, and
   zero-cost type abstraction through monomorphization.

3. *Syntax is instrument* — Two syntaxes (S-expressions and Wisp) for
   two tasks: programmatic generation and macro processing with
   S-expressions, readability in large codebases with Wisp. Both work
   for every construct and mix freely.

4. *Zero-to-metal abstraction ladder* — The ladder from =show "Hello!"=
   to inline assembly must be continuous, with no language boundary,
   binding layer, or runtime switch.

5. *Sets are the model* — Types are sets. Functions are relations.
   Refinement types are subsets. Universes are sets of sets. This is
   not metaphor — it is the compiler's semantic model.

* Project Status

MonadC is under active development. The compiler core is functional:
parsing, type inference, and codegen produce working binaries for a
large subset of the language. Known limitations include:

- Closures (=fn=, =lambda=, =defn=) lack a codegen strcmp handler —
  higher-order functions like =map= and =filter= are blocked.
- Several control forms (=cond=, =when=, =let=, =let*=, =letrec=) are
  registered builtins but lack strcmp dispatch.
- Map and set runtime construction is broken for untyped contexts
  (needs explicit =:: Map= / =:: Set= annotation).
- Core prelude modules are not yet automatically imported.
