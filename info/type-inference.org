Hindley-Milner (HM) type inference is Monad's primary type system — the same core algorithm as ML, Haskell, and OCaml. This page covers Algorithm W, unification, type variables, let-polymorphism, and type annotations.

* 1. Quick Reference

| Form                                | Description                        |
|-------------------------------------+------------------------------------|
| =(unify t1 t2)=                     | Structural type unification        |
| =(define [id :: a -> a] x → x)=     | Polymorphic identity               |
| =∀ a. a → a=                        | Universally quantified type        |
| =(define [x :: Int] 42)=            | Annotated binding                  |

Test coverage: HM inference is exercised by every typed codegen
fixture in [[file:../tests/codegen/types/hm/][tests/codegen/types/hm/]]. The polymorphic
identity and const functions are verified by codegen goldens
([[file:../context/language.org::466][context/language.org — HM surface]]).
HM/ITT bridge tests are in [[file:../tests/codegen/types/bridge/][tests/codegen/types/bridge/]].

MonadC uses Hindley-Milner (HM) type inference as its primary type
system — the same core algorithm as ML, Haskell, and OCaml. Inference
is O(n) for ground types and runs before codegen. For an overview of
the full type system, see [[file:types.org][Types]]. For advanced type features,
see [[file:advanced-types.org][Advanced Types]].

* 2. Algorithm W

The inference engine implements Algorithm W (Damas-Milner):

1. **Assign** a fresh type variable to each subexpression
2. **Unify** constraints generated by syntax structure
3. **Generalize** let-bound variables to polymorphic type schemes
4. **Instantiate** polymorphic schemes at use sites with fresh variables

| Phase | Step | Description |
|-------+------+-------------|
| 1     | Fresh variables | Each expression node gets a unique type variable |
| 2     | Constraint generation | Syntax structure generates equality constraints |
| 3     | Unification | Robinson's algorithm solves constraints |
| 4     | Generalization | Let-bound vars become =∀=. Free vars stay monomorphic |
| 5     | Instantiation | Each use of a polytype gets fresh type variables |

Implementation spans [[file:../infer.c]] (~750 lines) and
[[file:../infer.h]] (type definitions). See
[[file:../context/infer.org][context/infer.org]] for a detailed walkthrough of the
inference pipeline.

** Implementation Notes

- Inference via =infer_expr= ([[file:../context/infer.org::619]])
- Unification via Robinson's algorithm with union-find
  ([[file:../context/infer.org::525]])
- Occurs check prevents infinite types
  ([[file:../context/infer.org::479]])
- Generalization and instantiation
  ([[file:../context/infer.org::729]])

** Key Points

- O(n) for ground types; inference is not a bottleneck
- Let-polymorphism only — lambda parameters are always monomorphic
- Monomorphization at codegen time produces zero-cost native code
- Type annotations are optional; inference fills in all details

* 3. Unification

Two types are unified by structural matching:

#+BEGIN_SRC monad
(unify Int Int)                     ;; OK
(unify (-> Int Bool) (-> Int Bool)) ;; OK
(unify Int Bool)                    ;; error: type mismatch
#+END_SRC

#+BEGIN_SRC wisp
unify Int Int                     ;; OK
unify (-> Int Bool) (-> Int Bool) ;; OK
unify Int Bool                    ;; error: type mismatch
#+END_SRC

The unifier ([[file:../infer.c::150]]) walks type trees, substituting
type variables with concrete types as it goes. Occurs check prevents
infinite types (= α ≢ (→ α α)=). For type representation details, see
[[file:../context/types.org][context/types.org]].

** Key Points

- Unification is structural (syntactic) — two types match iff their
  tree structures are identical after substitution
- The occurs check ensures termination: α cannot appear inside its
  own substitution

* 4. Type Variables

Lowercase identifiers in type position are universally quantified:

#+BEGIN_SRC monad
(define (id x -> a) x)             ;; ∀a. a → a
(define (fst x y -> a) x)          ;; ∀a b. a → b → a
(define (swap [x :: a] [y :: b] -> (Pair b a))
  (Pair y x))
#+END_SRC

#+BEGIN_SRC wisp
define id x -> a
  x             ;; ∀a. a → a
define fst x y -> a
  x             ;; ∀a b. a → b → a
define swap x :: a  y :: b -> Pair b a
  Pair y x
#+END_SRC

Type variables are implicitly quantified. Inference generalizes
unconstrained variables at =let=-bound definitions. For explicit
universal quantification with constraints, see
[[file:type-classes.org#constraints-with-][Type Class Constraints]].

* 5. Polymorphism

Polymorphic functions instantiate at each use site:

#+BEGIN_SRC monad
(define [id :: a -> a]
  (lambda [x] x))

(show (id 42))          ;; id_Int — specialized by monomorphization
(show (id "hello"))     ;; id_String — separate LLVM function
#+END_SRC

#+BEGIN_SRC wisp
define id :: a -> a
  lambda x
    x

show (id 42)          ;; id_Int — specialized by monomorphization
show (id "hello")     ;; id_String — separate LLVM function
#+END_SRC

The compiler generates a separate LLVM function for each concrete type
instantiation via the =mono_cache= ([[file:../codegen.c::109]]).
Unannotated parameters are NOT boxed — each call generates zero-
overhead native code. See [[file:../context/codegen.org][context/codegen.org]] for
monomorphization internals.

* 6. Let-Polymorphism

Only =let=-bound (and top-level =define=-bound) names are generalized:

#+BEGIN_SRC monad
(define [id :: a -> a] x -> x)       ;; generalized
(let [f (lambda [x] x)]              ;; generalized
  (show (f 42))
  (show (f "hi")))
#+END_SRC

#+BEGIN_SRC wisp
define id :: a -> a
  x -> x       ;; generalized
let f = lambda x
        x
  show (f 42)
  show (f "hi")
#+END_SRC

Lambda-bound parameters (=lambda [x] x)= in a function without =define=)
are monomorphic — each is a single type variable that must be used
consistently. See [[file:lambda.org][Lambda]] for lambda expression semantics.

** Key Points

- =let= and =define= are the only generalization points
- Lambda parameters use the same type variable at every occurrence
- This is the Damas-Milner restriction; it keeps inference decidable

* 7. Type Annotations

Explicit annotations constrain inference:

#+BEGIN_SRC monad
(define [x :: Int] 42)              ;; annotated binding
(define (add [a :: Int] [b :: Int] -> Int)
  (+ a b))                          ;; fully annotated
#+END_SRC

#+BEGIN_SRC wisp
define x :: Int = 42              ;; annotated binding
define add a :: Int  b :: Int -> Int
  + a b                           ;; fully annotated
#+END_SRC

Annotations are optional — HM infers types from usage:

#+BEGIN_SRC monad
(define x 42)                        ;; inferred: Int
(define (add a b) (+ a b))          ;; inferred: Int → Int → Int
#+END_SRC

#+BEGIN_SRC wisp
define x 42                        ;; inferred: Int
define add a b
  + a b                            ;; inferred: Int → Int → Int
#+END_SRC

For =define= syntax details, see [[file:define.org][Define]]. For function type
syntax, see [[file:types.org#function-types][Function Types]].

** Key Points

- Annotations serve as documentation and as constraints on inference
- They are never required for well-typed programs (HM is complete for
  the ground type system)
- Mixed annotation/inference is fine: annotate the hot path, infer the
  rest

** Complete example:

#+BEGIN_SRC monad
;; Save as inference-demo.mon && monadc inference-demo.mon && ./inference-demo
(module Main)

;; Type inference in action — no annotations needed
(define x 42)                        ;; inferred: Int
(define y 3.14)                      ;; inferred: Float
(define (add a b) (+ a b))           ;; inferred: Int -> Int -> Int
(define (id x) x)                    ;; inferred: ∀a. a -> a

(show (add x 1))                     ;; 43
(show (id "hello"))                  ;; "hello"
(show (id 42))                       ;; 42

;; Mixed annotation and inference
(define [z :: Float] (+ y 1.0))
(show z)
#+END_SRC

** Common Pitfalls

- **Let-polymorphism only**: Lambda-bound parameters are monomorphic.
  A =lambda= parameter has a single type variable — it cannot be used
  at two different types within the same lambda body.
- **No higher-kinded inference**: =(fmap ...)= style requires explicit
  type annotations. The inference engine only handles types of kind `*`.
- **No rank-N types**: Explicit `∀` is parsed but inference cannot
  handle higher-rank types. Use top-level polymorphism instead.
- **Monomorphization can bloat code size**: Each concrete type
  instantiation generates a separate LLVM function. For heavy
  polymorphism, this can produce large binaries.
- **Type annotations guide, not override inference**: An annotation
  adds a constraint — if it conflicts with inferred types, unification
  fails with a type error.

* 8. Known Limitations

- No higher-kinded type inference (=fmap= style requires annotations)
- No GADT inference (pattern matching on type constructors)
- No rank-N types (explicit =∀= is parsed but inference is limited)
- Type classes use codegen-time resolution, not inference-time
  (see [[file:type-classes.org][Type Classes]])
- No bidirectional inference for dependent types — the NbE shadow pass
  runs separately from HM (see [[file:dependent-types.org][Dependent Types]])
