[[file:index.org][<- Back to Index]] · [[file:pair.org][Pairs]] · [[file:list.org][Lists]] · [[file:data.org][Data Types]] · [[file:maybe.org][Maybe]] · [[file:result.org][Result]] · [[file:booleans.org][Booleans]] · [[file:control.org][Control Flow]]

Pattern matching is compiled by the pattern match compiler
([[file:../pmatch.c]]) which desugars =pattern -> expr= clauses into
equivalent =(cond ...)= chains that the generic codegen can lower.
This is the central dispatch mechanism for ADTs, lists, and conditionals.

Test coverage: =rt_match_*.mon= fixtures in
[[file:../tests/codegen/][tests/codegen/]] cover ADT constructor matching, list
destructuring, guard clauses, and wildcard fallbacks.
Test architecture reference: [[file:../context/tests-arch.org][context/tests-arch.org]].

Pipeline:

1. Reader produces =AST_PMATCH= with =ASTPMatchClause= array
2. =pmatch_rename_anon_params= rewrites anonymous parameter refs
3. Codegen calls =pmatch_desugar= which generates =(cond ...)=
4. Codegen compiles the cond chain via normal dispatch

Desugaring happens at codegen time (after inference), not during
parsing, because the full type environment and ADT definitions are
only available after inference ([[file:../context/pmatch.org::56]]).

See [[file:../context/reader.org][context/reader.org]] for reader internals and
[[file:../context/codegen.org][context/codegen.org]] for LLVM lowering.

* 1. Implicit Pattern Matching (the default)

Inside a =define=, =lambda=, or =(let ...)= body, you do NOT need an
explicit =(match ...)= expression. If the body consists of multiple
clauses with =->= arrows, the function automatically pattern matches
on its parameters. This is the idiomatic way to write pattern-matching
functions in Monad.

See also [[file:define.org][Define]] for function definition syntax and
[[file:lambda.org][Lambda]] for anonymous function syntax.

** 1.1 Function with pattern clauses

#+BEGIN_SRC monad
;; Implicit match on the argument — no (match ...) needed
(define fact
  0 -> 1
  n -> (* n (fact (- n 1))))
#+END_SRC

#+BEGIN_SRC wisp
define fact
  0 -> 1
  n -> * n (fact (- n 1))
#+END_SRC

This is equivalent to:

#+BEGIN_SRC monad
(define fact (lambda (n)
  (match n with
    0 -> 1
    n -> (* n (fact (- n 1))))))
#+END_SRC

#+BEGIN_SRC wisp
define fact lambda n
  match n with
    0 -> 1
    n -> * n (fact (- n 1))
#+END_SRC

The compiler recognizes that the body of =define= is a sequence of
pattern clauses and wraps them in an implicit =match= on the
function's parameters. The same rule applies recursively: if a
pattern clause body itself contains clauses, those are also
implicit matches.

For recursive patterns see [[file:recursion.org][Recursion]].

** 1.2 Lambda with pattern clauses

Same applies to anonymous lambdas:

#+BEGIN_SRC monad
;; Wisp: default body is implicit match
(define classify :: Int -> String
  | 0       -> "zero"
  | n | > 0 -> "positive"
  | _       -> "negative")

;; S-expr: same thing
(define classify
  (lambda (n :: Int -> String)
    | 0 -> "zero"
    | n | (> n 0) -> "positive"
    | _ -> "negative"))
#+END_SRC

#+BEGIN_SRC wisp
define classify
  lambda n :: Int -> String
    | 0 -> "zero"
    | n | (> n 0) -> "positive"
    | _ -> "negative"
#+END_SRC

You can think of =define name pattern -> body= as the fundamental
form — the =match= expression is just the explicit version.

** 1.3 Why this works

The reader's =parse_lambda= function (reader.c:2714-2773) detects when
the body starts with a pattern clause token (=->=, =|=) and creates an
=AST_PMATCH= node wrapping the clauses instead of a regular expression
body. The codegen then desugars the =PMATCH= into a =cond= chain.

Key Points:
- =define=, =lambda=, and =let= bodies automatically pattern-match if they contain =->= clauses
- Explicit =(match ...)= is only needed for sub-expression matching
- The reader detects pattern clauses by looking for =->= or =|= tokens in the body position
- Implicit matching is syntactic sugar provided by the reader, not a separate IR node

* 2. Explicit match expression

Use =match= explicitly when you need to match on a specific
sub-expression (not the function parameter), or inside a non-function
body:

#+BEGIN_SRC monad
(match value with
  pattern -> body ...)
#+END_SRC

#+BEGIN_SRC wisp
match value with
  pattern -> body ...
#+END_SRC

Patterns are tried in order from top to bottom. The first matching clause
fires. There is no backtracking — once a pattern matches, its body is
evaluated and the result is returned.

See [[file:conditionals.org][Conditionals]] for =cond= as the desugaring target
and [[file:control.org][Control Flow]] for general flow control.

Key Points:
- Explicit =(match value with ...)= for matching sub-expressions
- Top-to-bottom ordering; first match wins (no backtracking)
- =match= desugars to =(cond ...)= at codegen time

* 3. Pattern kinds

The =PatternKind= enum ([[file:../reader.h::90]]) defines the
supported pattern forms:

| Pattern        | Example  | Matches                          |
|----------------+----------+----------------------------------|
| Wildcard       | ~_~        | Any value, no binding            |
| Variable       | ~n~        | Any value, bound as ~n~            |
| Int literal    | ~0~, ~-1~    | Exact integer                    |
| Float literal  | ~1.5~      | Exact float                      |
| String literal | ~"a"~      | Exact string                     |
| Char literal   | ~'\n'~     | Exact character                  |
| Empty list     | ~[]~ / ~nil~ | Empty list / null                |
| List with tail | ~[x      | xs]~                             | Destructure head and tail |
| List fixed     | ~[a b c]~  | Fixed-length list match          |
| Constructor    | ~(Just x)~ | ADT constructor + field bindings |

Wildcard (=_=) and variable (=n=) patterns both match any value; the
difference is that variable patterns bind the matched value for use in
the body, while wildcard discards it.

For more on ADT constructors see [[file:data.org][Algebraic Data Types]].
For list patterns see [[file:list.org][Lists]].

Key Points:
- Literal patterns test exact equality (=int=, =float=, =string=, =char=)
- Wildcard =_= matches anything without binding; variable patterns bind
- List patterns support fixed-length =[a b c]= and cons =[x|xs]= forms
- Constructor patterns destructure ADT fields by position

* 4. Guard clauses

#+BEGIN_SRC monad
(match x with
  n | (> n 0) -> "positive"
  n | (< n 0) -> "negative"
  _            -> "zero")
#+END_SRC

#+BEGIN_SRC wisp
match x with
  n | (> n 0) -> "positive"
  n | (< n 0) -> "negative"
  _            -> "zero"
#+END_SRC

Use ~|~ (pipe) after the pattern, then a guard condition, then ~->~
and the body. Multiple guards per clause are supported — each must be
satisfied for the clause to match.

The guard condition is combined with the pattern condition as
=(and pattern-condition guard-condition)= during desugaring.

See [[file:booleans.org][Booleans]] for boolean logic and
[[file:conditionals.org][Conditionals]] for conditional evaluation.

Key Points:
- Guards extend a pattern with an additional boolean condition
- Syntax: =pattern | guard-condition -> body=
- Multiple guards per clause: =pattern | guard1 | guard2 -> body=
- Guards desugar to =(and pattern-condition guard)=

* 5. Leading pipe separator

Pattern clauses may have an optional leading ~|~:

#+BEGIN_SRC monad
(match x with
  | 0 -> "zero"
  | _ -> "non-zero")

(lambda (x)
  | 0 -> "zero"
  | _ -> "non-zero")
#+END_SRC

#+BEGIN_SRC wisp
match x with
  | 0 -> "zero"
  | _ -> "non-zero"

lambda x
  | 0 -> "zero"
  | _ -> "non-zero"
#+END_SRC

The parser consumes ~|~ as a clause separator, not part of the
pattern. Both with and without leading pipe work identically. The
leading pipe is purely cosmetic for aligning multi-clause matches.

Key Points:
- Leading ~|~ is optional and purely cosmetic
- Does not affect semantics or codegen
- Encouraged for readability in multi-clause matches

* 6. Tail patterns (head | tail)

List patterns support destructuring with ~|~ separating head and tail:

#+BEGIN_SRC monad
(match xs with
  [x|xs] -> (show x)        ;; binds head to x, tail to xs
  []     -> "empty")
#+END_SRC

#+BEGIN_SRC wisp
match xs with
  [x|xs] -> show x
  []     -> "empty"
#+END_SRC

This is the standard Lisp cons-pattern: =[x|xs]= desugars to matching
a non-empty list with head bound to =x= and tail bound to =xs=.

See [[file:list.org][Lists]] for more list operations and
[[file:pair.org][Pairs]] for the cons-cell structure.

Key Points:
- =[x|xs]= destructures a non-empty list into head and tail
- =[]= matches the empty list (=nil=)
- Fixed-length =[a b c]= matches lists of exactly 3 elements

* 7. Exhaustive Checking

Constructor pattern matching requires a wildcard fallback clause.
Without it, the compiler emits an =(undefined)= fallback:

#+BEGIN_SRC monad
(match x with
  (Just v) -> (show v))
  ;; WARNING: no fallback for Nothing
  ;; compiles to (cond ((__adt_tag_ x 0) ...) (else (undefined)))
#+END_SRC

#+BEGIN_SRC wisp
match x with
  (Just v) -> show v
#+END_SRC

Always include ~_~ or ~default~ fallback clause for coverage. The
compiler does not currently perform exhaustiveness analysis — it simply
inserts an =(undefined)= guard for unmatched cases. Exhaustiveness-aware
fallback removal (where the compiler statically proves all branches are
covered) is future work.

For error handling see [[file:error-handling.org][Error Handling]] and
[[file:errors.org][Errors]].

Key Points:
- Missing wildcard causes an =(undefined)= fallback at runtime
- Always add =_ -> ...= as the last clause for robustness
- Exhaustiveness checking is not yet implemented

* 8. Desugaring Internals

=build_pattern_conditions= ([[file:../context/pmatch.org::184]])
generates condition expressions per pattern kind:

| Pattern        | Generated condition                            |
|----------------+------------------------------------------------|
| Wildcard       | always true                                    |
| Variable       | always true (binds the name)                   |
| Int literal    | =(= param value)=                              |
| Float literal  | =(= param value)=                              |
| String literal | =(string= param value)=                        |
| Empty list     | =(empty? param)=                               |
| List           | length check + element destructuring via =nth= |
| Constructor    | tag check via =__adt_tag_= + field destructuring |

Each condition is AND-ed with the previous clause's negation to ensure
first-match semantics. The resulting chain is emitted as a =(cond ...)=
expression.

See [[file:../context/pmatch.org][context/pmatch.org]] for the full compilation
algorithm and [[file:../context/codegen.org][context/codegen.org]] for LLVM IR
generation.

Key Points:
- Desugaring produces a =(cond ...)= chain with mutually-exclusive conditions
- Each clause's condition includes the negation of all prior clause conditions
- List patterns generate =nth= calls for element destructuring
- ADT patterns generate =__adt_tag_= calls for tag checking

* 9. Runnable Example

#+BEGIN_SRC monad
;; run with: monadc patterns-demo.mon

(define (classify [n :: Int] -> String)
  | 0       -> "zero"
  | n | (> n 0) -> "positive"
  | _       -> "negative")

(show (classify 0))   ;; prints "zero"
(show (classify 5))   ;; prints "positive"
(show (classify -3))  ;; prints "negative"

;; Implicit matching with guards + wildcard fallback
(define (fact : Int -> Int)
  0 -> 1
  n -> (* n (fact (- n 1))))

(show (fact 5))        ;; prints 120

;; Explicit (match ...) on a sub-expression
(define (describe-pair [p :: (Pair Int Int)] -> String)
  (match p with
    (Pair x y) | (= x y) -> "equal"
    (Pair _ _)           -> "not equal"))

(show (describe-pair (Pair 3 3)))  ;; prints "equal"
(show (describe-pair (Pair 3 4)))  ;; prints "not equal"
#+END_SRC

* 10. Gotchas

Common mistakes and surprising behaviours with pattern matching:

- **Missing wildcard =_= fallback** — the compiler inserts =(undefined)= for uncovered cases, causing a runtime error. Always add =_ -> ...= as the last clause.

- **Guards are evaluated after pattern match** — the pattern must match first; the guard is only checked if the pattern succeeds.

- **No backtracking** — patterns are tried top-to-bottom with first-match semantics. Once a clause fires, its body is evaluated; there is no backtracking to try alternative matches.

- **Variable vs wildcard** — =_= and =n= both match anything, but =_= does not bind. Using =_= when you intend to bind the value is a silent no-op.

- **Leading pipe =|= is purely cosmetic** — adding or removing =|= before a clause has zero effect on semantics or codegen.

- **Constructor patterns require inference first** — desugaring is deferred to codegen time because the full ADT definition is needed for tag resolution.
